§1 chat agent 跟 background agent 的 6 大本质区别

维度chat agentbackground agent影响
触发方user 直接 POSTcron / MQ / webhook / event需要触发器抽象 + 调度器
等待方user 在前端 SSE 流没人 · 结果异步通知不需要 stream · 需要 notify channel（email / slack / pr）
时间尺度秒-分分-小时-天不能塞内存 · checkpoint 必须每步持久化
失败可恢复用户重试就好必须自己 resume需要 idempotency key + state save
可观测有 user 反馈没人看 trace · 只看 metricSLO + alert 是唯一信号
cost 控制per-session budgetper-job + per-tenant 双层一个失控 job 能烧账户 24h 不被发现

"等不到 user 反馈"是设计约束的核心

chat agent 出错了，user 看到 "[Error]" 会自己 retry 或反馈。background agent 出错了， 默认没人发现。所以你必须：

  • 主动发声 · 失败 → 写 DLQ + page on-call · 不依赖 user 报
  • 每步留痕 · trace + checkpoint · 失败后能 replay 到失败点
  • cost 硬上限 · 单 job 不能花 > X · 不然失控不知道
  • 幂等 · MQ at-least-once · 同 msg 可能投 2-10 次

"没有 SSE"反而简化了一些事

chat agent 的"streaming + 中断 + reconnect"是一个非常烦的设计 · background 完全没这事。你只需要：

  • start → 跑完 → 写结果 · 不需要中途 yield
  • 失败 → 重试整个或从 checkpoint resume
  • 不存在 "user 关了浏览器" 的边界 case

§3 模式 1 · Scheduled · cron / APScheduler / Temporal

典型场景

  • Daily/weekly report：每天 9 点把昨天的 ticket / commit / metric 用 agent 总结成 1 页 brief
  • 缓存预热：每小时 prefetch top-100 query 的 RAG embedding
  • 数据质量巡检：每天扫 schema 漂移 / 异常值 · agent 写 ticket
  • 背景索引：每周 reindex 整个 doc base · 失败 fall back 上版本

3 档实现 · 按规模选

规模方案优点限制
开发 / 单机Linux cron + uv run0 依赖 · 0 服务 · 1 行 crontab没监控 · 没重试 · machine reboot 丢
小生产APScheduler + Postgres jobstorePython 内嵌 · jobstore 持久 · 简单单点 · scheduler crash = 错过
大生产Temporal / Airflow / PrefectHA · retry · time-travel · UI运维重 · 学习曲线

最小可跑 · cron + python · 30 行

# /etc/cron.d/taotao-daily
# 每天 09:00 跑 daily report
0 9 * * * agent  cd /opt/taotao && uv run python -m background.daily_report

# backend/background/daily_report.py
import asyncio, json, time, pathlib
from agent.graph import compile_async  # 复用主 graph

async def main():
    graph = await compile_async()
    job_id = f"daily-{time.strftime('%Y%m%d')}"
    cfg = {"configurable": {"thread_id": job_id}}
    state = {"messages": [
        {"role": "system", "content": "You write a 1-page daily brief."},
        {"role": "user", "content": "Pull yesterday's commits/issues/metrics and brief."},
    ]}
    final = await graph.ainvoke(state, config=cfg)
    pathlib.Path(f"/var/reports/{job_id}.md").write_text(final["answer"])
    print(json.dumps({"job": job_id, "ok": True}))

if __name__ == "__main__":
    asyncio.run(main())

关键点：thread_id = job_id · LangGraph checkpointer 自动按 thread 存状态 · 下一次同 job 重跑会复用 state（如果你想从 checkpoint resume；不想就用每天新 thread_id）。

陷阱 · cron 不会通知你失败

cron 默认 stdout/stderr 走 mail · 多数生产机器没装 mail。建议：脚本结尾主动 POST 给一个 /health/cron · 监控这个 endpoint 是否每 24h 收到一次。没收到就 alert · 这是 dead-man's switch 模式。

§4 模式 2 · Queue · Celery / RQ / SQS

什么时候上 queue

当出现以下任意一种 · 别再裸 cron / 同步 endpoint：

  • 请求量瞬时大于 worker 处理能力（削峰）
  • 需要重试（任务可能因临时故障失败）
  • 需要异步（请求方不想等几分钟）
  • 需要可见性（队列长度、堆积、worker 数）

3 档选型

RQ (Redis Queue)

极简 · Python only · Redis backend · 100 行 worker 自己能写。1k jobs/day 以内不疼。 缺：没有 chord/group · 复杂 workflow 难。

Celery

Python 生态事实标准 · Redis/RabbitMQ/SQS broker · chord/group/canvas 完整 · monitoring（Flower）成熟。 缺：配置复杂 · 文档老 · 调试痛。

AWS SQS

云原生 · 0 运维 · at-least-once · 自动 DLQ。缺：lock-in · 没有原生 chord · message size limit 256KB。

Temporal

不只是 queue · 是 workflow engine · 内置 retry / time-travel / human-in-the-loop / saga。 缺：自己运维成本高 · cloud 版贵。

Worker · 100 行雏形

# backend/background/worker.py
import asyncio, json, sqlite3, signal, time, uuid
from agent.graph import compile_async

DB = "/var/lib/taotao/jobs.db"

def init():
    with sqlite3.connect(DB) as c:
        c.execute("""CREATE TABLE IF NOT EXISTS jobs (
            id TEXT PRIMARY KEY,
            status TEXT NOT NULL CHECK(status IN ('queued','running','done','failed')),
            payload TEXT NOT NULL,
            result TEXT,
            attempts INT DEFAULT 0,
            created_at REAL,
            updated_at REAL
        )""")
        c.execute("CREATE INDEX IF NOT EXISTS idx_status ON jobs(status, created_at)")

def claim_one(worker_id: str) -> dict | None:
    """SELECT ... FOR UPDATE-equivalent in SQLite"""
    with sqlite3.connect(DB, isolation_level="IMMEDIATE") as c:
        row = c.execute("""SELECT id, payload FROM jobs
                           WHERE status='queued' OR (status='running' AND updated_at < ?)
                           ORDER BY created_at LIMIT 1""",
                        (time.time() - 600,)).fetchone()
        if not row: return None
        c.execute("UPDATE jobs SET status='running', updated_at=?, attempts=attempts+1 WHERE id=?",
                  (time.time(), row[0]))
        return {"id": row[0], "payload": json.loads(row[1])}

def finish(job_id: str, status: str, result: str):
    with sqlite3.connect(DB) as c:
        c.execute("UPDATE jobs SET status=?, result=?, updated_at=? WHERE id=?",
                  (status, result, time.time(), job_id))

async def run_loop():
    init()
    graph = await compile_async()
    worker_id = f"w-{uuid.uuid4().hex[:6]}"
    stop = asyncio.Event()
    signal.signal(signal.SIGTERM, lambda *_: stop.set())

    while not stop.is_set():
        job = claim_one(worker_id)
        if not job:
            await asyncio.sleep(2)
            continue
        try:
            cfg = {"configurable": {"thread_id": job["id"]}}
            final = await graph.ainvoke(job["payload"], config=cfg)
            finish(job["id"], "done", json.dumps(final))
        except Exception as e:
            finish(job["id"], "failed", str(e))

if __name__ == "__main__":
    asyncio.run(run_loop())

核心点 4 个：

  1. claim_one 用 IMMEDIATE 事务 + 时间戳 · 实现 visibility timeout（10min 没更新就视为 worker 死，重新可被 claim）
  2. thread_id = job_id · LangGraph checkpoint 帮 resume
  3. SIGTERM graceful · 收到信号不接新活，但跑完手头的
  4. attempts 字段 · 配 dispatcher 决定 retry vs DLQ

§5 模式 3 · Webhook · GitHub / Slack / Stripe

核心约束 · webhook 必须 200 快

所有主流 webhook（GitHub / Slack / Stripe / Linear / Notion）都3-5s 内必须 200， 否则视为失败、retry。agent 可能要跑 30s-30min。所以：

铁律

Webhook handler 只做 4 件事：(1) 验签名 (2) idempotency check (3) enqueue (4) 200 OK · 全部 < 100ms。真活让 worker 跑。

例 · GitHub PR review agent

# backend/routers/webhooks.py
from fastapi import APIRouter, Header, HTTPException, Request
import hmac, hashlib, json
from agent.background import enqueue

router = APIRouter(prefix="/webhooks", tags=["webhook"])

def verify_github(body: bytes, sig: str, secret: str) -> bool:
    mac = hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()
    return hmac.compare_digest(f"sha256={mac}", sig or "")

@router.post("/github")
async def github(
    request: Request,
    x_github_event: str = Header(...),
    x_hub_signature_256: str = Header(...),
    x_github_delivery: str = Header(...),
):
    body = await request.body()
    if not verify_github(body, x_hub_signature_256, GH_SECRET):
        raise HTTPException(401, "bad signature")
    payload = json.loads(body)
    if x_github_event == "pull_request" and payload["action"] in ("opened", "synchronize"):
        # idempotency: x-github-delivery is unique per delivery; re-deliveries reuse it
        enqueue(
            kind="pr_review",
            id=x_github_delivery,
            payload={"pr_url": payload["pull_request"]["url"]},
        )
    return {"ok": True}  # always 200 fast

5 个 webhook 必做 check

  1. 签名验证：GitHub/Stripe/Slack 都给 HMAC · 没有就拒
  2. idempotency key：用 delivery_id / event_id · 同 id 重复来视为已处理
  3. 事件白名单：只处理你需要的 action · 别 catch-all
  4. 立刻 200：永远不在 handler 里跑业务逻辑
  5. replay 友好：你应该能从 trace 找到原始 payload + 重新跑

§6 模式 4 · Event-stream · Kafka / NATS / Pulsar

当吞吐过 10k events/day 或者要做实时反应（<1s）· webhook + queue 撑不住 · 上 stream。这一档跟前面三个有本质区别：

  • Partition：相同 key 的 event 必到同一 partition · 保证顺序
  • Consumer group：N 个 worker 自动分担 partition
  • Offset：消费进度保存在 broker · 重启从 offset 继续
  • Replay：从历史 offset 重消费 · debug / migration 利器

典型 agent 应用

  • 客服 triage：每条 customer message 进 Kafka · agent 决定路由（FAQ / human / refund flow）
  • fraud scoring：每个 transaction event · agent + 规则混合打分
  • RAG ingestion：doc 变更 event → 切 chunk → embed → 写 vector store

Kafka consumer · 雏形

from aiokafka import AIOKafkaConsumer
import asyncio, json
from agent.graph import compile_async

async def consume():
    graph = await compile_async()
    consumer = AIOKafkaConsumer(
        "customer-messages",
        bootstrap_servers="kafka:9092",
        group_id="taotao-triage",
        enable_auto_commit=False,
        auto_offset_reset="latest",
    )
    await consumer.start()
    try:
        async for msg in consumer:
            event = json.loads(msg.value)
            cfg = {"configurable": {"thread_id": f"triage-{event['conversation_id']}"}}
            try:
                await graph.ainvoke({"messages": event["history"]}, config=cfg)
                await consumer.commit()  # 仅成功才提交 offset
            except Exception as e:
                # 不 commit · 下次重消费
                # 也可以发到 DLQ topic
                await dlq(msg, str(e))
    finally:
        await consumer.stop()

提交 offset 的纪律 · "成功才 commit" 是 at-least-once 的核心。如果你 agent 是幂等的（§8）· 这就够了。如果不幂等 · 你需要 exactly-once（事务性 producer + consumer 读 + DB 事务一起 commit · 复杂得多 · 90% 场景不需要）。

小公司不要上 Kafka 的 3 个理由

(1) Kafka 自己运维成本 = 1 个 SRE FTE。(2) 你的 throughput 99% 不需要 partition。 (3) Redis Streams + RQ 撑得了 5k events/day。非要实时 + >10k/day 才 justify。

§7 状态持久化 · checkpoint / resume / time-travel

chat agent 状态丢了 · user 重 ask 一遍。background agent 状态丢了 · 你不知道丢了。 所以持久化是核心约束。

Resume from checkpoint · 实操

LangGraph 的强：每个 node 跑完自动 checkpoint。失败重启时从最后一个成功 node 继续 · 不需要重头跑。

# 失败后从 checkpoint resume
from agent.graph import compile_async

async def resume_job(job_id: str):
    graph = await compile_async()
    cfg = {"configurable": {"thread_id": job_id}}
    # state=None · graph 自动 load 上次 checkpoint 继续
    final = await graph.ainvoke(None, config=cfg)
    return final

Time travel · 调试利器

LangGraph checkpointer 不是覆盖式 · 是 append-only。你可以：

  • graph.get_state_history(cfg) · 看每个 step 的 state 快照
  • 选某个历史 checkpoint · 改一个字段 · 从那点 fork 出新分支重跑
  • 专门用来定位 "为什么 agent 在第 5 步走偏" · 在第 4 步换个 prompt 重跑

Cursor BG / Claude Code 的等效

  • Cursor BA · checkpoint 是 git commit + chat session JSON
  • Claude Code · --continue 读 ~/.claude/sessions/<id>.jsonl · 把 message history 重灌
  • Devin · 自家 worker + state 在自有 DB · UI 给你 "rewind" 按钮

§8 Idempotency · 同一 event 触发 N 次只执行 1 次

所有 MQ / webhook 都是 at-least-once。你必须假设同一 event 可能投 2-10 次。 没做幂等 · 用户会被重复扣款 / 重复发邮件 / PR 被 review 5 遍。

3 档实现 · 按代价排

① Idempotency key in DB

每个外部 event 给一个稳定 key（GitHub delivery_id / Stripe event_id / message_id）· jobs 表 PK 就用它 · 重复 INSERT 走 ON CONFLICT IGNORE。

INSERT INTO jobs(id, ...) VALUES(?, ...)
ON CONFLICT(id) DO NOTHING;

② 内容哈希

没有外部 ID 时 · 用 sha256(payload) 当 idempotency key。代价是真不同 但内容一样的请求会被合并 · 通常无害。

③ 业务级 idempotency

"send_email" tool 自己记 sent_emails(message_id) · 同 id 二次调用直接 return cached。 这是最稳的 · 也最贵（每个 side-effect tool 都要写）。

额外 · 时间窗

"同一 user 30s 内同 query 视为重复" · 适合 button 防误点 · 不适合真业务幂等。

"幂等" 不只 enqueue · 还要 effect 幂等

常见错误：enqueue 做了去重，但 worker 跑两次都发了邮件。idempotency 必须延伸到 side-effect。 规则：所有"对外发号施令"的 tool（send_email · charge · publish）都自带 idempotency key 参数 · 上游 event_id 透传到这个参数。

§9 Failure / retry / backoff · 5 种策略 · DLQ

Retry 5 种策略

策略公式用在
Constant每次 30s简单 transient · LLM rate limit
Linear30s · 60s · 90s · ...慢恢复服务
Exponential1 · 2 · 4 · 8 · ... (cap)大多数 · 默认
Exp + jitterexp ± random大并发 · 防 thundering herd
Adaptive看 server 返回的 Retry-After header有 rate limit 头的 API

推荐：exponential + jitter · cap 5min · max 5 次 · 之后 DLQ。

import asyncio, random

async def with_retry(fn, max_attempts=5, base=1.0, cap=300.0):
    for attempt in range(max_attempts):
        try:
            return await fn()
        except Retryable as e:
            if attempt == max_attempts - 1: raise
            sleep = min(cap, base * 2 ** attempt) + random.uniform(0, base)
            await asyncio.sleep(sleep)
        except FatalError:
            raise  # 不重试 · 比如 401

区分 Retryable vs Fatal

  • Retryable：429 · 5xx · 网络超时 · LLM ratelimit · DB lock
  • Fatal：4xx (除 429) · validation error · auth 失败 · NotFound

把这个判断放在error class hierarchy · 不要每个调用点都写 if。

DLQ · 死信队列

跑 5 次还失败的 job · 不能再 retry 也不能丢 · 进 DLQ 等人 review。DLQ 设计 3 要点：

  1. 独立 store（jobs 表 status=dead 或单独 dlq 表）
  2. 原因结构化：分 transient_exhausted / permanent / poison_pill 三类
  3. 有 UI/CLI replay：手工 review 后能一键重投

真实坑 · poison pill

某条 message 触发了 worker 的 bug · worker crash · 重新 dequeue 还是它 · 又 crash · 死循环。 缓解：每 message 设 max_attempts (5)，超过 → DLQ + alert。SQS 自带 redrive policy 干这事。

§10 反编译 · Cursor BA · Devin · Claude Code · GitHub Action

Cursor Background Agent (2025)

  • 结束：自己 commit + push + 开 PR · UI 通知
  • 恢复：UI "continue" → 同一容器或拉新容器 from git state
  • cost cap：per-BA 默认 cap，超额停（公开文档没具体数）

Devin (Cognition 2024)

  • 触发：Slack 消息 / web UI / API
  • 容器：每 task 一个 VM，浏览器 + shell + IDE 共存
  • 状态：Cognition 自有 state DB · UI 显示 timeline + rewind
  • 结束：PR + Slack 通知 · 中途要确认会主动问
  • 恢复：rewind 到任一历史步骤 · 改 prompt 继续

Claude Code --continue / --resume

  • 触发：CLI · 用户主动 invoke
  • 状态：本地 ~/.claude/sessions/<id>.jsonl · 一个 session 一个文件
  • resume：claude --resume <id> 或 --continue 接最近 · 重灌 message 历史
  • plan mode：跑前可"先 plan 再确认 / 修改 / commit"
  • 区别：技术上还是 chat agent · 但提供了"暂停/继续"的会话级 background 体验

GitHub Action 跑 agent

  • 触发：push / PR / schedule / workflow_dispatch / repository_dispatch
  • 容器：GitHub-hosted runner 或自托管
  • 状态：job 内 ephemeral · 跨 job 用 actions/cache · 跨 run 用 artifact
  • 恢复：原生不支持 · 你得自己存 state（artifact / GitHub Issue body）
  • 使用：minimal cost 的 daily/weekly agent · PR review · auto-fix lint

面试角度

这 4 个产品对应了 4 种 background agent 形态。Cursor BA = "git-based long-running agent"， Devin = "VM-based 自治 agent"，Claude Code resume = "本地会话保持"， GH Action = "CI 里塞 agent"。被问到 "你怎么实现长跑 agent"，先问：是哪种形态？

§11 Cost / observability · trace 没人看也要存

chat agent 一次 $0.05 · 用户感知。background agent 单 job $5-50，跑 1000 个 job/day = $5-50k/day · 上线第 1 个月你就破产。所以：

Cost 三层硬上限

  1. Per-job cap：单 job > $X 强制中止 · 写 dlq + alert
  2. Per-tenant daily cap：每 tenant 每天 > $Y 暂停所有该 tenant 的 background
  3. Global daily cap：所有 background 总额 > $Z 全停 · 通常你没钱再 expose

Observability 必备 · 5 指标

指标SLO 例alert 条件
queue_depth< 100> 1000 持续 5min
job_duration_p95< 10min> 30min · 单 job hung
failure_rate< 2%> 5% / hour
cost_per_hour< $Xburn rate > 2× baseline
dlq_size< 10新增 > 0 立刻 page

Trace 留多久

Background agent 出事的复盘窗口比 chat 长。建议 30 天 hot · 1 年 cold (S3 + glacier)。 保留期短了你出 incident 调不出原始 trace · 长了 trace store 爆账单。

本仓库实现

backend/agent/observability/usage.py 已经记录每次 LLM 调用 cost。 把它 + per-job 标签 · 加 alert（Datadog / Honeycomb）即可。background 模式下 尤其建议把 OTel 接 sink · 因为没有 user 帮你发现问题。

§12 我们的实现 · backend/agent/background/

把这一本书涉及的 4 模式 + retry + idempotency + cost cap 浓缩成 ~400 行可跑代码：

backend/agent/background/
├── __init__.py
├── models.py          # SQLite jobs 表 schema + ORM
├── enqueue.py         # idempotent enqueue (key + payload + max_attempts)
├── worker.py          # poll loop + claim_one + visibility timeout + SIGTERM
├── retry.py           # exp + jitter + Retryable/Fatal 分类
├── triggers/
│   ├── cron.py        # APScheduler in-process · prod 接 cron
│   ├── webhook.py     # FastAPI router · GitHub/Stripe/Slack 验签 + enqueue
│   └── kafka.py       # aiokafka consumer · group_id + manual commit
└── tests/
    ├── test_idempotency.py
    ├── test_retry.py
    └── test_worker.py

使用：

# 1. cron · /etc/cron.d/taotao
*/15 * * * * agent uv run python -m agent.background.triggers.cron tick

# 2. queue · 在 supervisor / systemd 起 N 个 worker
[program:taotao-worker]
command=uv run python -m agent.background.worker
numprocs=4

# 3. webhook · 在 FastAPI 注册
from agent.background.triggers import webhook
app.include_router(webhook.router)

# 4. kafka (可选)
[program:taotao-kafka]
command=uv run python -m agent.background.triggers.kafka

所有 4 种触发共用同一个 worker pool。这是关键 · 不要每种触发一套独立 worker · 那样 worker 数量爆炸 · 也无法做统一 cost cap。
