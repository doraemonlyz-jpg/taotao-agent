Chapter 1 · 立场

为什么是 LangGraph · 跟 LangChain LCEL 的本质区别

不是 "LangChain 加了循环"——是另一种数据模型

很多人误以为 LangGraph 是 "LangChain 2.0 加了循环"。这是错的——它们解决的是不同抽象层的问题。 Senior 面试这个问题答 "一个能 loop 一个不能" 直接掉进 mid 档。

§1.1LangChain LCEL · DAG 风格的 chain

LangChain LCEL（LangChain Expression Language）的核心抽象是 Runnable——一个无状态的输入→输出函数，用 | pipe 串起来：

# LCEL 经典写法
chain = prompt | llm | output_parser
result = chain.invoke({"question": "hi"})

设计哲学：无状态 · 静态 DAG · functional composition。每个 Runnable 是纯函数 · 整个 chain 是 declarative。

这模型对 chain 完美 · 对 agent 不行：

  • 没循环——agent 是 think→act→observe→think 反复
  • 没状态——agent 要记 history / iter / 中间结果
  • 没 HITL——chain 不暂停 · agent 经常要等用户
  • 没 checkpoint——LLM 调一次几秒到几分钟 · 服务挂了不能从头再来
  • 没并发原语——agent 想同时调 3 个 tool · LCEL RunnableParallel 不够灵活

§1.2LangGraph · BSP 风格的 state machine

LangGraph 的核心抽象是显式 state graph + Pregel 执行器：

# LangGraph 核心结构
class State(TypedDict):
    messages: Annotated[list, add_messages]  # reducer 决定怎么 merge

g = StateGraph(State)
g.add_node("agent", agent_fn)        # node 读 state · 写 state update
g.add_node("tools", tool_fn)
g.add_conditional_edges("agent", route, ["tools", END])
g.add_edge("tools", "agent")

graph = g.compile(checkpointer=PostgresSaver(...))

设计哲学：显式 state · channels · supersteps。每个 node 读 state · 输出 update · reducer 合并 · 进入下个 superstep。这不是函数组合 · 是状态机 + 数据流。

§1.4BSP / Pregel · LangGraph 思想根源

LangGraph 不是凭空设计——它明确地基于 Google Pregel 论文（2010 · 大规模图计算系统），而 Pregel 又基于 BSP (Bulk Synchronous Parallel) 模型（Valiant 1990）。 理解 BSP 是理解 LangGraph 的钥匙。

BSP 三阶段（每个 superstep）

  1. Compute——所有 active node 并行跑，读上一 superstep 的 state · 输出 update（不直接改 state）
  2. Communication——node 输出的 update 通过 channel 传给后续 node
  3. Barrier——所有 node 等齐，reducer 合并 update 写 state，进入下一 superstep

为什么选 BSP 而不是 actor / dataflow？

  • vs actor（Akka / Erlang 风）——actor 异步消息 · 时序难追 · debug 噩梦。BSP 同步 · 每 superstep 是确定性 snapshot · checkpoint 自然
  • vs dataflow（Apache Beam / Flink 风）——dataflow 流式 · 没有"全局回到 user 等输入"的概念。BSP 把 superstep 边界当成天然 HITL 接入点
  • 对 agent 的好处——time travel / resume / 并发 fan-out 全是 BSP 自然产物
面试金句 · 这一段直接区分 senior

"LangGraph 不是 LangChain 加了循环 · 是另一种执行模型——基于 Pregel/BSP。每个 superstep 内 node 并行跑，barrier 同步，reducer 合并 update 写 channel，checkpointer 持久化。这给了它 4 个对 agent 关键的能力：循环（自然）· 并发（fan-out）· HITL（superstep 边界 pause）· time travel（每 step snapshot）。LCEL 是函数组合 · 解决 chain 问题；LangGraph 是 state machine · 解决 agent 问题。两者互补不替代——LangGraph 节点内部经常用 LCEL"

§1.5什么时候不用 LangGraph

不要无脑上 LangGraph。这些场景用 LCEL 就够：

  • 简单 RAG（query → retrieve → LLM → answer · 一遍走完）
  • 纯 prompt chain（多步 prompt 但不需 loop）
  • 批处理 ETL（input → transform → output · 静态 pipeline）
  • 只用 OpenAI function calling 的简单 ReAct（OpenAI Agents SDK 都比 LangGraph 轻）

LangGraph 的复杂度是有代价的——多了 State / channel / reducer / checkpoint 4 个概念。简单场景用 LCEL · 真有 agent 需求再上 LangGraph。

State + Reducer · 数据怎么流

90% 的 LangGraph bug 在 reducer 用错

§2.1State schema · 3 种写法

# 1) TypedDict (最常见)
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]
    iter: int
    user_id: str | None

# 2) Pydantic (要 validation)
from pydantic import BaseModel
from langgraph.graph import StateGraph

class State(BaseModel):
    messages: list
    iter: int = 0

# 3) dataclass (轻量 · 不要 type hint 校验)
from dataclasses import dataclass, field

@dataclass
class State:
    messages: list = field(default_factory=list)
    iter: int = 0

选哪种：

  • TypedDict——最常用 · 跟 LangGraph 文档对齐 · reducer Annotated 写法清晰
  • Pydantic——你的 state 字段需要 validation（如 enum / 数值范围）。注意 reducer 写法略不同，用 Field(..., metadata={"reducer": add_messages})
  • dataclass——已经在用 dataclass 的项目 · 但 reducer 注解支持弱 · 不推荐新项目

§2.2Reducer · 并发更新怎么 merge

这是 LangGraph 最容易忽视也最容易踩坑的概念。

问题：node A 和 node B 同一 superstep 都对 messages 字段输出 update——LangGraph 怎么 merge？

答案：取决于该字段的 reducer。

# 不写 reducer · 默认 = LastValue · 后到的覆盖前到的（不可预测！）
class State(TypedDict):
    messages: list  # ⚠️ A 和 B 同 step 写, 只留一个

# 写 reducer · A 和 B 的 update 累加
class State(TypedDict):
    messages: Annotated[list, add_messages]  # ✓ 两边的都保留 + dedup by id
    counters: Annotated[dict[str, int], merge_counters]  # 自定义

常用 reducer 速查

Reducer语义用在
(默认)LastValue · 覆盖单值字段（current_step / final_answer）
operator.addlist / str 拼接 · 数字相加累积 list / counter
add_messageslist of BaseMessage 拼接 + by-id dedup + ToolMessage 配对messages 字段几乎必用
operator.or_set union · dict mergetag 集合 / metadata 字典
自定义Callable[[X, X], X]复杂场景

§2.3add_messages · 看似简单实则复杂

add_messages 是 LangGraph 内置 · 表面像 list append · 实际做了 4 件事：

  1. by-id dedup——message 有 id 时按 id 去重 · 不重复加
  2. by-id update——同 id 的新 message 替换旧的（用于 streaming · partial → full）
  3. RemoveMessage 处理——传 RemoveMessage(id=X) 会从 list 删 id=X 的 message
  4. ToolMessage 配对——ToolMessage 必须紧跟其 tool_calls 的 AIMessage 后
# 删 message 怎么写
from langchain_core.messages import RemoveMessage

def delete_old_messages(state):
    # 删除前 N-3 条 message · 保留最近 3 条
    msgs = state["messages"]
    if len(msgs) > 3:
        return {"messages": [RemoveMessage(id=m.id) for m in msgs[:-3]]}
    return {}

§2.4自定义 reducer · 真实例

例：实现"sub-agent 历史"——多个 sub-agent 同步跑 · 每个产出自己的 history 段 · 主 agent 最后看到合并版。

from typing import TypedDict, Annotated

def merge_subagent_history(left: dict, right: dict) -> dict:
    """把多个 sub-agent 的 history 按 agent_name key 合并."""
    out = {**left}
    for name, msgs in right.items():
        if name in out:
            out[name] = out[name] + msgs   # 同名 sub-agent 累积
        else:
            out[name] = msgs
    return out

class State(TypedDict):
    user_input: str
    main_messages: Annotated[list, add_messages]
    sub_history: Annotated[dict[str, list], merge_subagent_history]

# sub_agent_A 输出: {"sub_history": {"researcher": [msg1, msg2]}}
# sub_agent_B 输出: {"sub_history": {"coder":     [msg3]}}
# 合并后: state["sub_history"] = {"researcher": [msg1,msg2], "coder": [msg3]}

§2.5Node 返回 partial update · 别返完整 state

常见错误：

# ❌ 错 · 返回完整 state · LangGraph 会把所有字段当 update 触发 reducer
def my_node(state):
    state["iter"] += 1
    return state   # messages 也被当 update · add_messages 会 dedup 但仍走一遍

# ✓ 对 · 只返要改的字段
def my_node(state):
    return {"iter": state["iter"] + 1}

原则：node 是 reducer——你返 {"k": v} 就是给 channel k 输入 v · reducer 决定怎么合 · 别返没改的字段。

最常踩坑 · 默认 reducer = 覆盖

不写 Annotated → 该字段 reducer = LastValue（覆盖）。 如果有 2 个 node 同 superstep 都更新该字段（典型：fan-out 用 Send 派 N 个 worker · 每个写 result），结果只有一个会保留 · 其他丢。必须写 reducer · 哪怕只是 operator.add。

Chapter 3 · 原语 2

Channels · State 只是糖

看到这章你才算懂 LangGraph 真实的数据模型

§3.1Channel 是什么

核心揭秘：你写的 State + Annotated reducer 在 LangGraph 内部都被编译成 channels。State 不是真原语 · channel 才是。

Channel 是 Pregel 里的命名数据通道——一个有 reducer 语义的 slot · 接收多个 node 的 update · 持久化为 checkpoint 的一部分。

# 你写
class State(TypedDict):
    messages: Annotated[list, add_messages]
    iter: int

# 编译后内部是
channels = {
    "messages": BinaryOperatorAggregate(list, add_messages),
    "iter": LastValue(int),
}

§3.25 种内置 channel 类型

Channel语义对应 reducer
LastValue<T>只保留最后一次 write · 多 write 同 step 报错(默认 · 不写 Annotated)
BinaryOperatorAggregate<T, op>所有 update 用 op(left, right) reduceoperator.add / 自定义
Topic<T>list 累积 · 可设置每 step 后 resetpubsub 风
EphemeralValue<T>只活 1 superstep · 不持久化临时变量
Context<T>全局只读 · 整个 invoke 期间不变config-like

§3.3node 跟 channel 的关系

§3.4显式用 channel 跳过 State sugar

偶尔你需要更细粒度控制——直接用 channel API。例：node 只读不写某字段。

from langgraph.graph import StateGraph
from langgraph.constants import START, END

# 高级 · 不用 TypedDict · 直接定义 channels
from langgraph.channels import LastValue, Topic, BinaryOperatorAggregate
from operator import add

graph = StateGraph(
    state_schema=None,
    input="input", output="output",
)

# （高级用法 · 通常 StateGraph + TypedDict 已足够）

实战 99% 用 TypedDict 就够。但知道 channel 存在让你能解释 reducer / Send / interrupt 内部行为——senior 必备。

§3.5Channel 跟持久化的关系

每个 superstep 后，channel 的当前值序列化进 checkpoint。这就是为什么：

  • state 字段类型必须可序列化（不要塞 db connection / file handle）
  • checkpoint 大小 = 所有 channel 当前值大小之和
  • Ephemeral channel 不进 checkpoint（设计上就是临时数据）
面试金句

"State 是 sugar · channels 是真实的 LangGraph 数据原语。每个 state 字段被编译成一个 channel · 字段的 Annotated reducer 决定 channel 类型（默认 LastValue · 写 reducer 就是 BinaryOperatorAggregate）。Node 是 channel 的 subscriber + writer——读当前值 · 输出 update · 让 channel 的 reducer 决定怎么 merge。每 superstep 后所有 channel snapshot 进 checkpoint · 这就是 time travel 的物理基础"

Chapter 4 · 原语 3

Checkpointer · 持久化 + 时间旅行

Resume / HITL / 多用户 / time travel 4 件事的物理基础

§4.1Checkpointer 是什么

Checkpointer = 每个 superstep 后把 channel 当前值持久化的存储后端。它让：

  • 服务挂了重启能 resume
  • interrupt() 触发后的 state 能存住等用户回来
  • 多用户对话隔离（按 thread_id）
  • time travel——回到任意历史 superstep · 改 state · 分支重跑

§4.24 种 backend · 怎么选

Backend持久化并发时间旅行用
MemorySaver❌ in-process不安全✓开发 · demo · 单测
SqliteSaver✓ 本地文件有限✓本地 / 单机小项目
PostgresSaver✓ Postgres✓✓生产首选
RedisSaver✓ Redis✓ · 极快✓高 QPS · 短期会话 (有 TTL)
 
# 生产用 Postgres
from langgraph.checkpoint.postgres import PostgresSaver
from psycopg_pool import ConnectionPool

pool = ConnectionPool(conninfo="postgresql://user:pwd@host/db", max_size=20)
checkpointer = PostgresSaver(pool)
checkpointer.setup()  # 创建表 · 只跑一次

graph = g.compile(checkpointer=checkpointer)

§4.3thread_id / checkpoint_ns / checkpoint_id 三层标识

每个 checkpoint 有 3 层 key——senior 必须能讲清：

层含义典型值
thread_id"对话线程"——一个用户的一次会话f"{user_id}:{session_id}"
checkpoint_ns"命名空间"——subgraph 隔离主 graph = "" · subgraph = "researcher"
checkpoint_idcheckpoint 唯一 ID（每 superstep 新生）UUID（自动）
 
# 配置只需 thread_id · 其他自动
config = {"configurable": {"thread_id": "alice:session-42"}}

graph.invoke({"messages": [...]}, config)
# 内部每 superstep 都 save: (thread="alice:session-42", ns="", id=<uuid>)

§4.4get_state / get_state_history / update_state · time travel API

cfg = {"configurable": {"thread_id": "alice:s1"}}

# 1) 看当前 state
state = graph.get_state(cfg)
print(state.values)        # 当前所有 channel 值
print(state.next)          # 下一个会跑的 node · () = 已完成
print(state.tasks)         # pending tasks (含 interrupt 信息)

# 2) 看历史 (time travel 基础)
for ckpt in graph.get_state_history(cfg):
    print(ckpt.config["configurable"]["checkpoint_id"],
          "·", ckpt.values.get("iter"),
          "·", ckpt.next)

# 3) 回到某个历史点 · 用那个 checkpoint_id 当 config
old_cfg = {"configurable": {
    "thread_id": "alice:s1",
    "checkpoint_id": "<the old uuid>"
}}

# 4) 修改某个 state 字段后从那点重跑
graph.update_state(old_cfg, {"messages": [HumanMessage("换个问法试试")]})
graph.invoke(None, old_cfg)  # 从该点 resume · 走分支

time travel 实战价值：

  • debug——agent 走偏了 · 回到岔路前 · 改 prompt · 看是不是变好
  • 用户 "重做这一步"——chat UI 让用户 edit 上一句重发
  • 探索 "what if"——eval 时同 state 跑不同 model 对比

§4.5checkpoint 大小 · 不要塞大对象

每个 superstep save 一个完整 channel snapshot——state 大 = checkpoint 大 = DB 慢 + 贵。

实战经验：

  • 不要把 retrieval 召回的 100 个 chunk 全塞 state——用完就 RemoveMessage 清
  • 大 binary（图片 / PDF）存对象存储 · state 只存 URL
  • 每 thread 设 checkpoint 数上限（用 cron 删老的）—— 否则 1 个长 thread 几 GB
  • monitor checkpoint write latency · 突变意味着 state 膨胀

§4.6事务性 · checkpoint 跟你的 DB 写一起

常见问：node 里更新了业务 DB · 但 checkpoint 写失败怎么办？

答案：默认不事务一致——checkpoint 写失败时业务 DB 已改，造成 state inconsistent。 解决方案：

  1. Idempotent node——node 业务操作设计成幂等 · resume 重跑没事
  2. Saga pattern——业务操作记录到自己 table · 失败时有 compensating action
  3. 同 DB 同事务——业务 DB 也用 Postgres + checkpointer 用 PostgresSaver · 包一个 transaction（高级用法 · 需要自己 hack）
面试金句 · resume 内部到底发生啥

"Resume 不重跑前面 node · 是因为 checkpointer 存了上次 superstep 末所有 channel 的 snapshot。Pregel 重启时不从 START 开始 · 是从 state.next 指示的 pending node 开始 · 这些 node 读的就是已 restore 的 channel 值。所以 resume 是 O(remaining work) 不是 O(total work)——是真 resume 不是 replay"

Chapter 5 · 原语 4

Interrupt + Command · HITL 的正确姿势

2024 末重写的 API · 老版本 NodeInterrupt 已过时 · 新人最常错

§5.1新 API · interrupt() + Command(resume=...)

2024 年 Q4 LangGraph 推出新 HITL API · 比老的 NodeInterrupt exception 风格简单太多。新代码必须用新版。

from langgraph.types import interrupt, Command

def human_review_node(state):
    # 1) interrupt 把任意 payload 给外部 · 同时 pause graph
    user_response = interrupt({
        "action": state["proposed_action"],
        "reason": state["reason"],
        "prompt": "approve or reject",
    })
    # 2) 用户 resume 后 · interrupt() 返回 Command(resume=X) 里的 X
    return {"approved": user_response.get("approved", False)}

# 客户端代码
cfg = {"configurable": {"thread_id": "t1"}}

# 第一次跑 · 跑到 interrupt 处停
out1 = graph.invoke({"proposed_action": "refund $50"}, cfg)
print(out1)  # 含 __interrupt__ 字段

# 用户 approve · resume
out2 = graph.invoke(Command(resume={"approved": True}), cfg)
print(out2["approved"])  # True

§5.2interrupt 的 4 个内部要点

  1. 必须有 checkpointer——interrupt 状态靠 checkpoint 持久化 · 没 checkpointer 直接报错
  2. 幂等性问题——interrupt 是函数式风格 · resume 时 node 从头重跑到 interrupt() 返回点 · 所以 interrupt 之前的代码要幂等（不要发邮件 / 不要改 DB · 副作用放 interrupt 之后）
  3. 多个 interrupt 同 node——支持 · 按调用顺序 resume · 但要小心顺序依赖
  4. 跨 node 多 interrupt——支持 · 用户可以 batch resume 多个
最常踩坑
def bad_node(state):
    send_email(state["user"])  # ❌ resume 时这行会重跑 · 邮件发 2 次
    response = interrupt({...})
    return {...}

def good_node(state):
    response = interrupt({...})
    # interrupt 之后才副作用 · resume 后才执行 1 次
    send_email(state["user"])
    return {...}

§5.3Command · 不只是 resume

Command 是 LangGraph 的"node 输出动态控制"机制 · 比 conditional_edges 灵活。它能同时做 3 件事：

from langgraph.types import Command
from typing import Literal

def router_node(state) -> Command[Literal["agent_a", "agent_b", END]]:
    decision = classify(state)
    return Command(
        update={"routed_to": decision},  # ① 更新 state
        goto=decision,                       # ② 指定下个 node (替代 conditional_edges)
        graph=Command.PARENT,                # ③ 往父 graph 跳 (subgraph 用)
    )

Command vs conditional_edges：

  • conditional_edges——静态定义路径 · graph compile 时已知
  • Command——动态 · node 执行时才决定 · 还能同时 update state
  • 实战：simple routing 用 conditional_edges（清晰），复杂场景用 Command（灵活）

§5.4HITL 4 种典型 pattern

① Approve / Reject (最常见)

def approve_node(state):
    resp = interrupt({"action": state["action"], "prompt": "approve?"})
    return {"approved": resp["approved"]}

② Edit before continue

def review_draft(state):
    resp = interrupt({"draft": state["draft"], "prompt": "edit if needed"})
    return {"draft": resp.get("edited", state["draft"])}

③ Provide missing info

def ask_clarification(state):
    resp = interrupt({"need": "date_range"})
    return {"date_range": resp["date_range"]}

④ Multi-step approval (chained)

def multi_step(state):
    legal = interrupt({"step": "legal review"})
    if not legal["approved"]: return {"final": "rejected by legal"}
    finance = interrupt({"step": "finance review"})
    return {"final": "approved" if finance["approved"] else "rejected"}

§5.5老版 NodeInterrupt · 看到要识别

遗留代码可能用 raise NodeInterrupt(...) + interrupt_before / interrupt_after compile 选项 · 或 graph.update_state 手动改后再 invoke。这是老 API · 新代码必弃。 看到老代码要能识别 + 知道怎么迁移。

Send · 真正的 map-reduce

动态并行的钥匙 · 跟 conditional_edges 是两个东西

§6.1Send vs conditional_edges · 本质区别

新人经常分不清 · 这俩看着都是"决定下一步去哪"实际设计目的不同：

conditional_edgesSend
语义"我下一步去 A 还是 B"——routing"我要派 N 个独立 task 给 X · 同时跑"——spawn
下游 node 数1 个 (一条路)N 个并行 instance
每个 instance 看到的 state共享 state独立 input · 不是 state
典型场景分支决策map-reduce / 并行 sub-task

§6.2map-reduce 实战 · 并行总结 N 篇文章

from langgraph.constants import Send
from typing import TypedDict, Annotated
from operator import add

class State(TypedDict):
    urls: list[str]
    summaries: Annotated[list[str], add]   # reducer 必须 · 多 worker 并写
    final: str

class SummaryInput(TypedDict):
    url: str

def prepare(state: State):
    # node 不直接返 update · 返 Send 列表
    return [Send("summarize", {"url": u}) for u in state["urls"]]

def summarize(input: SummaryInput):
    # worker · 看到的是 SummaryInput 不是 State！
    summary = llm.invoke(f"summarize: {fetch(input['url'])}").content
    return {"summaries": [summary]}   # reducer add 累积

def finalize(state: State):
    final = "\n\n".join(state["summaries"])
    return {"final": final}

g = StateGraph(State)
g.add_node("summarize", summarize)
g.add_node("finalize", finalize)
g.add_conditional_edges(START, prepare, ["summarize"])  # prepare 返 Send list
g.add_edge("summarize", "finalize")
g.add_edge("finalize", END)

graph = g.compile()
out = graph.invoke({"urls": ["u1", "u2", "u3"]})
# summarize 派出 3 个并行 worker · 每个写 summaries
# reducer add 把 3 个 [summary] 合成 [s1, s2, s3]
# finalize 看到完整 summaries · 合一

§6.3Send 的 3 个内部要点

  1. Send 接受的 input 不是 State——是任意 dict / TypedDict · 跟 worker node 签名一致
  2. worker node 看到的也不是完整 State——只是 Send 给的 input。但 worker 的输出会进 State channels · 受 reducer 处理
  3. Send 列表内 task 真并行——同 superstep 跑 · 共享 channel · reducer merge

§6.4常见 anti-pattern · Send 配错 reducer

# ❌ State.summaries 没写 reducer · 默认 LastValue
class State(TypedDict):
    summaries: list[str]   # 多 worker 同 step 写 · 只剩 1 个！

# ✓ 必须给 reducer
class State(TypedDict):
    summaries: Annotated[list[str], add]

Send + 没 reducer = 数据丢——这是新人最常踩。 记住：多 writer 同 step → 必须 reducer。

§6.5Send 高级 · 动态 fan-out 数

Send list 长度可以是 LLM 决定的——这才是它的真正威力：

def planner(state):
    # LLM 看 query 决定要拆几个 sub-question
    sub_qs = llm.with_structured_output(SubQs).invoke(state["query"])
    # 动态 fan-out · 可能 1 个可能 10 个
    return [Send("researcher", {"q": q}) for q in sub_qs.questions]

这是 Deep Research 模式（agent-design.html §5.2）的标准 LangGraph 实现。

面试金句

"Send 跟 conditional_edges 都返 'next' 但语义不同——conditional_edges 是 routing（选 1 条路）· Send 是 spawning（派 N 个独立 instance 并行）。Send 给 worker 的 input 不是 State 是任意 dict · 但 worker 的输出会进 State channels · 必须配对应的 reducer · 否则多 worker 同 step 写会丢数据。Deep Research / map-reduce / sub-agent 并发探索这种 fan-out 模式都依赖 Send"

Subgraph + ToolNode · 组合的极致

把 graph 当 node 用 · 把 tool 调用模式化

§7.1Subgraph · graph 是一等公民

LangGraph 的compiled graph 本身可以当 node 用。这是组合性的基础。

# 子 graph (一个完整 LangGraph)
class SubState(TypedDict):
    sub_messages: Annotated[list, add_messages]
    sub_result: str

sg = StateGraph(SubState)
sg.add_node("think", think_fn)
sg.add_node("act", act_fn)
# ... 完整 graph 定义
sub_graph = sg.compile()

# 主 graph 用 sub_graph 当 node
class MainState(TypedDict):
    messages: Annotated[list, add_messages]
    sub_result: str

g = StateGraph(MainState)
g.add_node("sub", sub_graph)   # ✓ compiled graph 当 node
g.add_node("after", after_fn)

§7.2State 共享 vs 隔离 · 2 种 subgraph 用法

用法 A · 共享 state（自动 key 映射）

主 State 跟 sub State 有同名字段 → LangGraph 自动映射 · sub 写的字段流回主。

class MainState(TypedDict):
    messages: Annotated[list, add_messages]
    sub_result: str

class SubState(TypedDict):
    messages: Annotated[list, add_messages]   # 同名 → 共享
    sub_result: str                            # 同名 → 共享
    sub_internal: int                          # 主没有 → sub 内部用

用法 B · 完全隔离（subgraph 当 worker）

主 State 跟 sub State 无同名字段 → 用 wrapper 函数显式转换。

def call_subgraph(state: MainState):
    # 显式喂 input · 提取 output
    sub_out = sub_graph.invoke({"sub_messages": [...], "sub_result": ""})
    return {"sub_result": sub_out["sub_result"]}

g.add_node("sub", call_subgraph)

选哪种：messages 这种共享意图明确用 A；context 隔离 / 不让 sub 看到主 state 用 B。 Claude Code Task tool 的 sub-agent 隔离（agent-design.html §3.2）就是 B 的极致。

§7.3ToolNode · prebuilt 助手

每次写 ReAct 都重写 tool dispatch 节点很烦——LangGraph 给了 ToolNode。

from langgraph.prebuilt import ToolNode
from langchain_core.tools import tool

@tool
def search(query: str) -> str:
    """Web search."""
    return mock_search(query)

@tool
def calc(expression: str) -> str:
    """Eval math."""
    return str(eval(expression))

tools = [search, calc]
tool_node = ToolNode(tools)   # 自动处理 last AIMessage.tool_calls

# 这一句替代之前手写的 tool_node 函数 (agent-coding.html Q3)
g.add_node("tools", tool_node)

ToolNode 内部做了 4 件事：

  1. 取 last AIMessage 的 tool_calls list
  2. 每个 tool_call 找对应 tool function · 调它（支持并行）
  3. 异常捕获 · 转 ToolMessage(content="error: ...") · 不让 graph 挂
  4. 返回 ToolMessage list 写回 messages channel

§7.4create_react_agent · 一行 ReAct

更进一步——LangGraph 给 prebuilt ReAct agent。

from langgraph.prebuilt import create_react_agent

agent = create_react_agent(
    model=ChatOpenAI(model="gpt-4o-mini"),
    tools=[search, calc],
    checkpointer=PostgresSaver(...),  # 可选 · 给持久化
    state_schema=MyState,              # 可选 · 自定义 State
    interrupt_before=["tools"],        # 可选 · tool 调用前 HITL
)

# 直接 invoke · 内部就是 agent + ToolNode loop
out = agent.invoke({"messages": [HumanMessage("hi")]}, cfg)

什么时候用 prebuilt vs 自定义：

  • prebuilt——MVP / 标准 ReAct 行为够用 · 不想 reinvent
  • 自定义——要插入额外 node（HITL gate / planning / reflection / supervisor）就自己拼

# CrewAI 经典写法 · 跟 LangGraph 完全不同的抽象
from crewai import Agent, Task, Crew, Process

researcher = Agent(
    role="Senior Researcher",
    goal="Find latest AI agent papers",
    backstory="You are an expert in agent research...",

  • 抽象太高 → 想细控制就难（要 hack）
  • 没有 first-class HITL（2025 后加了但弱）
  • 没有真正持久化（demo 用 SQLite hack）

  • 生产用的非常少——大多停留在 hackathon

§9.1场景 1 · 生产级客服 agent · HITL + 持久化必须

选 LangGraph。理由：

  • HITL on refund / sensitive op——LangGraph interrupt() 是同类最强
  • 持久化——PostgresSaver 直接生产可用
  • multi-tenant——thread_id 模型天然适配
  • observability——LangSmith 集成最深

反方观点："我用 OpenAI Agents SDK 行不行？" 答：HITL 和持久化要自己 build · 时间不一定省。

§9.2场景 2 · 周末黑客松快速 multi-agent demo

选 CrewAI。理由：

  • 20 行写完 demo · 不写 state / reducer / channel
  • "role + task" 抽象对 demo 来说够用
  • YAML 配置——非工程队友能改

注意：千万别把 CrewAI demo 直接搬上生产——长期看一定要重写成 LangGraph。CrewAI 是原型工具不是生产框架。

§9.3场景 3 · OpenAI Realtime 语音 agent

选 OpenAI Agents SDK。理由：

  • Realtime API 跟 Agents SDK 同团队设计 · 集成最好
  • handoff 模式适配语音 routing（"我转给计费专员"）
  • tracing 在 OpenAI dashboard 直接看

例外：如果你强烈反 vendor lock-in · 用 LangGraph + 手写 voice loop 也行 · 但工作量大

§9.4场景 4 · TypeScript 全栈项目

选 Mastra · 或 Vercel AI SDK + 手拼。理由：

  • 不要为了用 LangGraph 引入 Python 服务——多一个语言多一倍运维
  • Mastra 有 LangGraph 80% 能力 + TS 生态
  • 简单场景 Vercel AI SDK 够用

反方："我可以用 LangGraph.js" 答：LangGraph.js 比 Python 版落后 6-12 个月特性 · 不推荐主用

§9.5场景 5 · Pydantic-heavy backend 加 agent

选 PydanticAI。理由：

  • Pydantic schema 直接当 agent output / tool input · 零转换
  • FastAPI / SQLModel 风格一致 · 团队上手快
  • 简单 agent 场景下比 LangGraph 简洁 5x

边界：复杂 multi-step workflow 仍要 LangGraph——那就 PydanticAI agent + LangGraph orchestrator 组合

§10.1StateGraph.compile() · 你写的代码到 Pregel 的转换

# 你写的
g = StateGraph(State)
g.add_node("agent", agent_fn)
g.add_node("tools", tool_fn)
g.add_edge(START, "agent")
g.add_conditional_edges("agent", route, ["tools", END])
g.add_edge("tools", "agent")
graph = g.compile(checkpointer=cp)

# 1. 把 State 字段编译成 channels (LastValue / BinaryOpAgg)

§10.2Pregel._loop() · 核心 superstep 循环

# 简化伪代码 · 真源码在 pregel/__init__.py
async def _loop(self):
    while True:
        # Step 1: 决定本 superstep 要跑哪些 node
        tasks = self._prepare_next_tasks()
        if not tasks:
            break   # 没有更多 task · 结束

        # Step 2: 并行执行这些 node (asyncio.gather)
        results = await asyncio.gather(*[task.run() for task in tasks])

        # Step 3: 检查 interrupt
        if any_interrupted(results):
            # 保存 checkpoint · raise GraphInterrupt
            await self.checkpointer.aput(...)
            raise GraphInterrupt(...)

        # Step 4: 收集 update · 通过 channel.update(values) 走 reducer
        for name, update_value in collect_updates(results):
            self.channels[name].update([update_value])

        # Step 5: 持久化 checkpoint
        if self.checkpointer:
            await self.checkpointer.aput(thread_id, ns, ckpt_id,
                                          self._snapshot_channels())

        # Step 6: 检查 recursion_limit
        self.step += 1
        if self.step > self.recursion_limit:
            raise GraphRecursionError(...)

这就是 LangGraph 的"心脏"。一旦你看懂这 6 步 · 之前所有概念都自然 click：

  • 循环——while True 自然支持
  • 并发——asyncio.gather 多 node 同 step 跑
  • reducer——channel.update 时 BinaryOpAgg 调你的 reducer
  • checkpoint——每 step 末写
  • interrupt——raise 时 state 已 save · 下次 resume 从 _prepare_next_tasks 开始
  • time travel——你给的 checkpoint_id 决定从哪个 snapshot restore channels

附录 A · 面试

15 道 senior LangGraph 面试题

每题给 1-2 句答题骨架 · 配套深度

概念 (5)
  1. LangGraph 跟 LangChain LCEL 的本质区别？ — LCEL 是 functional composition 解决 chain · LangGraph 是 BSP state machine 解决 agent。详见 §1。
  2. 什么是 superstep？ — Pregel 执行单元：所有 active node 并行跑 · barrier 同步 · reducer 合并 · 持久化 · 进下一波。
  3. State 跟 Channel 关系？ — State 是 sugar · 编译后每个字段是一个 channel · 字段的 Annotated reducer 决定 channel 类型。
  4. 为什么需要 reducer？ — 多 node 同 superstep 写同一字段时决定怎么 merge · 没 reducer 默认 LastValue 会丢数据。
  5. LangGraph 为啥基于 BSP 而不是 actor？ — actor 异步消息时序难追 · BSP 同步 + 每 step 是确定 snapshot · 自然支持 checkpoint / time travel / debug。
原语 (5)
  1. add_messages 内部做啥？ — by-id dedup + by-id update + RemoveMessage 处理 + ToolMessage-AIMessage 配对（4 件事）。
  2. interrupt() 触发后 checkpoint 怎么变？ — 当前 channel snapshot 全 save · state.next 指向 pending node · resume 时 Pregel 从 _prepare_next_tasks 开始不重跑前面 node。
  3. interrupt 之前的副作用为啥危险？ — resume 时 node 函数从头重跑到 interrupt() 返回点 · 副作用会重复执行。所有副作用必须放 interrupt 之后。
  4. Send vs conditional_edges 区别？ — conditional_edges 是 routing（选 1 条路）· Send 是 spawning（派 N 个并行 worker）· worker 接收 input dict 不是 State · 输出还是写 State channel。
  5. Subgraph 跟 sub-state 怎么映射？ — 同名字段自动 share / 不同名要 wrapper 函数。共享适合 messages · 隔离适合 sub-agent context isolation。
生产 (5)
  1. MemorySaver 为啥不能上生产？ — in-process · 重启数据丢 + 多 worker 不能共享。生产用 PostgresSaver / RedisSaver。
  2. thread_id 怎么生成？ — f"{user_id}:{session_id}" · 必须 per-user 隔离 · 串号是 P0 安全事件。
  3. recursion_limit 怎么定？ — 看典型 task 步数 P95 加 buffer · 一般 15-25 · 必配 cost cap 双保险。
  4. checkpoint 大小膨胀怎么办？ — 大对象（图片 / 长文）存对象存储只存 URL · RemoveMessage 清旧 message · per-thread 设 checkpoint 数 cap · monitor write latency。
  5. resume 跟 replay 区别？ — resume 从 state.next 跑剩余 node（O(remaining)）· replay 从 START 重跑（O(total)）。LangGraph 默认 resume · 只有显式 update_state 后才类似 replay 分支。
