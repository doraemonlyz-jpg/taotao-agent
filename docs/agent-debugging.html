Chapter 1 · 通用流程

5 步通用 agent debug 流程

不论什么症状都先按这 5 步走 · 90% 问题在第 2-3 步就能定位

§1.0核心心法 · agent 不是普通服务

传统 backend service debug：看 log → 找 stack trace → fix。 Agent 不一样——它没崩，只是行为不符合预期。Stack trace 没有任何用，关键证据全在 LLM call 的输入输出里。

所以 agent debug 的第一原则："读 trace 不是 read code"。代码是写死的、能跑就证明无 bug；问题永远在 LLM 看到了什么 → LLM 决定了什么。

§1.1Step 1 · Reproduce + 收集证据

不要相信用户的 bug report——他们说的"agent 死循环"可能是"agent 慢"，他们说的"幻觉"可能是"tool 返回错"。先自己复现一遍。

  1. 拿到用户原 prompt + 复现条件（model / temperature / 时间点）
  2. 在 staging 环境跑同样 input
  3. 同时开 LangSmith / Langfuse trace 录制
  4. 记录 actual output vs expected output 的具体差异（不是"不对"，是"应该是 X 但给了 Y"）

§1.2Step 2 · 读 trace · 找 first divergence

把 trace 从头读到尾，找第一个不该发生的事——后面所有的怪都是这件事的连锁反应。"first divergence" 是 agent debug 最重要的概念。

典型 first divergence：

  • 第 1 步 LLM 误解了 user intent → 后面全跑偏
  • 第 3 步 tool 返回 error 但格式像正常结果 → LLM 把 error 当 fact 用
  • 第 5 步 retrieve 召回错 doc → LLM 基于错 doc reasoning

找到 first divergence 后再问"为什么"——通常一句话能讲清根因。

§1.3Step 3 · 隔离假设 · 二分定位

把可疑因素一个个排除掉。典型可疑因素：

  • LLM——换 temp=0 / 换 model 看是否复现
  • Prompt——把 system prompt 简化到最小 minimal repro
  • Tool——直接 unit test 调 tool 看返回
  • Retrieval——同 query 直接调 retriever 看 top-k
  • State / memory——dump state 看是不是有脏数据

每次只改一个变量。同时改 prompt + 换 model 会让你不知道是哪个解决的。

§1.4Step 4 · 修 + 加 regression test

找到根因后修了——必须把这个 case 加进 eval golden set。否则下次有人改 prompt 又会回归。 LangSmith / Langfuse 都支持一键"把这次 trace 加进 dataset"。

§1.5Step 5 · 类问题广义化 · 加 monitor

1 个用户报这个问题 → 大概率还有 N 个没报。修完后：

  • 跑全量 eval set 看 regression
  • 加监控指标 detect 同类异常（比如"tool returned error but agent treated as success"的 trace 比例）
  • postmortem doc：根因 + 修复 + 同类预防（这本书 §4 的复盘格式）
面试金句

"Agent debug 我永远遵循 5 步：复现收证 · trace 找 first divergence · 二分定位 · 修 + 加 regression test · monitor 同类。第一原则是读 trace 不是改 code——agent 没崩，是 LLM 看到的东西不对，证据全在 trace 里"

§ALoop / Termination 类 (6 个)

A1

死循环 · agent 永远不停

高频
症状

用户提单 task · agent 跑 10 / 50 / 200 步还在调 tool · cost 飞涨 · 用户 retry 加剧

真因（按概率排）
  1. tool 返回 ambiguous · LLM 觉得没拿到答案再调一次
  2. 没设 recursion_limit / max_iterations
  3. supervisor 在两 sub-agent 间反复横跳（A 说让 B 干 / B 说让 A 干）
  4. routing 函数 condition 写错（永远 return 同一节点）
修复
  1. 立刻设 recursion_limit=15-25（LangGraph）/ max_iterations（自写）
  2. tool description 加："如果返回 X 这就是 final answer 不要再调" · 让 LLM 知道何时停
  3. 加 action signature 检测——同 action+args 重复 N 次直接终止
  4. supervisor 显式跟踪 "已调用过哪些 sub-agent"（见 agent-coding §5.1）
预防

monitor "iteration count" P95 · 突变意味着 prompt / tool 改坏了

A2

提前终止 · 没答完就 stop

中频
症状

agent 第 1-2 步就给 final answer · 但答案明显不全 / 跳过了关键 sub-task

真因
  1. system prompt 写得不严——LLM 觉得说一句"好的我帮你做"就算 final
  2. 多 condition routing 提前进了 END · 别的 branch 没跑
  3. structured output schema 让 LLM 觉得"输出就完工"
  4. frontier model 太"自信"觉得自己想了就够（特别是开 reasoning model 后）
修复
  1. System prompt 明确 "Final Answer 只在 X 条件下输出" · 给具体 checklist
  2. output validator: 用小 LLM 验证 "this answer fully addresses the question?" · 否则回 agent 继续
  3. min_iterations 下限——某些 task 必须至少调过某 tool
预防

eval rubric 加 "completeness" 维度 · 不只看 correctness

A3

重复调同一 action

高频
trace 里 search("X") · search("X") · search("X") 调了 5-10 次同样参数
真因
  1. tool 返回的格式 LLM 解析不出来（用了 LLM 不熟悉的 schema）
  2. tool 返回 "no result" 但 LLM 不信
  3. temperature 不 0 · 每次想试运气
  4. 没把 observation append 进 history（agent 看不到自己已调过）
修复
  1. action+args 签名去重 · 重复时返回 "you already tried this exact call, try different args or finalize"
  2. routing/agent 步骤 temperature=0
  3. tool 返回标准 JSON · "no result" 用明确字段而不是空 string
  4. 验证 history 真的累积了（最常见的 bug 是 messages 没 add reducer）
A4

recursion_limit 一直撞

中频
每次都正好跑到 limit 抛 GraphRecursionError · 不是真任务多 · 是真有循环
基本就是 supervisor / router 构造了无限循环
修复
  1. 看 trace 找重复的 node 序列（A→B→A→B→...）
  2. 给 supervisor 加显式状态："research_done · code_done · writer_done" 三 flag
  3. supervisor prompt 里加 "DO NOT call same agent twice"
A5

supervisor 卡同 sub-agent 不放

中频
multi-agent 系统里 supervisor 反复路由到同一个 sub-agent · sub 每次给类似输出 · supervisor 不满意又调一次
supervisor 不知道 sub 已经干过 + sub 给的不是 supervisor 想要的格式
修复
  1. state 里加 "sub_agent_history"——supervisor 看到 "已调过 research_agent 2 次" 自然不会再调
  2. sub-agent 输出加 structured "status: done | needs_more_info | failed"
  3. supervisor prompt 教它什么时候 escalate / FINISH 而不是再调
A6

死锁 · agent 等 HITL · 用户等 agent

低频但严重
interrupt 触发 · 通知用户 · 用户没收到 / 没看到 · agent 永远 pending
notify channel 挂 / UI 没显示 pending state · 没 timeout
修复
  1. 每个 interrupt 设 timeout (e.g. 24h) · 超时 default behavior
  2. 所有 pending interrupt 进 cron 巡检 · stale 自动 escalate / cancel
  3. UI 必须明确显示 "等待你 approve" + 给链接
  4. notify 多渠道（in-app + email + 关键场景 SMS）
监控 "interrupt pending duration" P95 · 突变就是 notify 链路坏了

§BTool / Function calling 类 (6 个)

B1

LLM 不调 tool · 自己编答案

高频
明明给了 search tool · LLM 直接答 "Tokyo 人口大约 1400 万"——是它训练数据来的，没真调 tool
真因
  1. tool description 写得不诱人——LLM 觉得自己知道
  2. system prompt 没说 "always use tools for facts"
  3. 不用 OpenAI function calling · 用 ReAct text 模式 LLM 容易跳过
  4. frontier model 自信过头（GPT-4o · Opus 4.5）经常这毛病
修复
  1. system prompt 写："NEVER state facts without calling search first. Even if you think you know, verify"
  2. tool description 强调 "use this for ALL questions about X"
  3. 用 function calling 不要 ReAct text mode
  4. post-check：output 含数字/事实但 trace 里没 tool call → reject + retry
B2

tool 调用参数错

高频
tool schema 说要 date: "YYYY-MM-DD" · LLM 给 "yesterday" / "2025年8月3日" / null
真因
  1. function schema description 不严格 · 没给 example
  2. 用 ReAct text 没 schema enforcement
  3. open model 比 frontier 弱在这（Llama / Qwen 经常出 schema violation）
修复
  1. schema description 给具体 example "e.g. '2025-08-03'"
  2. 用 strict mode (OpenAI strict: true) / structured output
  3. tool 自己 do 防御 parsing——接受 "yesterday" 就内部解析
  4. schema 验证失败 → 不 raise · 回 LLM "schema error: expected X got Y"
B3

tool 返回 error 被当成功

高频且隐蔽
下游 API 504 · tool 返 {"error": "timeout"} · LLM 当数据用："根据查询结果，timeout 是 ..."
真因
  1. tool 返回结构没区分 success / error
  2. error 字段名跟 success 字段太像
  3. system prompt 没教 LLM 怎么处理 error
修复
  1. tool 返回标准 envelope {"status": "ok|error", "data": ..., "error": ...}
  2. system prompt: "if you see status=error, do not use the data; retry or escalate"
  3. 关键 tool 失败做 explicit fallback 而非靠 LLM 处理
监控 trace 里 "tool error" 但 final answer 没提 error 的 case · 这是隐藏故障
B4

调错 tool

中频
用户问"我的订单"——agent 调了 search_user_profile 不是 get_orders
tool description 模糊 · name 撞车 · 描述里关键词重叠
修复
  1. 每个 tool description 第一句明确 "use this when ..." 限定场景
  2. tool name 用动词 + 名词具体（get_user_orders > search_user）
  3. 负面描述 "DO NOT use this for X" · 帮 LLM 排除
B5

100+ tool · LLM 找不到对的

规模化
tool registry 100 个 · LLM 经常调错 / 调通用 tool 而不是更精确的
prompt 里 100 个 tool description 几千 token · LLM 注意力涣散
修复 = 上 tool retrieval
  1. tool description 也做 embedding · 按 user query top-K (5-20) 检索
  2. LLM 看到的 schema 只是 K 个 · 不是 100 个
  3. 分 namespace · 先 LLM router 选 namespace 再 retrieve
  4. 监控 retrieval recall：LLM 用的 tool 是 top-K 候选吗？不在则改进 retrieval
B6

parallel tool 结果冲突

中频
同时调 3 个 search · 一个说 X · 一个说 Y · 一个说不知道 · LLM 随便选
没显式策略 · LLM 默认 behavior 不可靠
修复
  1. system prompt 教冲突解决："when sources disagree: prefer most recent · prefer official · note uncertainty"
  2. 引入 confidence score · 让 LLM 输出 "based on source X (high confidence)"
  3. 关键场景加 voting agent · 多源印证

§CMemory / State 类 (6 个)

C1

用户串号 · 用户 A 看到 B 的对话

严重
客户投诉 "我看到了别人的订单信息"——P0 安全事件
几乎只有一个：thread_id / session_id 没正确隔离
修复
  1. thread_id = f"{user_id}:{session_id}" · 每用户独立
  2. vector store 按 user namespace 隔离 (Qdrant collection / Pinecone namespace)
  3. 所有 retrieval 强制带 user filter
  4. unit test "用户 A 调 agent 用户 B context 完全不可见"
CI 必须有 multi-tenant isolation test · 这是 P0 防线
C2

跨 session 记忆完全丢

中频
用户说 "我上次告诉过你我的偏好" · agent 完全没印象
没做长期 memory · 或做了但 retrieve 没命中
修复
  1. 引入 long-term memory（Mem0 / Letta / 自建）
  2. 区分 3 类 memory: episodic (events) / semantic (facts) / procedural (preferences) · 各自存
  3. 用户偏好用 structured DB row 不要塞 vector store
  4. 新 session 启动时主动 retrieve relevant memory inject system prompt
C3

context overflow · 老话被挤掉

高频 long session
long session 跑 30 turn 后 · agent 忘了开头说的限制 · 出错
超过 model context window · 老 message 被截 / sliding window 把关键 truncate 了
修复
  1. 用 compaction（让 LLM 自总结历史 · 见 Claude Code /compact）
  2. 关键决定 / 限制 pin 进 system prompt 不可被截
  3. 用更大 context model（Gemini 2M / Claude 200k）
  4. monitor token count P95 · 接近 limit 提前 compact
C4

summary 把关键信息丢了

中频
compaction 后 agent 行为变 · 某些细节"突然忘了"
summarizer prompt 不知道哪些是关键 · 默认丢 specific detail 留 general gist
修复
  1. summarizer prompt 教它必须保留：用户 ID / 限制 / 已确认的决定 / 数字 / 错误反馈
  2. compaction 触发时不丢历史 raw · 存 cold storage 备查
  3. 关键 fact 用 sticky 机制（永远 inject system prompt）
C5

long-term memory 召回错事

中频
用户问"今天天气"——agent 召回了"用户上次说他不喜欢下雨"——莫名其妙
memory retrieval recall 太宽 · 不相关也召回
修复
  1. retrieval 加阈值 · score 低于 X 不返回
  2. memory 加 metadata（topic / entity / time）· 按 metadata filter 而非纯语义
  3. retrieval 后让 LLM 二次判断 "这条 memory 跟当前问题相关吗"
C6

checkpoint resume 后行为变

低频但难找
crash 重启 · agent 从 checkpoint 继续 · 但跟从头跑结果不一样
真因
  1. state 里有非确定数据（time.time() / random / external API result 跟时间相关）
  2. checkpoint 不全（某些 channel state 没存）
  3. checkpoint 序列化丢 metadata（用 pickle vs JSON 行为不同）
修复
  1. state schema 严格 · 不要塞复杂对象
  2. 所有副作用前 capture 进 state · 可重放
  3. 定期 chaos test · kill 进程然后 resume 验证一致

§DOutput / Quality 类 (6 个)

D1

Hallucination · 编造事实

高频
agent 给的事实 / 数字 / 引用看着合理但根本不存在——cite 了不存在的 paper / 编了不存在的 API
真因（按概率）
  1. RAG 召回不到相关 chunk · LLM 还是答了
  2. LLM 没被 ground 到 source · prompt 没说 "based on context only"
  3. retrieve 召回了但 chunk 太碎不连贯 · LLM 自己拼成 plausible 答案
  4. 用 frontier model · 自信过头
修复（5 阶梯）
  1. system prompt: "answer ONLY based on context. If context doesn't have the answer, say 'I don't know'"
  2. structured citation: 强制 LLM 输出 [doc_id] · post-check 引用真存在
  3. retrieval 改 hybrid + rerank · 召回质量上去 hallucination 自然下降
  4. 降 temperature 到 0
  5. LLM-as-judge faithfulness check · 不通过 retry / fallback "我无法确定"
monitor faithfulness score per query · 突变意味着 retrieval 坏了或 model 换了
D2

答非所问 · 走偏题目

高频
用户问 "我的退款啥时候到" · agent 答 "我们的退款政策是 ..."——不是用户问的
真因
  1. RAG 召回的是相关但不是问题答案的 doc · LLM 直接 base 上面答
  2. system prompt 教 LLM "always reference policy" 太狠
  3. 之前 history 太长 · 当前 question 注意力被淹
  4. 有 multi-intent · LLM 答了简单的
修复
  1. output validator: 用小 LLM 评 "this answer addresses the user's question"
  2. retrieve 时区分 "policy doc" 跟 "actual user data" · 两类各 retrieve
  3. prompt 教 LLM "first identify what user is asking, then answer that specifically"
  4. multi-intent 解：先 LLM 拆 query → 各自答 → 合
D3

引用真但答案错

隐蔽
每段都有 [doc_X] 引用 · 看着可信 · 但引用的 chunk 根本不支持那句结论
LLM "找了个 chunk 凑数" · 表面合规实际编造
修复
  1. 引用 post-check：用 NLI model / 小 LLM 验证 "这个 chunk 真的 support 这句话吗"
  2. 不通过 → reject → reroute
  3. 给 LLM 看 chunk 时先做 quote 抽取 · 让它必须引用原文片段
eval 加 "citation faithfulness" 维度 · 不只 answer correctness
D4

structured output schema 验证不过

中频
用 Pydantic / JSON schema · LLM 输出偶尔 schema validation fail · agent 直接崩
真因
  1. 不用 strict mode（OpenAI strict: true 不开）
  2. schema 复杂（嵌套 / Union / Literal 多）· model 偶尔 violate
  3. open model 比 frontier 弱在这
修复
  1. OpenAI strict mode / Anthropic tool use · constrained generation
  2. schema 简化 · 平铺优于嵌套
  3. 验证失败 → 不 raise · 把 error 转成 message 让 LLM retry · 最多 3 次
  4. 用 BAML / Outlines / Instructor 强约束 grammar
D5

输出格式漂移 · 偶尔变样

中频
大多数 response 是 markdown · 偶尔变成纯文本 · UI render 错
temperature 不 0 / system prompt 弱 / context 长 LLM 忘了格式要求
修复
  1. format 要求 pin 进 system prompt 首尾两次 · LLM 注意力两端高
  2. 给 1-2 个 few-shot example 的 expected format
  3. output post-process · 不符 format 自动 fix（regex / LLM）
D6

语言混杂 / 翻译漂移

多语场景常见
用户用中文问 · agent 偶尔用英文答 / 中英混杂
retrieve 的 doc 是英文 · LLM 顺手用 source 语言答
修复
  1. system prompt: "always respond in the user's language regardless of source language"
  2. language detector inject · 明示 "user_language: zh-CN"
  3. output validator 检测 mixed language · retry
  4. 翻译 chunk 进 prompt 前预翻译（成本换准确）

§EProduction / Ops 类 (6 个)

E1

突然全挂 · LLM provider 故障

P0
凌晨 3 点 PagerDuty · 你的 agent 全 5xx · 一查 OpenAI status page 也红
single provider 依赖 · 别人挂你也挂
修复（事中）
  1. 立刻切 fallback provider（提前配好 router · LiteLLM / OpenRouter）
  2. 没配 fallback 就上降级模式 · agent 关掉 · 回到 FAQ-only / 排队
  3. UI 提示 "AI 服务暂时不可用 · 你也可以联系人工"
事前
  1. multi-provider router 做好 + 定期 failover drill
  2. monitor LLM API 5xx rate · 异常自动切
  3. circuit breaker · 连续失败断路防雪崩
E2

偶发 timeout · P99 latency 抖动

高频
99% 请求 OK · 1% timeout · 用户偶尔报 "agent 没回我"
真因（按概率）
  1. LLM provider 自身 P99 抖（OpenAI/Anthropic 一直有）
  2. tool 慢调用没 timeout
  3. retrieval 偶发慢（vector DB cold cache）
  4. specific user 的 prompt 特别长 → 处理特慢
修复
  1. tool 都设 timeout (e.g. 10s) · 超时 fallback 而非阻塞
  2. LLM call 设 hard timeout · 超时 retry 1 次（换 provider）
  3. vector DB warmup + cache 优化
  4. per-request token budget · 长 prompt 拒绝 / 分段
E3

成本一夜暴涨 10x

P0 Bill shock
早晨财务 ping · 昨天 LLM 账单 $50k · 平时 $5k · 出大事
真因
  1. 某个 user / bot 在 abuse（DDOS / 爬数据）
  2. 新 prompt 上线后某场景死循环吃 token
  3. RAG context 突变（doc 巨长不切块）→ 每次几万 token
  4. 有人手抖把 model 从 mini 改成 frontier 上线
修复（事中）
  1. 立刻看 cost-by-user dashboard · 找 top-10 abuser · ban / rate limit
  2. 看 cost-by-route · 哪条 endpoint 涨
  3. 看 cost-by-model · 是不是模型用错
  4. 必要时全局 rate limit 控总量
事前
  1. per-user / per-tenant cost cap · 超阈值自动断 + 通知
  2. per-route cost monitor + alert
  3. model upgrade / prompt change 必走 canary + cost diff check
  4. recursion limit + max tokens hard limit · 死循环最多吃多少有上限
E4

streaming 卡顿 / 断流

UX
用户看到 agent 流到一半停了 · 或反复出现间断
真因
  1. SSE / WebSocket 连接 idle timeout（中间代理 / CloudFlare 默认 30s 关）
  2. 用 astream（node-level）当 token streaming 用
  3. nginx / loadbalancer buffer 没关
  4. tool call 中间没 emit progress · 用户以为挂
修复
  1. 用 astream_events(version="v2") 拿 token-level event
  2. nginx / proxy 关 buffer + 加 X-Accel-Buffering: no
  3. 定期发 keepalive event · 30s 心跳
  4. tool call 期间 emit "still working ..." event
E5

race condition · 高并发数据竞争

低频但严重
同用户 10s 内 send 2 条 message · agent 处理交错 · state 错乱
同 thread_id 并发处理 · 没锁
修复
  1. 同 thread_id 加 distributed lock (Redis) · 串行处理
  2. 第 2 条入 queue 等第 1 条完
  3. UI 禁止用户在 streaming 中再 send（至少 disable input）
  4. checkpointer 用 optimistic concurrency control
E6

prompt 改了 latency 涨 3x

中频 release 后
上线新 prompt 后 P95 latency 从 3s 涨到 10s · 没改 model 没改 tool
真因
  1. 新 prompt 让 LLM 更"啰嗦"——output token 数翻倍 → decode 慢
  2. 新 prompt 让 LLM 多调 1-2 个 tool · 步数变多
  3. 新 prompt 长导致 prompt cache miss
  4. 新 prompt 触发 reasoning model 长 chain-of-thought
修复
  1. 看 trace 对比新旧 · 哪步多了 · output token 多了
  2. prompt 加 "be concise · < N words" 限制 output
  3. prompt cache 重新校准 · 长 prefix 先稳定
  4. 必要时 rollback + 在 staging 修
prompt 上线必走 canary + latency / cost / quality 三 check
完整 30 个总结

这 30 个 failure mode 覆盖生产 agent 99% 的故障。 遇到新 case 时先去 §2.0 索引看症状属哪类，再看对应 mode——大概率不用 reinvent the wheel。 每修一个 bug 后 把 case 加进 eval golden set——这是 senior 跟 mid 最大区别：把每次 incident 转成 regression test，下次不会回归。

Chapter 3 · 读 trace

LangSmith / Langfuse trace · 5 分钟入门

读 trace 是 agent debug 的"读 stack trace"——不会读就 debug 不了

§3.1trace 是什么 · 跟传统 log 区别

传统 log = 时间序列文本。 Agent trace = 带嵌套层级的 LLM call + tool call 完整事件流，每个 span 含：

  • type: chain / llm / tool / retriever
  • input / output: 完整 messages / args / result
  • latency: 这个 span 耗时
  • token: prompt / completion 各多少
  • cost: 算好的 $
  • metadata: model / temperature / user_id / run_id

§3.2读 trace 的 4 个标准动作

动作 1 · 看根 chain 总览

顶层 LangGraph chain · 总耗时多少 / 总 cost 多少 / 总 token / 总 LLM call 数。先有个总账。

动作 2 · 找异常 span

展开 timeline · 看哪个 span 特别长 / 报 error / 重复 N 次。异常 span 99% 就是 first divergence。

动作 3 · 看 LLM call 输入输出

展开异常 LLM call · 看：

  • messages——完整 system / user / assistant / tool messages
  • system prompt 是不是 expected 的版本？（有没有版本错）
  • last user message 是不是用户真说的？
  • 有没有 stale tool message 污染？
  • output——LLM 真说了什么？是不是被你 parse 错了？

动作 4 · 反推 root cause

从异常 span 往上跳父 chain · 看为啥到了这里。 "为啥 LLM 调了 search 而不是 get_orders？" → 看 LLM 看到的 system prompt + tool list + last message → 找到了 retrieve 召回的 chunk 误导。

§3.33 个 LangSmith 高级技巧

  1. Compare runs——把同 input 跑两次（不同 prompt / 不同 model）side-by-side 对比
  2. Add to dataset——异常 trace 一键加进 eval golden set · 修后跑 regression
  3. Custom metadata + filter——给每个 run 打 tag (env=prod / user_tier=enterprise) · 后续按 filter 找同类

§3.4埋 trace 的最佳实践

# LangChain 自动 trace

埋 trace 的 5 条原则
  1. 每 tool 都 @traceable · 别遗漏自定义 tool
  2. 每 run 带 user_id / session_id metadata · 后续按用户 debug 必备
  3. prod / staging 分 project · 不混
  4. 敏感数据 PII 在 trace 前 redact · 否则 LangSmith 存了违规
  5. cost dashboard 配 alert · 异常自动 ping

Chapter 4 · 复盘

5 个真实 agent 生产 incident · 完整复盘

面试问 "你 debug 过最难的 agent 问题" · 这 5 个抄一个改改就能讲

下面 5 个 incident 是 2024-2026 业界真实复盘的合成版本（细节脱敏）。每个用 What / When / Impact / Detection / Root Cause / Fix / Prevention / Lessons 8 段格式——这也是你写自己 incident postmortem 的模板。

Incident 1 · "客服 agent 透露其他用户订单" (P0 安全)

What

客服 agent 在用户 A 的会话里突然显示了 "用户 B 的最近 5 个订单" · 用户 A 在 Twitter 公开截图。

When · Impact

2025 Q3 · 持续 3 天 · 影响 1.2k 用户 · 媒体报道 · 公司股价 -3%。

Detection

用户 Twitter post + 客服内部 ping。内部监控完全没发现——因为没有"跨用户数据泄露"的 detector。

Root Cause

新上线的 long-term memory 服务 · vector store 用 Qdrant 单 collection · agent 检索用 user_id 做 metadata filter。但 Qdrant filter 在某些 query 下被路径优化跳过——返回了所有用户的 chunk。LLM 拿到混合 context 后选了最相关的（恰好是别人的）回答。

Fix (immediate)

  1. 立刻 rollback long-term memory feature · 关 retrieve
  2. contact 受影响用户 · 发 PR 道歉
  3. 提交安全 incident report

Fix (long-term)

  1. 每个用户独立 collection (Qdrant) / namespace (Pinecone) · 不靠 filter 隔离
  2. retrieve 后 post-check 所有 chunk 的 user_id 必须等于当前 user · 不等就 raise + log
  3. CI 加 multi-tenant isolation test (用户 A 跑 agent · 用户 B 数据完全不可见)

Prevention

  1. 新 feature 涉及多租户数据 必须 security review · 不只 code review
  2. vendor library 的 isolation 不可信任（不是 Qdrant 错 · 是用法错）
  3. "defense in depth"——namespace 隔离 + filter + post-check 三层

Lessons (面试金句)

"这件事让我学到 multi-tenant agent 不能只靠 query filter · vector DB 优化路径会绕过 filter。必须用 collection / namespace 物理隔离 + 应用层 post-check 两道。我们后来加了 CI test 强制每次发布跑 isolation 验证"

Incident 2 · "agent 凌晨 3 点把 prod 数据库 truncate 了"

What

SQL agent 给某用户的 query 生成了 TRUNCATE TABLE orders 并执行 · 删了 prod 一整张表的 1200 万行

Root Cause

  1. 用户 prompt: "remove old orders" · LLM 理解为 truncate
  2. SQL agent 的 routing 在前期版本对 SELECT vs WRITE 区分对了 · 后来 refactor 时有 bug 让 WRITE 也直接走 execute · 没经过 HITL
  3. execute 用的 DB role 有 TRUNCATE 权限 (overprivileged)

Fix

  1. 从备份恢复 (4h)
  2. 修 routing bug · WRITE 必走 HITL gate
  3. execute role 降权——只 SELECT + INSERT · 不能 DELETE/TRUNCATE/DROP
  4. destructive op 强制 LLM 输出 dry-run + 估影响行数 + HITL approve · 才执行

Lessons

"两个错误叠加才能 prod 出事——routing bug + overprivileged role。任意一道防线都能拦住。从此我们坚持 'least privilege + defense in depth' 双原则——所有 destructive op 必须既有 HITL 又有 DB 层 permission 拦截"

Incident 3 · "新 prompt 上线第二天账单暴涨 12x"

What

周二上线了改进版 system prompt · 周三早上财务 ping · 昨日 LLM 账单 $48k · 平时 $4k。

Root Cause

新 prompt 加了一句"think step by step before answering"——本意是提质量。结果：

  1. LLM output token 数从平均 200 涨到 800（多了 thinking 段）
  2. multi-turn 会话累积——下 turn 把上 turn 的 thinking 也带进去 · prompt cache miss
  3. 叠加 cost 12x

Detection · 太晚

没有 cost diff alert · 财务发现时已花了 24h。

Fix

  1. 立刻 rollback prompt
  2. 改进版本：thinking 用 <thinking> tag · 输出后 parse 掉不进 history
  3. cost canary：新 prompt 必先在 1% 流量跑 · 看 cost diff > 20% 自动 block 上线
  4. monitor cost-by-prompt-version · alert on spike

Lessons

"prompt 改一行能让账单 10x · 现在我们任何 prompt change 都走 canary + cost diff check + multi-turn cost simulation。prompt 改动跟模型升级一样严肃，不是 'just text'"

Incident 4 · "agent 死循环跑了 200 步 · 单 task 烧掉 $400"

What

research agent 给一个开放式问题 · 跑了 200 步 search · 1.2M token · $400 单 query · 用户体验是"等了 30 分钟没结果"

Root Cause

  1. 没设 recursion_limit · 默认 25 但被 override 到 999
  2. search tool 偶尔返 "no result" · LLM 不信 · 改 query 再试 · 50 个变体
  3. 没有 cost cap per query

Fix

  1. recursion_limit 强制 30 · 任何 override 走 review
  2. cost cap per query (e.g. $5) · 触达即终止 + 给 partial result
  3. search tool 返 "no result" 时给明确 hint "try different angle or accept absence"
  4. action signature 检测 · 同 search query 已试过则 skip

Lessons

"单 query $400 是因为 3 道防线全没——recursion · cost cap · 重复 detection。每道单独都能拦住。从此我们坚持 'budget triple lock'：iteration limit + token limit + dollar limit · 三个都设 hard"

Incident 5 · "audit log 显示 agent 给了高管错答案 · 影响商业决策"

What

CEO 用内部 BI agent 问 "Q3 north america sales" · agent 给的数 $42M · 内部财报真是 $52M · CEO 在 board 引用错了。

Root Cause

SQL agent 选了一个错的 table——选了 orders 不是 orders_completed。两表 schema 几乎一样但语义不同（orders 含未完成 / 取消）。LLM 没区分。

Fix

  1. 立刻通知 CEO + 给正确数
  2. schema retrieval 改进——给 LLM 看 table description (含 "use orders_completed for revenue queries")
  3. 关键 metric 用 metric layer 包装 · 不让 agent 直选 table · 走预定义 metric
  4. 所有 BI agent 输出强制带 source attribution · 高管能审 "数从哪来"
  5. 关键 query (revenue / cost / 财务) 强制 HITL review (data analyst approve)

Lessons

"text-to-SQL 在多表语义相近时极易选错。我们的解决：把核心 metric 包装成 metric layer——agent 不直接 join 表 · 调 'get_revenue(region, period)' 这种确定性 wrapper · LLM 选不出错。核心商业决策 + agent 必须有 audit + HITL · 不接受 silent fail"

每个 incident 都不是单点 bug · 而是多道防线同时缺失。 Senior 写完 fix 还要写 prevention——把这一类问题永久消灭。 —— 这本手册
