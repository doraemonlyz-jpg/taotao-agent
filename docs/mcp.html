MCP · 给 agent 一根 USB-C

Anthropic 在 2024-11 提出的 Model Context Protocol，把"agent ⇄ 工具源"的连接方式标准化。 这一页讲三件事：① MCP 解决了什么问题、协议长什么样； ② taotao-agent 怎么双向实现的（既当 server 把工具吐出去，也当 client 把别人的工具吃进来）； ③ 怎么把 Claude Desktop / Cursor 接到我们这个 server。

§ 1没有 MCP 之前 · agent 的 N×M 焦虑

你写了一个 agent，想让它能：① 读你电脑上的文件；② 查 GitHub issue；③ 操作 Notion；④ 跑数据库查询。 在 MCP 之前，每个工具源你都得手写一个 LangChain Tool 包装器，把 SDK 调用塞进去， 还得管认证、错误处理、参数校验。换一个 agent 框架，又得重写一遍。

这就是经典的 N×M 集成爆炸：N 个 agent 框架 × M 个工具源 = N×M 份连接代码。 Anthropic 看到这个痛点，从 LSP（Language Server Protocol，VSCode/IDE 那套）抄了思路 —— 定义一个协议，让"工具源"实现一次，"agent 端"调用一次，剩下全靠协议适配。 MCP 就这么来的。

一句话定义

MCP = JSON-RPC 2.0 + 三个 server 能力（tools / prompts / resources）+ 两种传输（stdio / streamable_http）。 Anthropic 主推、OpenAI 也加入了，2025 上半年起 Cursor / Cody / Claude Desktop / Continue 全都内置 MCP client。

§ 2三个核心概念 · 一张表认完

能力是什么典型场景
Tools 带 JSON Schema 参数的可被调用的函数。模型决定何时调、传什么参数。 查文件、跑 SQL、发推、改 Notion 页面
Prompts 预先写好的、参数化的系统提示模板。用户/IDE 主动选用，不是 LLM 自己决定。 "用 PR-review 模式审这段代码"，"按公司 commit 规范写 message"
Resources 带 URI 的只读数据。client 可以拉、可以订阅变更。比 tool 轻 —— 不需要 LLM 决策就能注入上下文。 当前文件内容、最近的日志、用户 profile

Tool 和 Resource 区别（很多人搞混）

Tool 是动作 —— LLM 看到 schema、自己决定要不要调、传什么参数。 Resource 是状态 —— 由 client（不是 LLM）主动拉取、放进 context window。 前者像调函数，后者像读文件。同一份数据可以同时作为 tool（"refresh_logs"）和 resource（"file://logs/today.log"）暴露。

§ 3协议长什么样 · JSON-RPC 一帧帧看

MCP 用 JSON-RPC 2.0，每条消息都是 {"jsonrpc":"2.0","id":N,"method":"...","params":{...}}。 下面是一次完整握手 + 工具调用的 trace（这就是 curl 我们 /mcp/ 端点能看到的东西）： 

→ initialize
{"jsonrpc":"2.0","id":1,"method":"initialize",
 "params":{"protocolVersion":"2025-06-18",
           "capabilities":{},
           "clientInfo":{"name":"cursor","version":"0.45"}}}

← server hello (告诉 client 自己有哪些能力)
{"jsonrpc":"2.0","id":1,"result":{
   "protocolVersion":"2025-06-18",
   "capabilities":{"tools":{...},"prompts":{...},"resources":{...}},
   "serverInfo":{"name":"taotao-agent","version":"1.27.1"},
   "instructions":"Tools cover math, time, web search, ..."}}

→ tools/list
{"jsonrpc":"2.0","id":2,"method":"tools/list","params":{}}

← 工具清单 + JSON Schema
{"jsonrpc":"2.0","id":2,"result":{"tools":[
   {"name":"calculator","description":"Evaluate a math expression...",
    "inputSchema":{"type":"object","properties":{"expression":{"type":"string"}},
                   "required":["expression"]}},
   ...]}}

→ tools/call
{"jsonrpc":"2.0","id":3,"method":"tools/call",
 "params":{"name":"calculator","arguments":{"expression":"2**16+7"}}}

← content blocks（可以混 text / image / resource link）
{"jsonrpc":"2.0","id":3,"result":{
   "content":[{"type":"text","text":"2**16+7 = 65543"}],
   "isError":false}}

就这么直白。没有魔法 —— 任何能发 HTTP/stdio + 写 JSON 的程序都能当 MCP client 或 server。

§ 4两种传输 · 选哪个

stdio

Client（Claude Desktop / Cursor）spawn 一个子进程， 双方用 stdin/stdout 收发 JSON-RPC，每行一帧。

用于：本地工具，安全靠"进程在用户身份下跑"自然解决。
启动：uv run python -m agent.mcp

streamable_http

MCP server 是一个普通的 HTTP 服务，client 往 /mcp/ 发 POST， 每个响应可以是单个 JSON 也可以是 SSE 流。

用于：远程工具，需要走网络的场景。
启动：跟我们 FastAPI 一起，挂在 /mcp。

还有一个 SSE 传输（已弃用）

早期 MCP 用 SSE + 单独的 messages 端点（双信道），后来发现挺别扭，2025-03 协议版换成了 streamable_http —— 单个端点既支持单次 JSON 响应、也支持 SSE 流。新代码一律用 streamable_http，老代码看到就当 fallback 兼容。

§ 5我们的服务端实现 · backend/agent/mcp/server.py

用官方 mcp Python SDK 的 FastMCP 高层 API，~100 行讲完。核心三步：

① 把 LangChain 的 @tool 翻成 MCP tool

LangChain tool 已经有 name / description / args_schema（Pydantic）， 我们用 inspect 重建一个 Python 函数签名，再交给 FastMCP.add_tool()。 FastMCP 自己会从签名推 JSON Schema 给 client 看。 

def _wrap_tool(t: BaseTool) -> Callable:
    schema = t.args_schema
    fields = getattr(schema, "model_fields", {})
    params = [
        inspect.Parameter(name, KW_ONLY, default=..., annotation=...)
        for name, finfo in fields.items()
    ]
    sig = inspect.Signature(parameters=params, return_annotation=str)

    def runner(**kwargs):
        return str(t.invoke(kwargs))

    runner.__name__ = t.name
    runner.__doc__ = t.description
    runner.__signature__ = sig   # FastMCP 看的就是这个
    return runner

② 白名单 · 不是所有 tool 都该被任何 client 调

默认只暴露只读、安全那一组：calculator / current_time / web_search / read_file / list_files / grep_in_files / recall / read_profile / load_skill。 write_file / python_repl / remember 这种有副作用的必须显式开。 

# .env
MCP_EXPOSED_TOOLS=calculator,current_time,web_search   # 自定义白名单
MCP_EXPOSED_TOOLS=*                                    # 全开（仅信任环境）

③ 两种传输共享同一个 FastMCP 实例

stdio · 给 Claude Desktop 用

# backend/agent/mcp/__main__.py
from .server import run_stdio
run_stdio()
# ↑ uv run python -m agent.mcp

HTTP · 挂到 FastAPI

# backend/app.py
_mcp = build_mcp_server()
app.mount("/mcp", _mcp.streamable_http_app())

@asynccontextmanager
async def _lifespan(_app):
    async with _mcp.session_manager.run():
        yield  # 这一行是关键

坑 · ASGI mount 不会传 lifespan

FastMCP 的 streamable_http app 内部用一个 session_manager 跑 anyio task group， 必须在 lifespan 里 async with。但 Starlette 的 app.mount() 默认不把外层 FastAPI 的 lifespan 事件传给被 mount 的子 app —— 结果就是 task group 永远不启动， 每次请求 500: "Task group is not initialized"。 修复方式是把 MCP 的 session_manager 嵌进 FastAPI 的 lifespan 里，亲自启动。

§ 6我们的客户端实现 · backend/agent/mcp/client.py

用 langchain-mcp-adapters（LangChain 官方维护）。配置文件复用 Claude Desktop 那套 {"mcpServers": {...}} 格式，意味着你能直接把 Claude Desktop 的 config 拷过来。 

// backend/mcp_servers.json  (gitignored)
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"]
    },
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    },
    "remote-thing": {
      "url": "https://api.example.com/mcp/",
      "transport": "streamable_http",
      "headers": {"Authorization": "Bearer $TOKEN"}
    }
  }
}

启用就是一行环境变量：

# .env
MCP_CLIENT_CONFIG=./mcp_servers.json

启动时 load_external_tools() 会一次性把所有外部 server 的 tools 拉下来， 命名为 <server>__<tool>（例如 filesystem_read_file），合并进 all_tools 和 HARNESS_TOOLS。 对图节点和 harness 完全透明 —— 它们看到的就是更多可调用的 LangChain tool。

三个非显然的工程决策

  • 命名前缀防冲突。我们自己有 read_file，filesystem MCP server 也有 read_file。 adapter 的 tool_name_prefix=True 让冲突不可能发生。
  • async-only 怎么办。MCP-client 工具只实现 ainvoke()，sync .invoke() 会 NotImplementedError。我们在 safe_run_tool 里 try/except，回退到 asyncio.run(tool.ainvoke(...))， 因为 ThreadPoolExecutor worker 线程没自带 loop，开一个临时的就行。
  • content blocks 拉平。MCP 返回的是 [{type:"text", text:"..."}] 数组（为了支持 image / resource link）。 我们在 safe_run_tool 把 text-only 数组拉平成纯字符串，LLM 看到的就是 "9*9 = 81" 而不是一坨 JSON。 非 text 的（图片）保持原样。

§ 7三种用法 · Claude Desktop / Cursor / curl

用法 ①：把 taotao-agent 接进 Claude Desktop

编辑 ~/Library/Application Support/Claude/claude_desktop_config.json：

{
  "mcpServers": {
    "taotao-agent": {
      "command": "uv",
      "args": ["--directory", "/Users/you/Code/agent-demo/backend",
               "run", "python", "-m", "agent.mcp"]
    }
  }
}

重启 Claude Desktop，对话框旁边会出现工具图标，里面就有我们的 9 个白名单 tool。

用法 ②：把 taotao-agent 接进 Cursor

Cursor → Settings → MCP → Add new MCP server，同样的 stdio 命令。或走 HTTP：

{
  "mcpServers": {
    "taotao-agent": {
      "url": "http://127.0.0.1:8000/mcp/",
      "transport": "streamable_http"
    }
  }
}

用法 ③：curl 直接调试

# 列工具
curl -s -X POST http://127.0.0.1:8000/mcp/ \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}'

# 调工具
curl -s -X POST http://127.0.0.1:8000/mcp/ \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":2,"method":"tools/call",
       "params":{"name":"calculator","arguments":{"expression":"2**16+7"}}}'

自循环测试

把 backend/mcp_servers.example.json 里的 loopback-self 块单独拷出来当配置， 我们的 client 会去调我们的 server，这是验证整条链路最干净的方法 —— 不依赖任何外部进程。

§ 8面试 5 问 · 答得出来基本算懂了

Q1 · MCP 跟 OpenAI function calling 区别是什么？
function calling 是模型层能力（"模型能输出 tool_call JSON"）。MCP 是系统层协议 ("agent 怎么发现外部工具源")。两者正交：MCP server 把工具暴露给 agent，agent 拿到这些 tool 还得通过 function calling 让模型决定用哪个。 类比：function calling 是 USB 协议本身，MCP 是 USB 集线器。
Q2 · 既然有 LangChain Tool 抽象，为啥还要 MCP？
LangChain Tool 是语言内抽象（Python 函数）。 MCP 是跨语言、跨进程协议 —— Notion 团队用 TypeScript 写一个 MCP server， 你用 Python agent 直接消费，不需要重写 SDK 包装器，也不需要把 Notion SDK 装进你的 venv。 本质是把工具源做成独立可部署的微服务。
Q3 · stdio 和 HTTP 该怎么选？
本地工具（要访问你电脑文件、跑你电脑上的 git）必须 stdio —— HTTP 服务跨不了用户身份边界。 远程工具（公司内部 API、SaaS 集成）必须 streamable_http —— 不能让用户机子去 spawn 服务端进程。 混合场景（Claude Desktop 调你公司的 API）：本地装一个 stdio MCP server 当"代理"， 它内部再去调远程 HTTP，是一种常见 pattern。
Q4 · MCP server 暴露 write_file / execute_shell 这种 tool 的安全风险？
极大。MCP client 是任意进程 —— 只要能连上你的端点（stdio 进程被启动 / HTTP 没认证）， 它就能让一个 LLM 帮它做任何 server 暴露的事情。Prompt injection 在 MCP 时代被放大成了横向移动漏洞。 缓解：① 默认白名单只放只读 tool（我们就是这么做的）；② HTTP 传输强制 Authorization token； ③ 副作用类 tool 要求 client 二次确认（MCP 协议有 elicitation 机制可以反向问 client）； ④ 进程沙箱 + 文件路径白名单。
Q5 · 大量 MCP tool 接入会不会撑爆 context window？
会。每个 tool 的 schema + description 都会塞进 system prompt，10 个 server 各 20 个 tool 就是 200 个 schema， 几 K token 起步。两种应对：① 渐进式披露（progressive disclosure）—— 先只塞一个 list_tools meta-tool，用了再展开；② 语义路由（我们 graph 那个 select_tools 节点就是干这个的）—— 每轮按用户输入 embedding 召回 top-K tool 才喂进去。两个方案各有 trade-off，详见 tool-design.html § 5。
