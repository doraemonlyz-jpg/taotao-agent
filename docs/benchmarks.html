看 5 个 leaderboard · 看 5 种陷阱 · 不被市场宣传忽悠

"我们家模型 SWE-bench 65% 第一。" 这句话能在 2026 年的 launch tweet 看到 5 次/周。 但 leaderboard 数字背后藏着一堆陷阱——sample selection / harness drift / contamination / oracle context。这一本教你怎么读每个主流 agent benchmark · 每个陷阱 怎么识破 · 自己 ship 模型怎么不被这些坑反噬。

10 章 · 30 题 · 60 词术语 · 阅读 ~50 分钟

§ 1为什么 senior 必须会读 benchmark

"MMLU 88 / HumanEval 95" 是 base model 时代的指标，不能预测 agent 任务表现。 Agent 任务有 4 个维度 base benchmark 测不到：

  1. 多步执行：错一步就全错 · ε⁻¹ 衰减
  2. 工具使用：知不知道什么时候调 · 不只是会不会调
  3. 长上下文检索：50 turn 后能不能 recall 最早的 spec
  4. 真实世界副作用：rm -rf / 多支付一次 / 删 prod DB

所以工业界发明了一系列 agent-native benchmark：SWE-bench (代码) · GAIA (通用助手) · BFCL (工具调用) · τ-bench (对话工具) · WebArena (网页) · OSWorld (桌面)。

面试会问："你怎么判断一个新模型在你的产品里好不好用？" 说"跑 leaderboard 看分数"会被扣分——senior 答案要含"哪些 benchmark 适用 / 不适用 · 数字之外还看什么"。

§ 2SWE-bench · 代码 agent 的事实标准

Princeton 2023 提出。从 12 个流行 Python 开源项目（Django / sympy / scikit-learn 等）的 真实 GitHub issue + PR 抽 2,294 条，每条任务：

  1. 给 agent: issue 描述 + 仓库 codebase（commit 在 fix 前那个状态）
  2. agent 输出: 一份 patch (.diff)
  3. 评判: 把 patch apply 上去，跑那个 issue 对应的 PR 里新加的测试，pass 算成功

2.1 三个变种 · 必须分清

变种条数难度2026 SOTA
SWE-bench Full2294原始 · 含很多模糊 / 不可解~50%
SWE-bench Verified500OpenAI 人工筛过 · 工业标杆~70%
SWE-bench Lite300更短 / 1 个文件就能 fix~75%
SWE-bench Live逐月新增2024-09 后的新 issue · 防 contamination~50%

最易错的细节

看到 "SWE-bench 65%" 必须问"哪个 split"。Full / Verified / Lite 数字不能直接比。 2025 起 Anthropic / OpenAI / Cursor 公布 score 默认指 Verified· 但开源模型 paper 经常报 Full（数字小但可比性差）或 Lite（数字大但容易吹水）。

2.2 "harness" 比模型更重要

同一个模型 GPT-4o · 在 SWE-bench Verified 上：

  • 裸 ReAct loop · ~24%
  • Aider 的 search-replace harness · ~31%
  • OpenHands (前 OpenDevin) · ~38%
  • Anthropic 自家 harness · ~49%

同模型差 25 个点——所以读 leaderboard 必须看 harness： "Claude 4 + ourharness 70%" vs "Claude 4 + Aider 56%" 完全不矛盾。 这也意味着 "X 模型 SWE-bench 65%" 不能等同于 "X 模型解决你公司 bug 65%"， 因为你的 harness 不一样。

2.3 SWE-bench 的 5 大局限

  • 只测 Python · 不测 Go / Rust / TS / 多语言
  • 只测 backend / lib · 不测前端 / UI / 异步并发
  • 测试本身就是 oracle—— agent 不需要写新测试
  • contamination：很多 base model 训练数据见过这些 PR diff
  • "成功" 二元判断 · 没区分"小修小改成功"和"重构整个模块"

§ 3GAIA · General AI Assistant

Meta 2023 发布的"通用助手" benchmark · 466 题，是 真实人类问的问题， 需要多步推理 + 工具（web/file/code）+ 多模态才能答。

3.1 难度三档

Level典型步数例子
Level 11–5 步"哪个国家在某届奥运会上获得最多金牌？" (web 一查)
Level 25–10 步"在这张图找到 ___，再去对应的 wiki 找 ___ 比例"
Level 310+ 步 · 多模态"分析这份 PDF · 找出表格里 X · 跟 web 上 Y 比，输出 markdown 报告"

3.2 SOTA 路径

2024 GPT-4 + 简单 harness · ~14% · 2025 H1 GPT-4o + Magnetic-One (多 agent) · ~38%· 2025 末 Claude 3.7 Sonnet 单 agent + 强 harness · ~52% · 人类基线 ~92%。

为什么 GAIA 重要

大部分 benchmark 是单步 / 单工具，GAIA 强制 agent 表现"端到端规划 + 多工具串联" 能力。跟你公司里"用户问个真实问题"的 distribution 最接近，所以 senior 面试拿 GAIA 当例子讲 evaluation 通常很受欢迎。

§ 4BFCL · Berkeley Function-Calling Leaderboard

UC Berkeley 2024 发布。专门测tool calling 准确度——给一个 prompt + 一组 tool schema， 模型应该输出哪个 tool 名 / 什么参数 / 是不是 parallel。已经迭代到 V3。

4.1 7 类子任务

类别测什么
Simple1 prompt → 1 tool · AST equality 判分
Multiple1 prompt → 选对的那个 tool（多 tool 候选）
Parallel1 prompt 同时调多个 tool
Parallel Multiple同时调 + 多候选
Irrelevance没有合适 tool · 模型应该回 "I can't"
Live用户真实 query · 不是合成的
Multi-turn对话连续多 turn · 测 state tracking

4.2 怎么读 BFCL 排名

  • 不要只看 overall。Live + Multi-turn 跟 Simple 差距常常 30+ 分 · 真实场景看 Live。
  • 注意 cost 列。BFCL 公开每个模型每条 sample 的 token 使用量。GPT-4 / Claude 强但贵 · Llama-3.1 70B 性价比高。
  • Irrelevance 别忽略。模型敢不敢说 "no" 是工业落地最痛点之一—— score 高的模型在生产里 hallucinate tool 调用的几率小很多。

2026 趋势

Hermes 4 / Qwen 2.5 / Llama 3.1 等开源模型在 BFCL Simple 已经接近 GPT-4o · 但 Multi-turn 仍然落后 8–15 分。开源 tool-calling 还没"完全追平"， 生产选模型时不能只看 Simple 数字。

§ 5τ-bench · 真实场景对话 + 工具

Sierra (Bret Taylor 公司) 2024 发布。两个领域 —— retail (165 task) + airline (50 task)。 Agent 跟另一个 LLM 模拟的用户对话 · 可以调 retail/airline domain 的 ~50 个 tool · 最后 evaluator 检查 DB state 是否变成预期。

5.1 跟 BFCL 的本质区别

BFCL 测"这一句话 该调哪个 tool"——单步 · zero-shot。 τ-bench 测"整个对话过程能不能稳"——多 turn · 状态机 · 用户会说不清楚 · 会改主意 · 会 push back。更像真实客服。

5.2 重要的指标 · pass^k

τ-bench 报 pass^k：同一 task 跑 k 次 · 全部通过算成功。 一般 k=8。这是稳定性指标——单次成功率 80% 的模型 pass^8 可能掉到 40%。 pass^1 在生产没意义，因为用户会重试看到不一致就走人。

5.3 SOTA

Claude 3.5 Sonnet pass^1 ~70% / pass^8 ~46% (retail) · GPT-4o pass^1 ~62% / pass^8 ~35%。 Anthropic 在 τ-bench 上一直 lead · 跟 Claude 在 multi-turn / instruction following 上的强项一致。

§ 6WebArena · 浏览器 agent

CMU 2023 发布。在 4 个真实 web app（OneStopShop / Reddit-like / GitLab-like / Map）的本地副本里 · 给 agent 800+ 任务（"找出本月销量第二的商品"等）。Agent 输入是 HTML/screenshot · 输出是 click/type/scroll 等 action。

6.1 变种

  • WebArena: 文本 HTML 输入 (DOM tree)
  • VisualWebArena: 加 screenshot · 测 VLM 能力
  • WebArena Live: 真实 production 网站 (有 OAuth 等)

6.2 SOTA & 注意

2024 年初 GPT-4 ~14%。2025 末 Claude 3.7 + 强 harness ~50%。仍远低于人类 78%。 读 WebArena 数字必须看 harness 类型——是纯 DOM 还是 set-of-marks （把 screenshot 上每个 element 标号）。后者数字会高 10 分以上。

§ 7OSWorld · 桌面 computer-use

THU + CMU 2024 发布。在 Ubuntu / Windows / macOS VM 里跑 369 个真实任务 （用 GIMP / Office / VSCode / Firefox 等），全部跨多个应用 / 文件系统。

Agent 输入 screenshot · 输出 pyautogui-style action。2025 H2 Claude Sonnet 4 + Computer Use ~22%（人类 ~72%）。差距巨大 · 桌面 agent 还远没 ready · 这是为什么 2026 Cursor / Devin 路线大多还停在 IDE 内（受控 env）而不是 full desktop。

§ 85 大读 leaderboard 陷阱 · 必识破

8.1 陷阱 ① · Contamination（污染）

模型训练数据见过 benchmark 答案。SWE-bench Full 大量 PR 是 GitHub 公开的 · base model 预训练时几乎肯定见过 fix patch。修法：看 SWE-bench Live（持续新增）· 或 GAIA（人审过 · 部分 hidden）· 或自己内部 holdout。

8.2 陷阱 ② · Harness drift

"X 模型在 SWE-bench Verified 70%" —— 别忘了问 "在哪个 harness 上"。 同模型不同 harness 差 25+ 分。实务读法： 只比较"同 harness 不同模型"的差值，绝对值意义不大。

8.3 陷阱 ③ · Sample selection

官方 leaderboard 默认 deterministic sampling (temp=0)，但有些公司 paper 报"best-of-N" 或"self-consistency"——同 prompt 跑 5 次取最好 · 数字 +5–10 分。 读法：搜 "pass^k"、"sampling"、"k=" 这些字眼，弄清是单次还是多次。

8.4 陷阱 ④ · Oracle context

SWE-bench 有"oracle" setting —— 提前告诉 agent "你应该改 file_a.py 和 file_b.py" · 这跳过了 file-localization 这个最难的步骤 · 数字会高 10–20 分。 看到 "oracle" 字样直接打折看。

8.5 陷阱 ⑤ · Cost / token / latency 不公布

GPT-4 在 SWE-bench Verified 跑一条平均 $4 + 12 分钟。"开源 7B 跑出 30%" 听上去美好 · 但你算上每条 200K token 输入和 50 turn loop · 实际跑 100 题成本可能跟 GPT-4 一样 · 还慢 3 倍。没有 $ + s 数字 的 leaderboard 必须打问号。

综合查表 · 读到一个新数字时问自己 5 个问题

  1. 哪个 split / level / variant？
  2. 什么 harness？跟你的 harness 接近吗？
  3. 几次采样？pass^1 还是 pass^k？
  4. 有没有 oracle context / hint 注入？
  5. cost · latency 公布了吗？

§ 930 道 senior 面试题

SWE-bench

  1. Full / Verified / Lite / Live · 各自适用什么场景？
  2. "X 模型 SWE-bench 60%" — 你会问哪 5 个 follow-up 问题？
  3. SWE-bench 不能测什么类型的 bug？
  4. 同模型不同 harness 差 25 分 · 这告诉你什么？
  5. 怎么防止评 SWE-bench 时被 contamination？
  6. SWE-bench Verified 是怎么从 Full 筛出来的？
  7. OpenHands / Aider / Cursor harness 区别在哪？

GAIA

  1. GAIA Level 3 跟 Level 1 难度差在哪？
  2. 为什么 GAIA 比 MMLU 更适合预测 agent 真实表现？
  3. 多 agent (Magnetic-One) 比单 agent 在 GAIA 上强 · 为什么？

BFCL

  1. BFCL 7 类子任务 · 哪类最贴近生产？
  2. Simple 高 / Multi-turn 低的模型在生产会怎样？
  3. Irrelevance 测什么？为什么生产很重要？
  4. 开源模型在 BFCL 还差 GPT-4o 哪一项？

τ-bench

  1. τ-bench 跟 BFCL 本质区别是什么？
  2. pass^k 为什么比 pass^1 重要？
  3. pass^8 = 50% 的模型生产上能不能用？

WebArena / OSWorld

  1. WebArena 跟 VisualWebArena 测什么不同的能力？
  2. set-of-marks 怎么帮 agent 提分？为什么是有 cost 的？
  3. OSWorld SOTA 还远低于人类 · 这说明什么生产 readiness？

读 leaderboard 方法论

  1. 5 大陷阱列举 + 各 1 个例子。
  2. oracle context 是什么？看到怎么处理？
  3. "我们家模型 contamination-free" 你怎么验证？
  4. 同 model 不同 paper 报不同数字 · 5 个原因。
  5. 给你一份新发布的 agent paper · 你前 5 分钟看什么？
  6. cost / latency 一栏没公布 · 你怎么估？
  7. 怎么判断一个 leaderboard 是否还可信？

实操 · 你公司

  1. 你公司想从 GPT-4o 换 Claude 4 · 你会跑哪 3 个 benchmark + 哪个内部 holdout？
  2. 设计一个跟 SWE-bench 类似但适合你 codebase 的内部 benchmark · 4 步流程。
  3. 新模型在所有 public benchmark 都 +5 分 · 但你公司内部 holdout 掉 3 分 · 你怎么解释？

§ 11下一步 · 把这套用起来

  1. 把当前你公司用的模型在 SWE-bench Verified / GAIA / BFCL Live 三个 leaderboard 各去查一下排名 · 写 1 页内部 brief
  2. 挑公司内部 1 个真实场景 · 抽 100 条 · 做成 holdout · 对比 leaderboard 数字看差距
  3. 写一份 "新模型评估 5 步法" SOP · 团队复用
  4. 面试时被问 "X 模型怎么样" · 用 § 8 的 5 问反问 + 自己内部 holdout 的数据回答
