What you're looking at, before the microscope

§1.1What runs when you say hi

最短路径——一句无工具的闲聊会经过 5 个节点，调 1 次大模型， 并在后台启动 1 个异步 extractor。trace 面板里会按时间顺序冒出这些事件：

START │ ▼ summarizer // noop — 消息 ≤ 24 条 │ ▼ perception // 注入 system prompt + profile + 3 facts + 2 reflections │ ▼ input_guardrail // 提示注入正则扫一遍 → pass │ ▼ planner // trivial 启发式短路 → route=direct │ ▼ executor // 大模型直接答 + 流式 token │ ▼ critic // 启发式跳过 (短答案 + 0 工具) │ ▼ extractor // 启发式判定 → 跳过 (此轮无值得记忆的内容) │ ▼ output_guardrail // PII 正则扫一遍 → pass │ END

感知到的延迟主要来自一个 LLM 调用的 prefill + decode。其余节点都是几毫秒到几十毫秒级。

II Part Two — State

The plumbing every node reads & writes

一个 TypedDict，几个 reducer，决定了图能不能写得清爽

LangGraph 0.6 鼓励用 TypedDict（不是 Pydantic）做 state—— 因为它要在节点之间序列化、要走 checkpointer 落盘、要容忍部分字段缺失。 Taotao Agent 的 AgentState 长这样：

backend/agent/state.py

class AgentState(TypedDict, total=False):
    # --- 5. Perception input ---
    user_input: str
    attachments: list[dict[str, Any]]

    # --- 1. LLM messages (short-term memory) ---
    messages: Annotated[list[AnyMessage], add_messages]   # reducer: append

    # --- 4. Planning ---
    plan: list[str]
    current_step: int
    route: Literal["direct", "supervisor", "research", "code", "write"]

    # --- 6. Action / tool execution receipts ---
    tool_calls_made: int
    routed_tools: list[str]      # which tools the executor saw this turn

    # --- O. Orchestrator bookkeeping (single-turn) ---
    subagent_history: list[str]  # reset by perception every turn

    # --- Reflection ---
    critique: str
    critique_passed: bool
    revisions: int

    # --- Guardrails ---
    blocked: bool
    block_reason: str

    # --- Long-term memory hits (retrieved on demand) ---
    recalled_memories: list[str]
    recalled_reflections: list[str]

    # --- Rolling summary of compacted older messages ---
    summary: str

    # --- Final ---
    final_answer: str

    # --- Observability ---
    trace: Annotated[list[TraceEvent], add]
    session_id: str

§2.1两个 reducer 是关键

大部分字段是替换语义——后写覆盖前写。但两个字段是累加语义，靠 reducer 实现：

  • messages: Annotated[..., add_messages]——LangGraph 内置 reducer， 合并消息列表的同时支持 RemoveMessage 标记和去重
  • trace: Annotated[..., add]——纯列表拼接（用 operator.add）， 每个节点 emit 的事件都追加到尾部
trap I fell into

想给 subagent_history 也加 add reducer——错。 因为 reducer 跨 turn 也累加，第二轮就会带上第一轮的历史，导致 supervisor 误判。 改成普通 list 字段 + perception 每轮显式 reset 为 []， 每个 sub-agent 自己 list(state[...]) + [my_name] 全量回写。 简单直接，不用记 reducer 行为。

§2.2SqliteSaver 持久化

state 走 AsyncSqliteSaver 落到 data/checkpoints.sqlite， 所以 backend 重启不丢会话——前端用同一个 session_id 接回去就能续。

III Part Three — Perception

The context-engineering node

Agent 的"感知"其实是把对的东西放到对的位置——而不是看 webcam

Perception 不解析图像、不识别意图。它做的是 context engineering—— 为本轮对话拼好喂给 LLM 的消息列表。它要决定：

  • 哪些静态内容应该缓存？（system prompt + skills index）
  • 哪些动态内容应该每轮重算？（profile 快照、retrieved facts、reflections、当前问题）
  • 什么时候用 HyDE 改写召回查询？（短查询、模糊查询）
  • 哪些 turn-state 需要重置？（tool_calls_made、revisions、subagent_history、routed_tools）

§3.1静态 / 动态分层 + Anthropic prompt cache

Anthropic 的 cache_control 是这个 agent 最大的成本优化—— 系统提示一旦标了 {type: "text", text: "...", cache_control: {type: "ephemeral"}}， 下一轮的同一段 prefill 只收 1/10 价。

但要让 cache 真正命中，需要把消息严格分两层：

backend/agent/nodes/perception.py

def _build_static_system(prompt_caching: bool):
    """Build the SystemMessage. Static = identical across turns,
    so we attach Anthropic cache_control to make it cacheable."""
    skills_block = skills_index_block()
    text = STATIC_SYSTEM_PROMPT
    if skills_block:
        text += "\n\n---\n" + skills_block

    if prompt_caching:
        return SystemMessage(content=[
            {"type": "text", "text": text,
             "cache_control": {"type": "ephemeral"}},
        ])
    return SystemMessage(content=text)


def perception(state: AgentState) -> dict:
    cfg = get_settings()
    msgs = state.get("messages") or []

    # Inject the static (cacheable) system message exactly once per thread.
    if not any(isinstance(m, SystemMessage) for m in msgs):
        prompt_caching = cfg.model.startswith("anthropic:")
        msgs.append(_build_static_system(prompt_caching))

    # Per-turn dynamic context goes into a HumanMessage so the static
    # SystemMessage above stays byte-identical and stays cached.
    dynamic_parts = []
    profile_block = get_profile().to_prompt_block()
    if profile_block: dynamic_parts.append(profile_block)
    ...
    msgs.append(HumanMessage(content="\n\n".join(dynamic_parts)))

    return {
        "messages": msgs,
        "recalled_memories": facts,
        "recalled_reflections": reflections,
        "tool_calls_made": 0,
        "revisions": 0,
        "routed_tools": [],
        "subagent_history": [],
    }
pattern to steal

静态 prompt 进 SystemMessage（带 cache_control），动态 context 进 HumanMessage。 如果你把"今天的日期"或"用户名字"这种每轮变化的字符串拼进 system prompt， cache 永远不命中——你白白每轮交全价 prefill。 这个分层一次到位，agent 的 input 成本能砍 60-90%。

§3.2HyDE 召回 — 短查询的救星

用户问 "我之前让你记的什么来着？" 这种短查询，直接拿这句话去 ChromaDB 做向量召回， 命中率很差——它和"我喜欢简洁的回答并尽量引用来源"在向量空间根本不挨着。

HyDE (Hypothetical Document Embeddings) 的解决思路：先让一个 cheap model 写个假想答案， 用这段假想答案去做向量查询。假想答案在语义空间里更接近真正存的内容。

backend/agent/memory/long_term.py

def recall_hyde(self, query: str, k: int = 4) -> list[str]:
    """Generate a hypothetical answer first, then search by it.
    Used for vague / short queries where direct vector search misses."""
    from ..nodes.llm import get_fast_llm
    hypo = get_fast_llm(temperature=0.3).invoke(
        f"In one short paragraph, what would the answer to this be? "
        f"Don't worry about being accurate — we just need it for retrieval.\n\n"
        f"Q: {query}"
    ).content
    return self.recall(hypo, k=k)

Perception 节点通过启发式判断要不要走 HyDE—— 只有当前消息 < 12 个字符时才走，否则直接用原话召回（避免每轮多一次 LLM 调用）。

§3.3Turn-state reset 的取舍

Perception 是图的第二个节点（在 summarizer 之后），但它是每轮都跑的第一个"做事"节点。 所以把"本轮计数器"清零的责任放在它这里最合适：

    return {
        ...
        "tool_calls_made": 0,    # reset ReAct loop counter
        "revisions": 0,          # reset critic retry counter
        "routed_tools": [],      # clear last turn's tool route trace
        "subagent_history": [],  # clear supervisor visit log
    }

看起来啰嗦，但这是 checkpointer-friendly 的写法——每个字段都有明确的"谁负责重置"。 如果你忘了重置 subagent_history，supervisor 在第二轮会以为它已经跑过 researcher 了， 然后强制路由到 writer，跳过本轮真正需要的研究。

IV Part Four — Guardrails

In & out · cheap regex first, expensive checks never

大部分护栏不需要 LLM——你只要会写正则

Guardrails 在两个位置：输入端（perception 之后、planner 之前） 和输出端（critic 通过、extractor 完成之后、END 之前）。

§4.1Input · 提示注入 + 预算检查

输入护栏做两件事，都不调 LLM：

  1. 正则匹配常见的 prompt-injection 句式（"ignore previous instructions"、"reveal your system prompt"、…）
  2. 查 session 累计花费，超 AGENT_SESSION_BUDGET_USD 直接拒绝本轮

backend/agent/nodes/guardrails.py

_INJECTION = re.compile(
    r"(ignore (all )?previous|disregard (the )?(above|prior)|"
    r"reveal (your )?(system )?(prompt|instructions)|"
    r"jailbreak|DAN mode|developer mode)",
    re.IGNORECASE,
)

def input_guardrail(state: AgentState) -> dict:
    sid = state.get("session_id", "")
    user_text = state.get("user_input", "")

    # 1) Cost guardrail — refuse turn if over session budget
    cost = usage_tracker.session_cost(sid)
    cap = get_settings().session_budget_usd
    if cap > 0 and cost >= cap:
        emit("input_guardrail", "guardrail",
             {"action": "block", "reason": "budget"}, session_id=sid)
        return {"blocked": True, "block_reason": f"Budget cap ${cap:.2f} hit"}

    # 2) Prompt injection — fast regex, no LLM
    if _INJECTION.search(user_text):
        return {"blocked": True, "block_reason": "prompt-injection pattern"}

    return {"blocked": False}

§4.2Output · PII 脱敏

输出护栏扫一遍 final_answer，把疑似 PII 用 [REDACTED] 替换。 规则集小但精——更激进的 PII detection 应该用 Presidio 这种独立工具， 但 demo 阶段几条正则覆盖 95% 场景：

_PII = [
    (re.compile(r"\b[\w.+-]+@[\w-]+\.[\w.-]+\b"),                "[REDACTED-email]"),
    (re.compile(r"\b(?:\d[ -]?){13,16}\b"),                       "[REDACTED-card]"),
    # Phone — require explicit separators so we don't redact numbers like 65535.99999
    (_PHONE := re.compile(
        r"\b(?:\+?\d{1,3}[ .-])?"
        r"(?:\(\d{2,4}\)[ .-]?|\d{2,4}[ .-])"
        r"\d{3,4}[ .-]\d{3,4}\b"
    ), "[REDACTED-phone]"),
]
bug we shipped & fixed

第一版的 phone 正则是 \b\d{7,11}\b—— 结果用户问"sqrt(2^32-1)"得到答案 65535.99999， 被它当成手机号脱敏成 [REDACTED-phone].99999。 改成必须有显式分隔符（空格 / 横杠 / 点 / 括号）的版本才不再误伤。 正则护栏的可怕之处不在漏检，而在误检。

V Part Five — Planner & Supervisor

路由层 · 启发式短路 + 死循环熔断

两个会爆炸的节点，靠"在调 LLM 之前先想想能不能不调"和"硬编码兜底"压住

§5.1Planner · trivial 短路

Planner 决定走 direct（ReAct 直跑）还是 supervisor（多 agent 编排）。 绝大多数日常问题都该走 direct——只有"做研究 + 算数 + 写文档"这种明显多阶段任务才值得 supervisor。

所以 planner 的第一件事是启发式短路：

backend/agent/nodes/planner.py

def _trivial(text: str) -> bool:
    """Cheap, deterministic shortcut — skip the LLM entirely for short
    chitchat / yes-no / single-tool questions."""
    t = text.strip().lower()
    if len(t) < 80 and "\n" not in t:
        return True
    if any(kw in t for kw in ["hello", "hi", "thanks", "你好", "谢谢"]):
        return True
    return False

def planner(state: AgentState) -> dict:
    if _trivial(state["user_input"]):
        return {"route": "direct", "plan": []}
    # otherwise — call the fast LLM with structured_output to decide
    ...

这一招让 90% 的对话少一次 LLM 调用。 在本地 14B 上能省 5-15s，在 hosted Sonnet 上省 1-3s + 几个 cent。

§5.2Supervisor · 三层熔断对付死循环

Supervisor 是这个项目最难写的节点——它要根据"已经跑过哪些 sub-agent"决定下一步。 第一版只看最近 8 条消息，结果用本地 14B fast model 跑时 会反复 next: research → researcher → next: research → researcher … 直到撞 LangGraph 默认 25 步 recursion limit。

修法是三层熔断，按硬度排：

backend/agent/nodes/supervisor.py

MAX_PER_SPECIALIST = 2
MAX_TOTAL_HOPS = 5

def _force_route(history: list[str]) -> str | None:
    """Returns a forced next step (bypassing the LLM) when state alone
    tells us where to go — or None to let the LLM decide."""
    if "writer" in history:
        return "done"             # writer ran → must end
    if len(history) >= MAX_TOTAL_HOPS:
        return "write"            # too many hops → force writer
    if (history.count("researcher") >= MAX_PER_SPECIALIST and
        history.count("coder") >= MAX_PER_SPECIALIST):
        return "write"
    return None

def supervisor(state: AgentState) -> dict:
    history = list(state.get("subagent_history") or [])

    # Layer 1: Hard short-circuit on definitive state (no LLM call)
    forced = _force_route(history)
    if forced is not None:
        return {"route": forced}

    # Layer 2: Ask the fast LLM, but show it the history + caps explicitly
    history_blob = " → ".join(history) + f"   counts={counts}"
    d = (prompt | llm).invoke({"hist": history_blob, ...})
    nxt = d.next

    # Layer 3: Belt-and-braces — even if LLM picks a capped specialist, override
    if nxt == "research" and counts["researcher"] >= MAX_PER_SPECIALIST:
        nxt = "code" if counts["coder"] < MAX_PER_SPECIALIST else "write"
    elif nxt == "done" and "writer" not in history:
        nxt = "write"          # done before writer makes no sense

    return {"route": nxt}
pattern to steal

每个 sub-agent 把自己的名字 append 进 subagent_history—— supervisor 看 history 而不是看 message transcript。 message-based 的判断在小模型上几乎一定爆炸；history-based 的判断只要 5 行 Python 就能做。 这条 pattern 通杀所有"循环类 agent"（包括 ReAct 本身——只是那里的"history"叫 tool_calls_made）。

VI Part Six — Executor

Custom ToolNode · 三个 production 才会想到的细节

为什么不直接用 langgraph.prebuilt.ToolNode

LangGraph 自带的 ToolNode 跑起来是好用，但有三个生产级缺陷：

  1. 没有单调用超时——一个慢工具能挂住整个 agent
  2. 没有结果缓存——同样的 calculator 调用每轮都重算（即使是确定性输入）
  3. 没有结果截断——一个 read_file 把 50KB 文件全塞回 context，下一步 LLM 直接爆 token

所以 Taotao Agent 把它替换成自己的 tool_node，靠一个 safe_run_tool wrapper 解决：

backend/agent/nodes/executor.py + tools/safe_exec.py

# --- safe_exec.py ----------------------------------------------------
_EXECUTOR = concurrent.futures.ThreadPoolExecutor(max_workers=8)
_CACHE: OrderedDict[str, str] = OrderedDict()
_CACHE_MAX = 256

def safe_run_tool(tool: BaseTool, args: dict | str, *, cacheable: bool = True) -> str:
    """Wrap any LangChain tool with: timeout + LRU cache + truncation."""
    cfg = get_settings()
    key = _cache_key(tool.name, args)

    # 1. LRU cache hit → return immediately
    if cacheable:
        cached = _cache_get(key)
        if cached is not None:
            return cached

    # 2. Run with per-call timeout (in a worker thread)
    fut = _EXECUTOR.submit(tool.invoke, args)
    try:
        result = fut.result(timeout=cfg.tool_timeout_s)
    except concurrent.futures.TimeoutError:
        return f"[ERROR] tool {tool.name} timed out after {cfg.tool_timeout_s}s"
    except Exception as e:
        return f"[ERROR] {tool.name}: {e!r}"

    # 3. Truncate big outputs before they explode the next prompt
    out = _truncate(str(result), cfg.tool_result_max_chars)

    if cacheable:
        _cache_put(key, out)
    return out

# --- executor.py -----------------------------------------------------
def tool_node(state: AgentState) -> dict:
    """Custom replacement for langgraph.prebuilt.ToolNode that dispatches
    each tool through safe_run_tool (timeout + cache + truncation)."""
    last = state["messages"][-1]
    tool_msgs = []

§6.2Tool routing · top-K 相关性绑定

12 个工具全绑给 LLM 的话，每轮 prompt 多 1500-2000 token 描述。 Executor 在调 LLM 之前先用 ChromaDB 算当前问题和每个工具描述的余弦相似度， 只绑 top-K 个：

# --- tools/router.py -------------------------------------------------
ALWAYS_ON = {"remember", "recall", "update_profile", "read_profile", "load_skill"}

def select_tools(query: str) -> list[BaseTool]:
    cfg = get_settings()
    k = cfg.tool_route_topk
    if k <= 0: return all_tools()      # router disabled → bind all

    idx = _index()  # cached Chroma collection of tool descriptions
    hits = idx.query(query_texts=[query], n_results=k)
    picked_names = set(hits["ids"][0]) | ALWAYS_ON
    return [t for t in all_tools() if t.name in picked_names]

ALWAYS_ON 是关键——记忆类工具 (remember / recall / update_profile / read_profile / load_skill) 不参与相关性排序，永远绑给 LLM。 因为用户随时可能说"记住我喜欢简洁回答"，这种意图很难从 query 里语义匹配出来。

数据点

开 AGENT_TOOL_ROUTE_TOPK=5（再加 5 个 ALWAYS_ON = 实际绑 10 个） 相比绑 12 个，executor prompt 短约 30%，本地 14B 上 prefill 快 3-5 秒。 准确率几乎没掉——因为 LLM 想用的工具一般本来就在 top-3。

§6.3ReAct loop · 用 max_loop_iters 兜底

Executor 自己的循环（"想 → 调工具 → 看结果 → 想 → ..."）也可能失控。 route_after_executor 在每次工具调用后检查 tool_calls_made：

def route_after_executor(state: AgentState):
    last = state["messages"][-1]
    if getattr(last, "tool_calls", None):
        if state.get("tool_calls_made", 0) < get_settings().max_loop_iters:
            return "tools"
        # exceeded — bail out to critic with what we have
    return "critic"

§7.1python_repl · 唯一的"危险"工具

python_repl 是这套工具里安全压力最大的一个——用户输入的代码会被实际跑。 解决方案：开新进程跑，硬超时，stdout/stderr 都截断。 不用 exec()，不用 docker（demo 阶段太重）：

backend/agent/tools/python_repl.py

@tool
def python_repl(code: str) -> str:
    """Run Python code in a fresh subprocess. Return stdout (or stderr).
    10-second hard timeout. Output truncated to 2000 chars."""
    try:
        proc = subprocess.run(
            [sys.executable, "-c", code],
            capture_output=True, text=True, timeout=10,
            cwd=get_settings().workdir,                    # jail in workspace
        )
    except subprocess.TimeoutExpired:
        return "[ERROR] python_repl timed out after 10s"
    out = proc.stdout or proc.stderr or "(no output)"
    return out[:2000] + ("\n…[truncated]" if len(out) > 2000 else "")
production warning

这个 sandbox 对真生产环境不够——子进程仍能 open() 任意文件、 requests.get() 任意 URL、os.system() 任意命令。 生产里要么用 docker / firecracker / nsjail，要么用 Pyodide WebAssembly。 Demo 因为运行在你自己机器上、唯一调用者是你自己，这种程度可以接受。

§7.2记忆类工具 · 暴露给 LLM 的 KV 接口

三个记忆工具的签名设计有讲究——既要让 LLM 容易用对，又要让 prompt 短：

@tool
def remember(text: str, kind: str = "fact") -> str:
    """Save a fact you want to recall in future conversations.
    Use sparingly — prefer 'update_profile' for user attributes."""
    mid = get_memory().remember_if_new(text, kind=kind)
    return f"saved id={mid}" if mid else "already known (skipped duplicate)"

@tool
def update_profile(key: str, value: str) -> str:
    """Set a structured user attribute (preferred over 'remember' for
    things like name / language / response_style / timezone)."""
    get_profile().update(key, value)
    return f"profile.{key} = {value!r}"

注意 remember 的 docstring 引导 LLM 偏向用 update_profile。 这是个 prompt-engineering trick——通过工具描述影响 LLM 的工具选择倾向， 而不需要在 system prompt 里多写一段。

VIII Part Eight — Critic

Reflexion 节点 · 启发式跳过 + 一次有限重试

真实场景里 80% 的回答不需要批评——别硬上一刀

Critic 在 executor 之后跑，模式是 Reflexion：

  1. 读 last AIMessage 内容 + tool 调用历史
  2. 用 fast_llm 做"这个回答合格吗？"二元判断
  3. 如果不合格 + 还没重试过 → 把 critique 注入消息历史，retry to executor
  4. 如果合格 / 重试过一次 → pass，写一条 reflection 到长期记忆

§8.1启发式跳过

批评器跑一次本身代价不小（一次 fast_llm 调用 + structured_output）。 如果上一步明显是琐碎答案，直接跳过：

backend/agent/nodes/critic.py

def _should_skip_critic(user_q: str, draft: str, tool_calls_made: int) -> str | None:
    """Heuristic gate — return reason-string to skip, or None to run critic."""
    q, d = user_q.strip().lower(), draft.strip()
    # Pure chitchat — no tools used, short answer, not a question
    if tool_calls_made == 0 and len(d) < 200 and "?" not in q:
        return "trivial-answer"
    # Pure greeting
    if any(g in q for g in ["hello", "hi", "thanks", "你好", "谢谢"]):
        return "greeting"
    return None

§8.2有限重试 · 第二次必发车

Reflexion 论文里允许 N 次重试。实践中 N > 1 几乎没有边际收益—— 第二次还过不了，第三次大概率也过不了，反而把延迟和 token 都搭进去：

def critic(state: AgentState) -> dict:
    revisions = state.get("revisions", 0)

    # Force-pass on the second attempt — diminishing returns
    if revisions >= 1:
        return {"critique_passed": True,
                "critique": "(second pass — shipping)"}

    skip_reason = _should_skip_critic(...)
    if skip_reason:
        return {"critique_passed": True, "critique": "", "revisions": 0}

    # Otherwise run the actual critic LLM call
    verdict = (prompt | get_fast_llm().with_structured_output(Verdict)).invoke(...)
    if not verdict.passed:
        # Inject critique back into messages so executor sees it on retry
        get_reflections().add_if_new(verdict.notes, source="critic")
        return {"critique_passed": False, "revisions": revisions + 1, ...}
    return {"critique_passed": True}
pattern to steal

把启发式 gate 放在每一个"会调 LLM"的节点入口。 Planner / Critic / Extractor 三个节点都用了同样的模式：先用一段 Python 看能不能跳过， 实在跳不过才调 LLM。这一招让本地 14B 跑日常对话从 60s+ 降到 15-25s。

IX Part Nine — Extractor

Mem0-style 后台抽取 · 异步线程 · 启发式 gate · 写前去重

不让"自动记忆"变成"每轮多一次延迟 + 重复事实塞爆库"

Mem0 的核心思路：每轮对话结束后，让一个 LLM 看 user_input + final_answer， 自动抽出三类东西——

  • Facts：用户陈述的客观事实（"我在北京"）
  • Profile updates：用户偏好或属性（"喜欢简洁回答"）
  • Reflections：模型自己的反思（"这次没引用来源，下次注意"）

朴素实现的问题是三连击：

  1. 每轮多 1 次 LLM 调用 → 用户感知延迟 +5-15s
  2. 同样的事实被反复抽出来 → 向量库越塞越满
  3. 琐碎对话也跑抽取 → 大量空跑成本

Taotao Agent 的 extractor 用三招都解决：

§9.1① 异步线程 · 不阻塞用户

Extractor 节点本身只是启动一个后台线程，立刻 return 让图继续往下走。 实际的 LLM 调用 + 数据库写入都在线程里：

backend/agent/nodes/extractor.py

def extractor(state: AgentState) -> dict:
    sid = state.get("session_id", "")
    user_text  = state.get("user_input", "")
    final      = state.get("final_answer", "")
    critique   = state.get("critique", "")
    tool_calls = int(state.get("tool_calls_made", 0))

    # Heuristic gate — skip 70% of turns where there's nothing to extract
    if not _should_extract(user_text, final, critique, tool_calls):
        emit("extractor", "memory_update",
             {"facts": [], "reflections": [], "skipped": "gated"}, session_id=sid)
        return {}

    # Spawn the actual extraction work in the background — don't block END
    threading.Thread(
        target=_run_extraction,
        args=(sid, user_text, final, critique),
        daemon=True,
    ).start()
    return {}

§9.2② 启发式 gate · 70% 的对话直接跳过

_PERSIST_KEYWORDS = ("remember", "记住", "i prefer", "my name is",
                     "我喜欢", "我叫", "my ", "我的")

def _should_extract(user: str, final: str, critique: str, tool_calls: int) -> bool:
    """Skip extraction unless this turn likely contains memorable content."""
    u = user.lower()
    if len(user) < 20 and tool_calls == 0:
        return False                           # pure chitchat
    if critique and critique != "(second pass — shipping)":
        return True                            # critic flagged → save reflection
    if any(kw in u for kw in _PERSIST_KEYWORDS):
        return True                            # explicit memorable signal
    if tool_calls >= 2:
        return True                            # non-trivial multi-step → maybe save facts
    return False

§9.3③ 写前去重 · cosine similarity ≥ 0.92 跳过

抽出来的东西先和向量库已有内容做相似度比较，超阈值就跳过。 这一步在 LongTermMemory.remember_if_new / ReflectionMemory.add_if_new 里实现：

def remember_if_new(self, text: str, kind: str = "fact",
                    threshold: float | None = None) -> str | None:
    threshold = threshold or get_settings().dedup_threshold     # default 0.92
    existing = self._coll.query(query_texts=[text], n_results=1)
    if existing["distances"] and existing["distances"][0]:
        # Chroma returns L2 distance; convert to cosine similarity ≈ 1 - d/2
        sim = 1 - existing["distances"][0][0] / 2
        if sim >= threshold:
            return None                                       # too similar, skip
    return self.remember(text, kind=kind)

X Part Ten — Memory

4 个独立存储 · 每个都有自己的"召回 / 写入 / 失效"语义

不要把所有东西都塞进一个 ChromaDB collection

很多 agent demo 把"记忆"做成一个 ChromaDB collection， 然后所有东西都往里塞——facts、reflections、profile、tool 历史、对话片段…… 然后召回时拿用户问题查 top-K，全部喂回。

问题是：不同性质的记忆有不同的召回条件、生命周期、查询方式。 Taotao Agent 拆成了 4 个独立存储：

§10.1Short-term · messages + 滑窗摘要

消息条数超 24 时，summarizer 节点把前面的"老消息"压成一段 SystemMessage 摘要， 用 RemoveMessage 把原消息从历史里抹掉：

backend/agent/nodes/summarizer.py

SUMMARIZE_THRESHOLD = 24
KEEP_RECENT = 8

def summarizer(state: AgentState) -> dict:
    msgs = state.get("messages") or []
    if len(msgs) <= SUMMARIZE_THRESHOLD:
        return {}
    older, recent = msgs[:-KEEP_RECENT], msgs[-KEEP_RECENT:]
    summary = get_fast_llm().invoke(
        f"Summarize this conversation in 4-6 bullets:\n\n{_format(older)}"
    ).content
    return {
        "summary": summary,
        "messages": [RemoveMessage(id=m.id) for m in older]
                  + [SystemMessage(content=f"[Summary of older turns]\n{summary}")],
    }

§10.2Long-term facts · ChromaDB + HyDE + dedup

Facts 用 ChromaDB 持久化，每条带 metadata（source、kind、session_id、timestamp）。 召回支持两种路径：

  • recall(query, k=4)：直接向量查询，适合长 / 具体的 query
  • recall_hyde(query, k=4)：先生成假想答案再查，适合短 / 模糊的 query

写入永远走 remember_if_new，先和库里最相似的一条比，超阈值跳过。

§10.3Reflections · 独立 collection 而非混在 facts 里

Reflections 是模型对自己的批评，比如"这次没引用来源"、"我犯了个算术错误"。 它们不该被当 facts 召回（"我之前让你记的什么？"不应该返回"我犯了个算术错误"）， 但应该在 perception 阶段单独召回 2 条放进 system context 里—— 让模型在下一轮对话时主动避免重复同样的错。

所以 facts 和 reflections 各占一个独立的 ChromaDB collection。 perception 分开召回，分别注入到 prompt 不同位置。

§10.4Profile · 结构化 KV 而非向量

"用户名字"、"偏好语言"、"回答风格"这种结构化属性不该进向量库。 原因：

  • 它们要每轮全量注入 prompt，向量召回 top-K 反而漏掉
  • 它们要支持精确更新（"把我的名字改成 Alice"），KV 写一行搞定
  • 它们要支持枚举展示（前端 Profile 面板），KV 直接 list()

所以 Profile 是个简单 JSON 文件，加线程锁避免并发写冲突：

backend/agent/memory/profile.py

class Profile:
    def __init__(self, path: Path):
        self.path = path
        self._lock = Lock()

    def update(self, key: str, value: Any) -> dict:
        with self._lock:
            data = self._read()
            data[key] = value
            self._write(data)
        return data

    def to_prompt_block(self) -> str:
        data = self.all()
        if not data: return ""
        lines = [f"- {k}: {json.dumps(v, ensure_ascii=False)}" for k, v in data.items()]
        return "[User profile]\n" + "\n".join(lines)

§10.5Skills · markdown + YAML frontmatter

Skills 是可被 agent 按需加载的指令——本质就是一组 system prompt 片段， 但分成索引（每轮注入，让 LLM 知道有哪些技能可用）和正文 （只在 load_skill(name) 被调用时才注入完整 body）。

这种"两段式"设计跟 Anthropic Skills 一脉相承——避免每轮把所有指令都全量喂进去。

backend/data/skills/concise-answers.md

when_to_use: User asks short factual questions or chitchat

一个 70 行的文件，决定了 agent 能不能跑 5 种 provider

llm.py 是整个 agent 里最简单也最关键的文件。它做三件事：

def _provider_kwargs(name: str) -> dict:
    """Pick up env-supplied overrides so the same code path serves
    hosted APIs *and* local OpenAI-compatible servers / Ollama."""
    kwargs: dict = {}
    if name.startswith("openai:"):

        if base:
            kwargs["base_url"] = base

§11.1热切换的精妙之处

整个热切换链路只有 3 步：

  1. POST /model {"model": "anthropic:claude-haiku-4-5"} 进来
  2. update_runtime_model() 改 get_settings() cache 里的 model 字段
  3. reset_llm_cache() 把 _build.cache_clear()

XII Part Twelve — Observability

JSONL trace + SSE bus + token meter

一个事件，三种消费者

Agent 的可观测性比传统服务难——你不能光看 HTTP 日志。每个节点、每个工具调用、 每个 sub-agent 决策都得能实时看到，事后还能翻历史。

Taotao Agent 的方案：每个节点显式调 emit(node, kind, payload)， 事件同时进三个 sink：

  • JSONL 文件 data/traces.jsonl ——事后回看 / 离线分析
  • SSE event bus——实时推到前端 trace 面板
  • UsageCallback——挂在每个 LLM 调用上，捕获 token 用量到 usage.snapshot()

class EventBus:
    """In-memory pub/sub keyed by session_id. /chat subscribes; emit() publishes."""
    def __init__(self):
        self._subs: dict[str, list[asyncio.Queue]] = defaultdict(list)

    def subscribe(self, sid: str) -> asyncio.Queue:
        q = asyncio.Queue()
        self._subs[sid].append(q)
        return q

    def publish(self, sid: str, evt: dict) -> None:
        for q in self._subs.get(sid, []):
            q.put_nowait(evt)

§13.1SSE 客户端 · 手写 vs EventSource

浏览器自带 EventSource，但不能 POST——它只能 GET。 Taotao Agent 的 /chat 是 POST（要塞 message body），所以前端用 fetch + ReadableStream 手写 SSE 解析：

frontend/src/api.ts

export function streamChat(message, sessionId, handlers): () => void {
  const ctrl = new AbortController();

  (async () => {
    const resp = await fetch(`${BASE}/chat`, {
      method: "POST", body: JSON.stringify({message, session_id: sessionId}),
      signal: ctrl.signal,
    });
    const reader = resp.body.getReader();
    const decoder = new TextDecoder();
    let buf = "";

    while (true) {
      const {value, done} = await reader.read();
      if (done) break;
      buf += decoder.decode(value, {stream: true}).replace(/\r\n/g, "\n");

      // SSE frames separated by blank line
      let idx;
      while ((idx = buf.indexOf("\n\n")) !== -1) {
        const frame = buf.slice(0, idx);
        buf = buf.slice(idx + 2);
        let event = "message";
        const dataLines: string[] = [];
        for (const rawLine of frame.split("\n")) {
          if (rawLine.startsWith("event:")) event = rawLine.slice(6).trim();
          else if (rawLine.startsWith("data:")) dataLines.push(rawLine.slice(5).trimStart());
        }
        dispatch(event, dataLines.join("\n"));
      }
    }
  })();

  return () => ctrl.abort();
}

关键细节是用 \r\n → \n 归一化（sse-starlette 在某些环境送 CRLF）， 并且每个 frame 用双换行分隔。 dispatch 根据 event 字段路由到不同 handler—— session / trace / token / done。

前后端分离的静默漂移（silent drift）：后端改 ChatIn.session_id → sessionId，前端没人发现，上线 422。 我们用 openapi-typescript 把 FastAPI 自动生成的 OpenAPI 3.1 schema 转成 TypeScript 类型，让编译期而不是运行期报红。

  1. 用 backend venv 跑一行 Python，把 app.openapi() dump 到 frontend/openapi.json
  2. openapi-typescript 把它转成 1100+ 行 schema.gen.ts
  3. 前端用 Schemas["ChatIn"] 这种派生类型，不再手写

import type { Schemas } from "./api/index";

export type PruneIn     = Schemas["PruneIn"];

实际收益的三个常见场景：

  • 后端改字段名 · npm run gen:api 后所有用旧名的地方编译失败 · 安全重命名
  • 后端加端点 · Schemas["NewModel"] 自动出现，IDE 智能提示有所有字段
  • 后端改字段类型（float → Decimal）· OpenAPI 类型从 number 变 string · 所有当数字用的地方编译失败提醒你转换
细节 · response 类型不准的根因

openapi-typescript 只能精确出 request 类型 —— 因为 FastAPI 只在 endpoint 显式标 response_model=SomeModel 时才把 response schema 写进 OpenAPI。 我们大多数端点返回 raw dict，所以 schema.gen.ts 里 response 是 unknown / 含糊 union。短期：response 类型继续手写并注释对齐哪个端点；长期：给关键端点逐步加 response_model=，下一次 codegen 自动收紧。

什么时候不要 codegen：API 极不稳定（每天改、生成文件 review 比手写还累）、多 client 用不同 schema 子集（一刀切类型反而碍事）、已经有 BFF 层做翻译。

XIV Part Fourteen — Tradeoffs

每个设计都有它的反面

这里写的是"为什么这么做"——以及"代价是什么"

§14.1为什么双档模型？

赞成：路由 / 摘要 / 批评 / 抽取这些"分类型"任务用 Sonnet 是浪费—— Haiku 准确率几乎一样，价格 1/4，速度 2-3×。

代价：要管两个模型的 capability 差异（比如 Haiku 不太擅长复杂结构化输出）， planner / critic 的 prompt 得为弱模型重新调，否则准确率会掉。

§14.2为什么 prompt cache 只对 Anthropic 开？

赞成：OpenAI / Gemini / 本地模型都没有 Anthropic 那种显式 cache_control 标记。 OpenAI 有自动 cache，但触发条件、命中率都不可控。本地模型则完全没 cache 这一说。

代价：换到 OpenAI 时整个 system prompt 要每轮全价 prefill。 用 tool_route_topk 缩短 prompt 是部分缓解，但仍然贵 5-10×。

§14.3为什么 ChromaDB 不用 Pinecone / Weaviate？

赞成：ChromaDB 嵌入式，零 infra——本地起一个 SQLite 就跑。 Demo 不需要 RAG 集群级吞吐。

代价：单机性能上限低（百万级向量后查询 > 200ms）； 没分布式；没 hybrid search（要同时做 BM25 + vector 得自己拼）。 搬上线前要换 Qdrant / Weaviate / Vespa。

§14.4为什么遍地启发式 gate？

赞成：这是 demo 性能优化最大的杠杆——planner / critic / extractor 三个节点的 gate 让 90% 的对话少 3 次 LLM 调用， 本地 14B 上能省 30s+ 延迟。

代价：启发式规则会漏—— 比如琐碎对话里偶尔藏一句"我刚搬到上海"，extractor 启发式判定 < 20 字符就跳过，结果没存。 解决方案要么是开关切换"完整模式 vs 性能模式"，要么定期跑一次 batch 抽取兜底。

§14.5为什么 extractor 用 threading 而不是 asyncio？

赞成：threading 简单粗暴，不污染 graph 的 async/sync 边界。 Extractor 节点本身既能在 sync graph 里跑（get_sync_graph）， 也能在 async graph 里跑（get_graph），用线程就不用关心。

代价：线程数没有上限，高并发下可能堆积； 没有 backpressure 机制；如果 extraction 失败，错误也只在 trace 里看见。

§14.6为什么不用 langgraph.prebuilt.ToolNode？

赞成：自定义 tool_node 接 safe_run_tool 之后， 时timeout / cache / truncation 三件套全到位，且错误信息能 emit 到 trace， 用户能在前端看到"calculator timed out"而不是 graph 直接抛异常。

代价：失去了 prebuilt ToolNode 的某些功能（如自动并行调用多个独立工具）。 Demo 阶段够用，真生产里要补。

XVI Part Sixteen — Patterns to steal

从这套代码里能搬走的 8 个 pattern

不一定要全用 LangGraph，但这些模式应该刻进肌肉

  1. 静态/动态 prompt 分层 + cache_control—— 系统提示永远只放静态内容，动态 context 进 HumanMessage。 Anthropic 上能省 60-90% input cost。
  2. 双档 LLM router—— get_llm() + get_fast_llm() 两个函数搞定。 路由 / 摘要 / 批评 / 抽取全部走 fast tier。
  3. 启发式 gate 包住每个 LLM 调用—— planner 跳琐碎查询、critic 跳短答案、extractor 跳无信息轮次。 本地模型上能砍 50%+ 延迟。
  4. Custom ToolNode + safe_run_tool wrapper—— 每个工具调用三件套：timeout、LRU cache、result truncation。 生产环境的最后一道防线。
  5. Tool routing top-K + ALWAYS_ON 集合—— 语义相关性挑 K 个工具绑给 LLM，记忆类工具永远绑。 prompt 短 30%，准确率不掉。
  6. Subagent history-based 熔断—— 每个 sub-agent append 自己名字到 list，supervisor 看 list 而不是看消息。 通杀所有"会循环"的多 agent 系统。
  7. Mem0-style 后台 extractor + 启发式 gate + 写前去重—— 异步线程不阻塞用户、gate 跳无意义轮次、cosine 阈值防止重复事实。 三连击让"自动记忆"真能用。
  8. HyDE 召回针对短查询—— 模糊 query 先生成假想答案再做向量查询，命中率显著提升。 启发式触发（短查询才走，长查询不浪费）。

§17.1加一个新工具 · 完整 recipe

这是项目里 最常做 的扩展。30 行 Python + 改两行 import = 新工具上线， graph 和 harness 两个端点都自动能用。这一节用真实合入项目的 current_time 工具走全流程。

第 1 步 · 写工具文件

工具就是个普通函数 + @tool 装饰器。LangChain 自动从类型注解和 docstring 生成 JSON schema 喂给 LLM。文件丢在 backend/agent/tools/<name>.py：

# backend/agent/tools/current_time.py
from __future__ import annotations
import datetime as dt
from zoneinfo import ZoneInfo
from langchain_core.tools import tool


@tool
def current_time(timezone: str = "UTC") -> str:
    """Get the current wall-clock time in the given IANA timezone.

    Use this whenever the question depends on "now" — relative dates
    ("yesterday", "next Monday"), age calculations, freshness checks
    on web search results, scheduling, etc. NEVER guess the current
    date from your training data.

    Args:
        timezone: IANA timezone name like "Asia/Shanghai", "UTC",
                  "America/Los_Angeles". Defaults to "UTC".
    """
    try:
        tz = ZoneInfo(timezone)
    except Exception:
        return f"Unknown timezone {timezone!r}. Try 'UTC' or an IANA name like 'Asia/Shanghai'."
    now = dt.datetime.now(tz)
    return now.strftime("%Y-%m-%d %H:%M:%S %Z (%A, week %V)")

核心约定 · 抄就完事了

写法作用
@tool 装饰器把函数包装成 LangChain BaseTool
函数名LLM 看到的 tool name · 小写下划线、描述性
docstring 第 1 段给 LLM 看的"什么时候用我"，比函数名更重要
Args: 段每个参数的语义。LLM 严格按 docstring 选参数
类型注解自动生成 JSON schema · 强约束 LLM 输出
默认值让参数可选，模型可以省略
return str返回什么 LLM 就看到什么。不要返回 dict（会被 str() 后丢信息）

7 条最佳实践

  1. docstring 写 "Use this when..." — 模型不靠工具名挑工具，靠这一句
  2. 永远 return str — 把结构化数据格式化成 LLM 能读的文字（如 "x=42, y=10"）
  3. 错误也返回字符串 — return f"Error: {e!r}. Try ..." 给 LLM 自我修复的提示，不要 raise
  4. 参数尽量少 — 超过 3 个参数 LLM 容易漏填，考虑用 args_schema + Pydantic 拆分
  5. 用 Literal[...] 做枚举 — mode: Literal["fast","accurate"]，自动 schema 强约束
  6. 副作用工具留 dry_run — 写文件 / 改数据库的工具加 dry_run: bool = False，让 LLM 先预览
  7. 耗时工具加超时 — harness 的 safe_run_tool 已经统一加了 30s 上限；graph 里要自己加 asyncio.wait_for

进阶 · 不一定现在用
