Hermes 是什么 · 它给 开源 agent 解决了什么问题

一页讲清三件事：① Nous Research 的 Hermes 模型系列是什么、为什么 agent 圈子离不开它； ② Hermes 那套 <tool_call> XML 格式怎么工作、为什么后来变成开源 LLM 调工具的事实标准； ③ 在你这个 taotao-agent 里怎么一行命令切到 Hermes，能跑得多快。

§ 1一句话 + 三层含义

Nous Research 出的开源 LLM 系列，靠"会调工具的开源模型"在 agent 圈出名。 听到 "Hermes" 这个词，根据上下文可能指三个不同东西，区分清楚就不会混。

含义是什么
① 模型 Llama / Mistral / Qwen 的微调系列。最新 Hermes 4 在 2025 年 8 月发布，14B / 70B / 405B 三个尺寸，base 是 Llama-3.1。亮点：hybrid reasoning（推理可以 prompt 控制开关）+ uncensored（不像 Llama Instruct 动不动 refuse）+ 训练目标里显式写了 tool calling。
② Tool-call 格式 Hermes 团队定义的协议：用纯文本 XML 标签 <tool_call>{...}</tool_call> 描述一次工具调用。后来 Qwen / Mistral / DeepSeek 自己 fine-tune tool 版本时全都抄了类似 schema，成开源圈事实标准。
③ 团队 Nous Research，Discord 上的开放权重研究社区，没融过资，纯靠贡献者推动。除了 Hermes 还做过 Capybara、DeepHermes、Genstruct 等实验性数据集。

§ 2Hermes 模型简史 · 一张表说完

版本时间base关键升级

为什么 "2 Pro" 是分水岭

2024 年 5 月之前，开源里"能调工具"的模型都得靠 ReAct prompt 硬凑（让模型按 Thought:/Action:/Observation: 格式输出，再用正则解析），不稳。Hermes 2 Pro 是第一个把 tool calling 当成训练目标的开源模型， 从此 ChatOllama / vLLM 等推理框架可以直接把 tools=[...] 参数喂进去，模型出标准结构化结果。

§ 3Hermes 的 tool-call 格式 · 跟 OpenAI 区别

OpenAI / Anthropic 的工具调用是结构化字段（API 层就有 tool_calls 的 schema）。 Hermes 不一样，它把 tool call 编码成消息文本里的 XML 标签。两边对比：

OpenAI 风格

{
  "role": "assistant",
  "content": null,
  "tool_calls": [{
    "id": "call_abc",
    "function": {
      "name": "calculator",
      "arguments": "{\"expr\":\"2+2\"}"
    }
  }]
}

Hermes 风格

{
  "role": "assistant",
  "content": "<tool_call>\n{\"name\":\"calculator\",\"arguments\":{\"expr\":\"2+2\"}}\n</tool_call>"
}

结果回灌时也是文本：

{
  "role": "tool",
  "content": "<tool_response>\n{\"result\": 4}\n</tool_response>"
}

这个设计为什么后来变标准

  • 任何 base LLM 都能学：不需要改 chat template，不需要加 special token，纯文本就行。
  • system prompt 里就能描述：把工具 schema 用 JSON 写在 system 里，加一句 "if you need a tool, emit <tool_call>...</tool_call>"，base 模型 zero-shot 就大概率会用了。
  • streaming 友好：完全是文本流，不需要等结构化字段闭合再返回客户端。
  • 多 call 简单：连续多个 <tool_call> 块就是 parallel tool call，不需要数组 schema。

推理框架在背后做了啥

Ollama / vLLM / LM Studio 这些推理框架收到 tools=[...] 参数后，会自动把工具 schema 注入 system prompt （Hermes 风格 XML），然后流式扫描模型输出，遇到 <tool_call> 就提取出来组装成 OpenAI 风格的 tool_calls 字段返回给你。所以你的应用代码完全不用改， 换 Hermes 跟换 GPT-4 在调用层是一模一样的。

§ 4实测 · Hermes 3 8B vs Qwen 2.5 14B

在你这个 taotao-agent 里跑一条 "compute 17 * 23 with the calculator tool"， 两个本地模型对比（M3 Max，Ollama 0.23.x，2026-05）：

模型大小首次响应tool call 正确率结果
ollama:hermes3:8b 4.7 GB ~6s ✓ 正确生成 {name: calculator, args: {expr: "17 * 23"}}，内容字段空，标准 schema
ollama:qwen2.5:14b 9.0 GB ~20s ✓ 正确，但偶尔 args 字段名漂移（expression vs expr）
ollama:deepseek-r1:14b 9.0 GB — ✗ 不支持 Ollama 模板不暴露 tool calling，全部跑 ReAct 文本格式，不进 graph
ollama:deepseek-coder-v2:16b 8.9 GB — ✗ 不支持 同上

结论

本地能跑、能调工具、又不占显存的"性价比之王"目前是 hermes3:8b。 小 4 GB、快 3 倍、tool 调用更稳。Qwen 14B 在长文本/中文回答上仍然更好，但纯 tool-driven workflow 建议默认 Hermes。

§ 5在 taotao-agent 里怎么用

1 · 装模型（一次）

ollama pull hermes3:8b      # 4.7 GB · ~80s 在 100 Mbps 上

2 · 在 picker 里切（每次）

装完打开 frontend，model picker 自动会列出 ollama:hermes3:8b 并标 tools 角标 — 因为：

  • backend/agent/models_catalog.py 的 discover_ollama() 直接调 http://localhost:11434/api/tags，列你装的全部模型。
  • OLLAMA_NO_TOOL_PATTERNS 黑名单里没有 "hermes"，所以默认按 "支持工具" 处理（实测正确）。

或者写到 backend/.env 当默认：

AGENT_MODEL=ollama:hermes3:8b
AGENT_FAST_MODEL=ollama:qwen2.5:7b   # router/critic 仍用 qwen 小模型也行

3 · 跑一把

graph 和 harness 两个端点都不用改一行代码 — 之前讲的那套 "应用代码无感" 就是这个意思。直接发：

curl -N -X POST http://127.0.0.1:8000/chat/v2 \
  -H 'Content-Type: application/json' \
  -d '{"message":"compute 17 * 23 and tell me the result"}'

SSE 流里能看到一模一样的 tool_call / tool_result 事件 — Ollama 已经把 Hermes XML 翻译成标准格式喂给 langchain。

什么时候 不 选 Hermes

  • 需要中文长文 / 长 reasoning chain → Qwen 14B / 32B 更好
  • 已经接 Anthropic / OpenAI 且不在乎花钱 → Sonnet 4.5 / GPT-4o 更准更稳
  • 纯代码生成 + 不调工具 → DeepSeek-Coder-v2 仍然是开源代码王者

§ 6面试可能会被问的几句话

  • Q: Hermes 跟 ChatML 是一回事吗？ 不是。ChatML 是 chat 消息模板（<|im_start|>...<|im_end|> 那套）。Hermes tool 格式是套在 ChatML 里面的工具调用协议。两者正交。
  • Q: 为什么 OpenAI 不用 Hermes 这种 XML 格式？ 因为 OpenAI 的模型可以在 SFT 阶段直接学结构化字段（API 端拿到的就不是文本），不需要"靠 prompt 教模型用文本编码工具调用"这条 hack 路径。开源模型没这个特权（API 层就是裸文本），所以走 Hermes 这条更实用。
  • Q: parallel tool call 怎么处理？ 模型连续输出多个 <tool_call> 块，推理框架按顺序提取，组成数组返回。本质跟 OpenAI 的 tool_calls: [{},{},...] 等价。
  • Q: Hermes 4 的 hybrid reasoning 是什么？ system prompt 加一句 think before answering 时模型走 chain-of-thought，去掉这句就直接答。等于把 OpenAI o-series 的"思考预算"开放成 prompt-controlled 开关。
  • Q: Hermes 跟 DeepSeek 比谁更适合本地 agent？ 看任务。纯 reasoning DeepSeek-R1 强但不调工具；tool-driven workflow 必须选 Hermes / Qwen。在 Ollama 上 DeepSeek 系全部不暴露 function calling，不是模型问题是 Ollama template 问题，但用户视角就是 — 不能用。
