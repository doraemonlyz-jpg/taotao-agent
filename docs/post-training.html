Book 19 · Post-Training Field Guide

什么时候 真正 需要 fine-tune · 什么时候应该 prompt + RAG

面 Anthropic / OpenAI / Mistral / Cohere / Together / Fireworks 这些"做模型"的公司， post-training 是必考题。这一本不教你跑训练，教你判断和决策—— LoRA 跟 full SFT 差什么 · DPO 比 RLHF 强在哪 · catastrophic forgetting 怎么防 · 数据怎么标 · 7B / 70B 该选哪个起步 · 何时用 prompt + RAG 即可。配 3 个真复盘。

§ 1Pre-train / Post-train / RAG / Prompt · 一张图说清

"fine-tune" 这个词在工业界现在指代 4 种完全不同的事，混着说必出错。先把谱系梳清：

阶段做什么谁做典型成本
Pre-training 从 random weights 训出 base model · 海量 raw text · next-token prediction Meta / OpenAI / Anthropic / DeepSeek 等 $1M–$100M+ · 1000–10000 GPU-月
Continued pre-train (CPT) 在 base 上灌新 corpus（领域 / 语言 / 时间）· 仍 next-token 有大量私有 corpus 的公司 $50K–$500K
SFT (instruction tuning) 在 base 上做 (prompt, response) 对学习 · 改成"会聊天" 大部分应用公司 LoRA: $50–$5K · Full: $5K–$50K
Preference tuning (DPO/RLHF) 给定 (prompt, chosen, rejected) · 调整偏好 对 style / safety 有要求的公司 DPO: $100–$10K · RLHF: 10× DPO
RAG 不改模型 · inference 时把检索到的文本塞 prompt 大部分应用公司起手 $0 训练 · 推理多 1–3 倍 token
Prompt eng 不改模型 · 也不改 corpus · 只改 system prompt / few-shot 所有人 $0

面试官最常问

"什么时候 fine-tune · 什么时候 prompt + RAG？" senior 答案要带三条决策线：① 知识类问题 → RAG 优先 · ② 风格 / 格式 / safety类 → SFT 或 DPO · ③ 新能力 / 大幅行为改变 → CPT + SFT。 永远先问"我能不能用 prompt 解？"——这是 cost / 迭代速度 / 可解释性的最低风险路。

§ 2SFT (Supervised Fine-Tuning) · 入门必备

SFT 是所有 instruct / chat 模型的第一步。流程极简——给模型看一堆 (prompt, ideal_response)，让它学着照样回答。重点全在数据：

2.1 数据格式 · ChatML / Alpaca / ShareGPT

Alpaca 风（早期）

{
  "instruction": "explain RAG",
  "input": "",
  "output": "Retrieval-augmented..."
}

ChatML（现代标准）

{
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "explain RAG"},
    {"role": "assistant", "content": "..."}
  ]
}

2.2 数据规模 · 多少条够

目标典型条数示例
学一种新输出格式（如 JSON）500–2000OpenAI fine-tune 起步推荐
学一种领域 tone（如 legal / medical）5K–20KMed-PaLM SFT 数据量级
从 base 训一个通用 chat model50K–500KAlpaca 52K · UltraChat 1.5M
state-of-art instruct tuning1M+ 高质量Llama-3 instruct · Qwen-2.5

质量 >> 数量

LIMA paper (Meta 2023) 实证：1000 条精挑数据 + Llama-65B SFT 能打过 52K 条 Alpaca 训的模型。10 条用心写的远胜 1000 条 GPT 自动生成的。

2.3 关键超参 · 4 个就够

  • learning rate: 1e-5 ~ 5e-5（base 越大越小 · LoRA 可大到 1e-4 ~ 3e-4）
  • epochs: 1–3 · 超过 5 几乎一定 overfit
  • batch size: 32–256（global · 含 grad-accum）
  • warmup: 3% steps · 防初期梯度爆炸

2.4 一段最小 SFT 代码（Hugging Face TRL）

from trl import SFTTrainer
from datasets import load_dataset

trainer = SFTTrainer(
    model="meta-llama/Llama-3.1-8B",
    train_dataset=load_dataset("HuggingFaceH4/ultrachat_200k", split="train_sft"),
    args=dict(
        learning_rate=2e-5,
        num_train_epochs=1,
        per_device_train_batch_size=4,
        gradient_accumulation_steps=8,
        warmup_ratio=0.03,
        bf16=True,
    ),
)
trainer.train()

§ 3LoRA & QLoRA · 让 fine-tune 平民化

Full SFT 要存全部权重的优化器状态——8B 模型大约要 80GB+ VRAM（Adam 状态 8 倍权重）。 LoRA 只训低秩矩阵，把可训参数砍 100–1000 倍，QLoRA 再加 4-bit 量化， 让 70B 模型在单张 A100 80GB 就能 fine-tune。

3.1 LoRA 原理 · 一句话

原 weight W ∈ R^(d×k) 不动，旁边加一个低秩补丁 ΔW = B·A，其中 B ∈ R^(d×r)、A ∈ R^(r×k)， r 通常 8–64。训的时候只更新 A, B · 推理时合并 W + BA。 

原 forward: h = W x
LoRA forward: h = W x + (α/r) · B (A x)

参数量: 2 · d · r  (vs full: d · k)
当 d=k=4096, r=16 时: 131K vs 16M, 节省 99.2%

3.2 LoRA 关键超参

超参常见值调参思路
r (rank)8 / 16 / 32 / 64越大越接近 full SFT · 16 通常足够 instruct 任务
alphar · 2 (e.g. r=16 → α=32)scale = α/r · 实际 LR 乘子
target_modulesq_proj, k_proj, v_proj, o_proj所有 attention proj 最稳 · 加 MLP (gate/up/down) 更激进
lora_dropout0.05 / 0.1小数据集时调大防过拟合

3.3 QLoRA · 4-bit 量化 + LoRA

QLoRA paper (2023, Tim Dettmers) 把 base model 量化到 NF4 (4-bit normal float)， 只在前向时 dequantize 到 bf16 算 attention，训 LoRA adapter（bf16）。三个关键 trick：

  • NF4: 比 INT4 更适合权重的正态分布
  • Double Quantization: 把 quant constant 自身再量化 · 省 0.4 bit/param
  • Paged Optimizer: optimizer state 用 CPU offload + paging · 防 OOM spike

VRAM 实测速查

方案Llama-3.1-8BLlama-3.1-70B
Full SFT (bf16)~80 GB~700 GB · 8×A100
LoRA (bf16)~24 GB~200 GB · 4×A100
QLoRA (NF4)~8 GB~48 GB · 单 A100

3.4 LoRA vs Full SFT · 怎么选

选 LoRA 当选 Full SFT 当
预算紧 · 单卡性能差距是关键 KPI
需要多个 adapter（per-customer / per-style）不需要切换 adapter
数据量 < 50K数据量 > 100K · 信号足够
实验探索阶段定型阶段 · 准备一锤子

实证：在大多数 instruct 任务上，LoRA r=16 的效果比 full SFT 差 1–3%， 但成本省 90%+。商业上几乎不值得追那 1–3%，除非要 ship 旗舰模型。

§ 4偏好对齐 · DPO / KTO / RLHF / RLAIF

SFT 让模型"会回答"，但不一定回答得"令人满意"。偏好对齐（preference tuning） 是给定 (prompt, chosen, rejected) 对，让模型偏向 chosen。

4.1 RLHF 流水线 · 经典三步

step 1. SFT          base + instruction data → SFT model
step 2. Reward model SFT model + (prompt, chosen, rejected) → RM (打分器)
step 3. PPO / GRPO   RM 给奖励 · policy gradient 更新 SFT model
                     + KL penalty 约束不偏离 SFT 太远

问题：RM 自己有 bias、PPO 训不稳、需要 4 套模型同时在显存（policy / ref / RM / value）· 2–4× SFT 成本。

4.2 DPO · 把 RLHF 简化成监督式

DPO paper (Stanford 2023) 数学等价：去掉 RM，直接用 closed-form loss 让 π(chosen) / π(rejected) 比值大于 reference model。训练像 SFT 一样简单 · 一个 loss。 

L_DPO = -log σ( β · [ log(π(chosen)/π_ref(chosen))
                    - log(π(rejected)/π_ref(rejected)) ] )

β: 0.1 ~ 0.5 · 类似 KL temperature
π_ref: 通常就是 SFT 后的 frozen model

4.3 KTO · 不需要成对数据

KTO (ContextualAI 2024): 只需要"这个 response 好 / 不好"二元标签， 不需要 chosen vs rejected 两两比较。生产数据收集成本大幅降低。

4.4 RLAIF · 用 AI 当 labeler

Anthropic Constitutional AI 的核心：不是人类标偏好，是用强模型（如 Claude / GPT-4） 根据一份 "constitution"（一组原则）打分。规模化必经之路，但有"模型 bias 自我强化"风险。

4.5 横评 · 何时选哪个

方法数据需求训练复杂度效果典型场景
SFT(prompt, ideal)★baseline新任务起步
DPO(prompt, chosen, rejected)★★SFT + 1–3%多数 alignment 任务
KTO(prompt, response, ±)★★≈ DPO生产收集 thumbs up/down
IPO / SimPO同 DPO★★稍优 / 减 length biasDPO 出问题时换
PPO (RLHF)(prompt, chosen, rejected) + RM★★★★SFT + 3–8%旗舰模型 · 团队有 RL infra
GRPO同 PPO · 无 value model★★★≈ PPO · 更省内存DeepSeek R1 / Qwen 数学推理

2024–2026 工业界趋势

SFT + DPO 已成为 90% 公司的实际选择——简单 · 可复现 · 效果接近 PPO。 PPO / GRPO 主要在数学 / 代码 / 长程 reasoning（有客观 reward 信号）上仍占优势。

§ 5数据 pipeline · 4 套常见配方

"fine-tune 效果好不好 80% 靠数据" 不是夸张。下面 4 套 pipeline 覆盖大部分实际场景：

5.1 配方 A · 纯人工标注（高质量小数据）

  1. 定义任务 + 示例 (10 个 gold examples)
  2. 招 5–10 个 domain expert 标 500–2000 条
  3. 双标 + IAA (Inter-Annotator Agreement) ≥ 0.7 才算可用
  4. SFT 1–2 epoch · 评 holdout 200 条人审

用例：legal / medical / 高保真 customer service

5.2 配方 B · 模型蒸馏（distillation）

  1. 用 GPT-4 / Claude 生成 5K–50K 条 (prompt, response)
  2. 人审过滤 10–20% 低质 → 留 4K–40K
  3. LoRA SFT 一个 7B / 8B 学生模型
  4. 评测 vs teacher · 通常能逼近 80–90%

用例：把 GPT-4 能力压到自己服务器跑（Vicuna / Alpaca / Phi 都是这思路）

合规警告

OpenAI ToS 禁止用其 output 训"竞争模型"。Anthropic 类似。 如果蒸馏给商业产品 · 用 Apache 2.0 / MIT license 的开源模型生成（如 Mistral / Llama / Qwen） 或买商业授权数据。

5.3 配方 C · self-improve / RLAIF

  1. SFT 一个 baseline (M0)
  2. M0 sample N 个候选 / prompt
  3. 用 reward model（或 GPT-4 当 judge）打分
  4. 选 top-1 当 chosen · bottom-1 当 rejected · 做 DPO
  5. 得到 M1 · 重复

用例：math / code 这类有客观 reward 信号的任务（DeepSeek R1 / OpenAI o1 路线）

5.4 配方 D · 生产日志反向标

  1. 线上记录 (prompt, response, user_thumbs_up_or_down)
  2. thumbs_up 当 chosen · thumbs_down 当 rejected · 配同 prompt
  3. 每周 / 每月做一次 KTO 或 DPO
  4. A/B 新版 · 监控 thumbs_up rate

用例：成熟产品的持续优化 · 几乎免费拿数据

§ 63 真翻车复盘

6.1 翻车 A · catastrophic forgetting

背景：在一个英文 instruct 模型上做 1 万条中文医疗对话 SFT。

症状：fine-tune 后中文医疗任务 +40 分，但英文通用问答能力暴跌 25 分。 "what is RAG?" 模型回答："RAG 是一种检索增强..."（强行用中文回答英文问题）。

根因：catastrophic forgetting · LR 太大 + 数据分布偏移 · 模型把英文权重覆盖了。

修法：

  • ① 用 LoRA 而不是 full SFT（减少权重移动）
  • ② 数据混 30% 通用英文（"replay" 旧分布）
  • ③ LR 砍半 · epoch 减到 1
  • ④ 训练时跑 holdout MMLU · 看到 -3% 立刻 early stop

6.2 翻车 B · DPO length hacking

背景：用 GPT-4 当 judge 自动生成 5 万条偏好对做 DPO。

症状：训完模型在 LMSys Arena 排名上升 · 但实际产品 A/B retention 下降 · 用户反馈"模型变啰嗦了"。

根因：GPT-4 当 judge 有 length bias——同质量下偏好长答案。 DPO 学到了"长 = 好"。

修法：

  • ① 在 judge prompt 里加 "ignore length, judge content quality"
  • ② 改用 SimPO（在 loss 里减去长度归一项）
  • ③ 评测加 "response length distribution" · 监控不能比 SFT 长 > 20%

6.3 翻车 C · LoRA merge 后精度掉

背景：LoRA 训完后 merge 到 base 当 deployment artifact。

症状：在 GPU 上跑 merged model 的精度比 "base + LoRA adapter 分开加载" 差 1–2 分。

根因：merge 时 W + (α/r) BA 在 fp16 / bf16 下有数值精度损失， LoRA 的低秩更新被舍入误差吞掉一部分。

修法：

  • ① merge 时升到 fp32 计算 · 再 cast 回 bf16 存
  • ② 部署时直接保留 adapter（PEFT 库支持 runtime LoRA · 推理几乎零开销）
  • ③ 高 LoRA rank（r ≥ 32）+ QLoRA 时尤其严重 · 保留 adapter 更安全

§ 7评估 · fine-tune 效果怎么测

7.1 三层评估金字塔

  1. 自动 benchmark · 通用能力（MMLU / HumanEval / GSM8K）+ 任务特定（你自己的 holdout）
  2. LLM-as-judge · 用 GPT-4 / Claude 给同 prompt 的 baseline vs new 打分
  3. 人评 / 线上 A/B · 最终真实 · 但慢 · 贵

7.2 必跑的 regression suite

每次新版必跑 · 任何一项掉 >3% 立刻 block 上线：

类别数据集 / 套为什么
通用语言MMLU 5-shot查 catastrophic forgetting 第一信号
代码HumanEval / MBPPfine-tune 容易把 code 能力训丢
数学GSM8K如果 base 有 reasoning 别训没了
