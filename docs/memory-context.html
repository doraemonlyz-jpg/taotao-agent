Chapter 2 · 短期

Short-term · 3 种压缩策略

conversation 长了塞不下 ctx 怎么办

§2.13 种主流策略

策略原理优势劣势
Sliding window只留最近 N 轮简单 · 0 LLM 成本丢早期重要信息
Summary用 LLM 把老 message 压成摘要保留关键 · ctx 节省每 N 轮一次 LLM 调用 · 摘要可能丢细节
Hierarchicalrecent N 原文 + 中段摘要 + 远期 1 句话总结综合最优 · 平衡成本/保真实现稍复杂

§2.2Hierarchical 实战

def hierarchical_compress(messages, llm):
    if len(messages) < 20:
        return messages   # 短不压

    recent     = messages[-10:]                   # 最近 10 条原文
    mid        = messages[-30:-10]                # 中段
    old        = messages[:-30]                    # 远期

    mid_summary = llm.invoke(f"用 200 字摘要这段对话，保留事实和决定：\n{mid}")
    old_summary = llm.invoke(f"用 50 字总结这段早期对话主旨：\n{old}")

    return [
        SystemMessage(f"早期对话回顾：{old_summary}"),
        SystemMessage(f"中段对话摘要：{mid_summary}"),
        *recent,
    ]

§2.3什么时候压 · 阈值

  • token 计数——超过 ctx window 60% 触发压缩 · 留 buffer 给生成
  • turn 计数——满 20 turn 压一次（避免每 turn 都跑 LLM）
  • cost 计数——单次 input token > X · 触发

§3.2Vector-only · 最简实现

def save_memory(text, user_id, metadata={}):
    emb = embed(text)
    vector_db.upsert({
        "id": uuid.uuid4().hex,
        "vector": emb,
        "text": text,
        "user_id": user_id,
        "created_at": now(),
        **metadata,
    })

def retrieve_memory(query, user_id, k=5):
    emb = embed(query)
    return vector_db.search(emb, k=k,
        filter={"user_id": user_id})  # pre-filter 必须

§4.3Letta (MemGPT) · OS-style hierarchy

论文级创新（UC Berkeley）——把 memory 当 OS 内存层级管：

  • main context (RAM)——LLM 实时看到的
  • recall storage (page cache)——上下文外但快查
  • archival storage (disk)——所有历史 · vector DB · 按需 page in

LLM 用tool call 主动 page memory · 不是被动注入。research 友好 · 工程稍重。

§4.4选型决策

  • 简单 chat agent · 想 1 周 ship → Mem0
  • 客户事实经常变 · 需要时间维度 → Zep
  • research / long-horizon agent · 想 LLM 主动管 → Letta
  • 已有 RAG 想加 memory → Cognee（结构 RAG + memory）
Chapter 5 · 写

Memory write · 抽取 / 评分 / dedup

乱写的 memory = 越用越蠢

§5.14 步 write pipeline

  1. Extract——LLM 从 conversation 抽稳定事实（不抽情绪 / 临时上下文）
  2. Score——每条 fact 评 importance（1-5）· filter < 3 的
  3. Dedup——查现有 memory · 相似度 > 0.85 → 合并 / 跳过
  4. Conflict resolve——新 fact 跟旧 fact 冲突 → mark 旧 invalid + 存新

§5.2Extraction prompt 模板

EXTRACT = """
你是 memory 抽取员。从下面 conversation 中抽值得长期记住的事实。

只抽：
✓ 用户的稳定偏好 (品味 / 习惯 / 价值观)
✓ 用户的事实属性 (年龄 / 职业 / 地点 / 关系)
✓ 用户做过的重要决定

不抽：
✗ 情绪变化 (今天心情)
✗ 临时上下文 ("现在在开会")
✗ 已经被新对话推翻的旧信息

输出 JSON list:
[{"fact": "...", "importance": 1-5, "category": "preference|attribute|decision"}]

Conversation:
{conv}
"""

§5.3Dedup · semantic + LLM 二选一

def dedup_or_merge(new_fact, user_id):
    similar = vector_db.search(embed(new_fact), filter={"user_id": user_id}, k=3)
    for s in similar:
        if s.score > 0.92:
            return "skip"   # 太像 · 跳
        if s.score > 0.78:
            # 中等 · 让 LLM 决定 merge / replace / keep both
            decision = llm.with_structured_output(MergeDecision).invoke(
                f"old: {s.text}\nnew: {new_fact}\n应该 merge / replace / both?"
            )
            if decision.action == "merge":
                merged = llm.invoke(f"合并 fact: {s.text} + {new_fact}").content
                vector_db.upsert(id=s.id, text=merged, vector=embed(merged))
                return "merged"
            elif decision.action == "replace":
                vector_db.upsert(id=s.id, text=new_fact, vector=embed(new_fact))
                return "replaced"
    save_memory(new_fact, user_id)
    return "added"
最常踩坑 · 不 dedup

用户连续 5 次说 "我喜欢咖啡" → 5 条几乎一样的 memory · retrieve 时全占 top-k · 其他重要信息被挤出。必须 dedup · 用 0.85+ 相似度阈值。

Chapter 6 · 读

Memory read · 检索 / 排序 / 注入

read 错了比没 memory 还糟

§6.13 阶段 read pipeline

  1. Retrieve——多源召回（vector / KV / graph）top-N each
  2. Rank——按 (similarity + recency + importance) 综合排
  3. Inject——选 top-K 注入 prompt · 决定放哪个位置

§6.2综合 score · 不只看 similarity

def memory_score(memory, query, now):
    sim = cosine(embed(query), memory.vector)
    age_days = (now - memory.created_at).days
    recency = math.exp(-age_days / 30)         # 30 天半衰
    importance = memory.importance / 5
    return 0.6 * sim + 0.2 * recency + 0.2 * importance

权重按场景调——客服重 recency · 偏好重 importance · 事实查询重 sim。

§6.3注入位置 · 别塞 system prompt 头

常见错误：把 memory 塞 system prompt 最前面。问题：

  • system prompt 一改 prompt cache miss · 成本翻倍
  • "lost in middle" · LLM 容易忽略

正确：

messages = [
    SystemMessage(stable_system_prompt),     # 不变 · 全 cache
    SystemMessage(f"用户档案：\n{memory_block}"),  # 变 · 但单独 message
    *conversation_history,
    HumanMessage(current_query),

§7.14 种 forgetting 策略

策略原理用

§7.2Consolidation · 像人睡觉合并记忆

# cron 每周跑
def consolidate(user_id):
    # 1. cluster 相似 memory
    memories = vector_db.fetch_all(filter={"user_id": user_id})
    clusters = kmeans(memories, k=20)

    for cluster in clusters:
        if len(cluster) < 3: continue
        # 2. LLM 抽象成 1 条更高层 fact
        abstract = llm.invoke(f"把这些 memory 合并成 1 条更通用的事实：\n{cluster}")
        save_memory(abstract.content, user_id, importance=5)
        # 3. 老的降 importance（不删 · 留作 reference）
        for m in cluster:
            update_importance(m.id, m.importance - 1)

§7.3软删 vs 硬删

面试常问。建议软删：

  • 加 deleted_at 字段 · retrieve 时 filter
  • 定期归档（> 1 年）到冷存储 · 不立刻物理删
  • 合规要求 (GDPR) → 用户主动 request 时真删

§7.4本项目实践 · recency × usage 复合分 prune

我们的 backend/agent/memory/long_term.py 实现了 §7.1 表里第二档（recency decay）+ 一点 LFU 思想的混合： 

# 写入 · 多两个 metadata 字段
def remember(self, text, ...):
    self.collection.add(metadatas=[{
        ..., "last_used_at": now, "use_count": 0,
    }])

# 读 · 每次 recall 自动 bump 命中行
def recall(self, query, k=4):
    result = self.collection.query(...)
    self._bump_usage(result["ids"][0])      # use_count++ · last_used_at = now
    return result["documents"][0]

# 定期清 · 复合分排序砍底部
def prune(self, *, max_keep=500, half_life_days=14, dry_run=False):
    # score = 0.7 * exp(-age/half_life) + 0.3 * log1p(use)/log1p(max_use)
    scored = sorted(zip(ids, metas),
                    key=lambda p: self._score(p[1], max_use, half_life_days))
    n_drop = min(total - max_keep, total // 2)   # 永远不删超过一半 · blast radius limit
    victims = [pid for pid, _ in scored[:n_drop]]
    if not dry_run:
        self.collection.delete(ids=victims)

三个 senior 取舍点：

  • recency 用指数衰减不是线性 · 14 天后还剩 37% 价值、56 天剩 2% · 半衰期物理直觉
  • usage 用 log1p 归一避免热门 fact 把冷门压成 0 · 长尾分布的常用归一手法
  • 权重 7:3 偏 recency · 因为"用户偏好会变"是 agent 场景的最大 risk · 做 docs RAG 应该反过来 3:7

暴露为 POST /memory/prune，必须支持 dry_run=true 让用户预览 victim 再决定 —— 这跟 terraform plan / kubectl --dry-run 同源。 生产挂 cron 周日凌晨跑一次。

§9.3Context 设计 · 4 区块顺序

顺序很重要·影响 cache 命中率：

  1. Stable system prompt（最稳）—— mark cacheable
  2. User profile / memory（per-user 稳）—— mark cacheable
  3. Conversation history（增长）—— 部分可 cache
  4. Current query（每次新）—— 不 cache

原则：稳定的在前 · 变化的在后 · cache hit 最大化。

§9.41M context · 何时该用

  • ✓ 代码库整库分析（Cursor / Devin）
  • ✓ 整本书 / 整本论文 QA
  • ✓ 长视频 transcript 分析
  • ✗ 客服对话——RAG 更合适 + 成本低
  • ✗ 高频 ChatGPT 风——latency 30s 不可用

§9.5Hybrid · long context + RAG 不矛盾

2025-2026 实战：

  • RAG retrieve top-50 chunk
  • 不 rerank · 全塞 1M context（省 rerank 调用）
  • LLM 自己在 1M 里挑相关

条件：retrieval 后总 token < 500k · 否则 lost-in-middle 还是会咬人。

Chapter 10 · Eval & 翻车

Memory eval + 真实复盘

memory 怎么 eval · 5 个真翻车看明白

§10.1Memory eval · 4 类指标

  • Recall——给个 query · top-K 是否含 ground-truth memory
  • Precision——top-K 中真有用的比例
  • Consistency——同 query · LLM 多次 retrieve 一致吗
  • Forgetting accuracy——该忘的忘了吗 · 不该忘的没忘

§10.2翻车 1 · memory 串号

背景：客服 agent · alice 的偏好出现在 bob 对话里。

根因：memory write 时漏了 user_id filter · vector DB 全局检索。

fix：codereview 强制 memory API 必传 user_id · pre-commit hook 静态检查。

§10.3翻车 2 · memory drift

背景：用户 6 个月前说 "现在住北京" · 后来搬上海 · agent 仍说 "你在北京 · 推荐附近的..."

根因：旧 memory 没 invalidate · 新 memory 也存了 · retrieve 时按 sim 拿到旧的。

fix：

  1. 属性类 memory 用 KV (per-key 1 个 value) 不用 vector (多 value)
  2. vector memory 加 recency decay 让新的优先
  3. Zep 的 bi-temporal model 自动 invalidate

§10.4翻车 3 · cost 暴涨

背景：上线 3 个月 LLM cost 涨 5x。

根因：每条 memory 都加 system prompt · prompt cache 失效（system 一改全 miss）· 所有用户全价。

fix：memory 单独 message · 不混 system prompt · cache hit rate 从 3% 升到 65%。

§10.5翻车 4 · privacy leak

背景：内部 agent 共享 org-level memory · 客户机密 prompt 出现在另一客户 conversation。

根因：org-level memory 没 PII scrub · 直接把 raw conversation 入了 shared store。

fix：

  1. org-level memory 必须 LLM 抽抽象 fact 不是原文
  2. Presidio 兜底 PII 检测
  3. access control 强制 audit log

§10.6翻车 5 · memory 越用越多 · retrieve 噪声大

背景：上线 2 年 · 老用户每人有 10000+ memory · top-5 经常完全无关。

根因：没 forgetting · 没 consolidation · 重复 memory 占满 top-K。

fix：

  1. 每周 cron 跑 consolidation（§7.2）
  2. importance < 2 的 90 天硬删
  3. retrieve top-30 → LLM rerank → top-5（多一步过滤）
