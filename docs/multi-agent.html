§1 什么时候 不 该用 multi-agent · 默认答案：默认别用

所有 multi-agent 章节之前先讲这一章 · 这是 senior 面试的第一个识别信号： 候选人能不能讲出"什么时候单 agent 就够"。

OpenAI · Anthropic 的官方建议

  • OpenAI Practical Guide to Building Agents (2024) · 原文："start with a single agent and add complexity only when necessary"。
  • Anthropic engineering blog · "Building effective agents" (Dec 2024) · 列了 5 种 workflow（prompt chaining / routing / parallelization / orchestrator-workers / evaluator-optimizer），并明确："单 agent 应是首选，multi-agent 只在以下三种情况下值得"：
    1. 任务有明显的并行子任务（research 5 个产品的对比 → 5 路并发）
    2. 任务需要多个独立 perspective 才能保证质量（红队审查）
    3. 不同步骤需要极不同的工具集 / 模型（vision agent + code agent）

反例 · 这 4 种 90% 的人选了 multi-agent · 错的

反例 1 · "我让 PM agent 跟 dev agent 协作"

大多数情况 PM 部分就是一段 system prompt。两个 agent 在 内部"对话"只会让 token 翻 3 倍，效果还不如 actor + critic。

反例 2 · "用 supervisor 路由不同 worker"

很多 worker 之间差异只是 prompt 不同 · 一个 agent 带不同 system + 工具集 切换就够。supervisor 反而成单点。

反例 3 · "10 个 agent 模拟一家公司做项目"

对 demo 视频很惊艳，对真实任务 = 上下文爆炸 + 错误传播 + 调试地狱。AutoGen 早期 demo 全是这样，但生产没人用 10 个 agent。

反例 4 · "我用 voting 提升对话质量"

voting 只在有限答案空间有效。开放生成做 voting 等于多花 N× 钱然后 LLM judge 凑合挑一个，准确率不升。

判断框架 · 3 道题

  1. 如果我把 multi-agent 折叠成 1 个 agent + 多个 system prompt section + 多个 tool · 还能完成任务吗？能 → 别用 multi-agent。
  2. 这些 agent 之间有真正独立的输出吗？还是只是步骤拆开？只是步骤拆 → 用 prompt chaining，不用 multi-agent。
  3. 每个 agent 是否需要不同的 model / 工具集 / 上下文窗口？否 → 不用。

三个都不能立刻 yes，回到单 agent。本书剩下 9 章都假定你已经过了这一关。

§2 模式 1 · Hierarchical / Supervisor router

是什么

一个 supervisor LLM 在每个回合决定下一步交给哪个 worker · worker 跑完返回 supervisor · supervisor 再决定下一步或终止。LangGraph 官方教程的 default multi-agent pattern · 我们项目的 graph 版就是它。

结构 · 我们的 graph 版

          ┌──────────────┐
   user──▶│  supervisor  │ ◀──── (decides next)
          └──┬─────┬─────┘
             │     │
       ┌─────▼┐  ┌─▼────┐  ┌──────┐
       │resear│  │coder │  │writer│
       │cher  │  │      │  │      │
       └──────┘  └──────┘  └──────┘
                    │
                ┌───▼──┐
                │critic│
                └──────┘

实现见 backend/agent/graph.py + nodes/supervisor.py。supervisor 用 get_fast_llm() 跑 · 节省 cost。worker 用主模型。

何时用

  • 任务步骤清晰、可枚举 · 有明显的 worker 边界（research / draft / review）
  • 需要显式的工作流可视化（LangSmith 拓扑图、给老板看的"agent 在干啥"）
  • 不同 worker 用不同 system prompt 或不同模型（researcher 用 retrieval-aug，coder 用 structured output）

失败模式 · 3 个真实坑

坑 1 · supervisor 自己塞太满

每个 worker 的输出都堆在 supervisor 上下文里 · 跑 6-8 步后 supervisor token 爆。 解：worker 输出过 summariser 再塞给 supervisor · 不要原文。

坑 2 · supervisor 反复 ping 同一个 worker

因为 prompt 没说清"已经做过的事别重做"。解：supervisor prompt 里维护 past_actions 列表 · 显式提示"researcher 已被叫过 2 次"。

坑 3 · 没有终止条件

supervisor 一直觉得"还能更好" · 跑到 max_steps 才停。解：定义明确的 "done" tool（final_answer）+ 一个独立的 quality gate（critic）。

对应代码

我们的 supervisor / worker 节点都在 backend/agent/nodes/。看图谱用 backend/agent/graph.py 里的 build_graph()。

§3 模式 2 · Hub-and-spoke / Sub-agent-as-tool

是什么

主 agent 不让出控制权 · 而是把 sub-agent 当成一个普通 tool调。主 agent 在 tool_call 阶段说 dispatch_subagent(role="researcher", task="...")， sub-agent 在隔离 context里跑完，把简短结论返回作为 tool result。

谁用

  • Claude Code 的 Task tool · 全部官方 sub-agent 都是这个模式
  • Cursor Agent / Devin · 主 loop 派 sub-loop 跑 verify / search
  • 我们的 harness 版 · dispatch_subagent 在 backend/agent/harness/subagent.py

对比 supervisor 的优势

主 ctx 不被污染

sub-agent 跑了 20 个 tool call · 主 agent 只看到一段 200 字总结。 Claude Code 主 ctx 不爆的关键就是这个。

主 agent 仍是决策者

不像 supervisor 把"决定 worker"和"做 worker"切两层 · 主 agent 同时干两件事 · 模型 reasoning 链不被打断。

对比 supervisor 的劣势

  • 主 agent prompt 里要列清楚每个 role 能干嘛 · 否则 LLM 不知道派谁
  • 不容易并发派 N 个相同 role · 我们靠 asyncio.gather 在 loop 里手工做
  • 没有跨 sub-agent 通信 · sub-agent 互相不知道对方存在

看代码

backend/agent/harness/subagent.py · 80 行 · _run_inner_loop() 把所有 sub-agent 状态隔离 · 主 loop 只 await 一个 future。

@tool
def dispatch_subagent(role: str, task: str, max_steps: int = 8) -> str:
    """Spawn a sub-agent in an isolated context to run `task`."""
    # 主 ctx 看到的就是 sub-agent 的 final_answer · 不看中间过程
    return _run_inner_loop(role, task, max_steps)

§4 模式 3 · Debate · 多视角辩论

是什么

N 个 persona（典型 3 个：optimist / skeptic / synthesiser）轮流就同一个问题发言 · 每一轮看到所有人前一轮的回复 · 最后一个 synthesiser 写共识。Du et al. 2023 · "Improving Factuality and Reasoning in LLMs through Multiagent Debate" 实证 +6-12% 在 reasoning benchmark 上。

典型 transcript

[optimist · r1] We should ship feature X · users are asking for it.
                 ANSWER: ship X
[skeptic · r1]  But X needs DB migration · risk of downtime > value.
                 ANSWER: defer X
[synth · r1]    Ship X behind a feature flag · default off · roll out 1%.
                 ANSWER: ship X behind flag

[optimist · r2] Synthesiser's flag plan is fine · agreed.
                 ANSWER: ship X behind flag
[skeptic · r2]  If we limit to 1% the migration risk is acceptable. agreed.
                 ANSWER: ship X behind flag
[synth · r2]    Consensus reached.
                 ANSWER: ship X behind flag
→ converged at r2 · final = synth's answer

我们的实现

backend/agent/multi_agent/debate.py · 关键设计：

  • 每个 agent 强制 end with ANSWER: <...> 行 · 方便 normalise + 检测收敛
  • 收敛检测：先 string 全等，否则 fast LLM 判 "all agents agree?" 一字 yes/no
  • 终止：收敛或 max_rounds（默认 3）
  • 最后还有一轮独立的synthesis · 让一个 fresh LLM 读完整 transcript 写最终答案 · 避免最后一个发言的 agent 偏向

何时用 · 何时别用

用

  • 有真正争议的决策（PM 评审、架构选型、安全分析）
  • 需要红队 + 蓝队对抗（jailbreak detection、policy review）
  • 问题没有 ground truth但需要 robust 的判断

别用

  • 简单事实问答 · debate 只会让 token 翻 3-6 倍
  • 有 ground truth 的题目（math、code）· voting / reflection 更好
  • 低 latency 要求的对话场景 · debate 一般跑 6-9 个 LLM call

失败模式

  • Agree to disagree · 两个 persona 立场太刚 · 永远不收敛 · 必须靠 max_rounds 兜底
  • Persona collapse · 跑到 r2 r3 大家都被同化成同一个声音 · 解决：每轮重申 persona prompt
  • Synthesiser bias · 最后一个 agent 决定 final · 解决：用独立的 fresh LLM 做 synthesis（我们就是这么做的）

§5 模式 4 · Voting · 自一致性 (Self-Consistency)

是什么

Wang et al. ICLR 2023 · "Self-Consistency Improves CoT Reasoning"。 对同一个问题用 temperature>0 跑 N 次 · 答案规范化后取众数。在 GSM8K 上 N=40 时 PaLM 540B 提升 +17.9%。生产里一般 N=5-7 就 saturate。

关键三件事

  1. 独立性 · N 个 voter 互相不看对方答案 · 这是和 debate 的根本区别
  2. 归一化 · 答案要能 normalise 到同一个键去数票（lowercase + 去标点 + 截 200 字）
  3. tie-break · 平票时用 fast LLM judge 选"justification 写得更好的那个"

我们的实现 · 60 行

# backend/agent/multi_agent/voting.py
async def _gather(task, n, sid, temperature):
    return await asyncio.gather(*[
        asyncio.to_thread(_one_voter, f"v{i+1}", task, sid, temperature)
        for i in range(n)
    ])

def run_vote(task, *, n=5, temperature=0.7, session_id=""):
    votes = asyncio.run(_gather(task, n, session_id, temperature))
    counter = Counter(_normalise(v["answer"]) for v in votes)
    top, top_count = counter.most_common(1)[0]
    tied = [a for a,c in counter.items() if c == top_count]
    if len(tied) > 1:
        # tie-break · fast LLM picks best-justified raw answer
        ...
    return Result(final=..., meta={"distribution": dict(counter), ...})

何时用 · 何时别用

用

  • 数学 / 推理题 · 答案是数字或 A/B/C/D
  • 是非判断 · 安全分类 · 标签预测
  • 关键决策（投资、医疗）且有可枚举答案

别用

  • 开放生成（写代码、写文章）· 答案空间太大没法 normalise
  • 对话 / 多轮交互 · voting 一次性 · 没法增量
  • 低延迟 · N=5 = 5× 单次延迟（除非完美并发）

实战经验

  • N 不是越多越好 · 实证 N=5-7 saturate · N=11 多花钱无收益
  • temperature 必须 > 0 · 否则 N 个 voter 输出一模一样 · 投票变摆设
  • 异常值要丢 · 偶尔一个 voter 输出 "I don't know" · normalise 之后会被 counter 当成单独一票 · 实际上应该 drop · 我们没做这一步 · 你可以加

§6 模式 5 · Handoff · OpenAI Swarm 风

是什么

OpenAI 在 2024-10 开源 Swarm，提出 "Routines & Handoffs" pattern · 没有 supervisor · 每个 agent 自己决定要不要把整个对话上下文 传给另一个 agent。终止条件 = 某 agent 不调用 transfer_* tool 而是直接返回文本。

对比 supervisor 的本质区别

SupervisorHandoff (Swarm)
谁决定下一步专门的 supervisor LLM当前正在跑的 agent 自己
额外 LLM 调用每步多 1 个 supervisor 调用0 · 决策内嵌
上下文传递worker 输出 → supervisor 摘要 → 下个 worker整个 message history 直接传
终止supervisor 决定当前 agent 不再 transfer 即终止
可视化清晰 graphflat sequence

典型场景 · 客服 triage

user: "我想退款 + 顺便问一下另一个订单到哪了"

[triage]    分类 · 退款 + 物流 · 先转 refund agent
            TRANSFER: refund_agent

[refund]    处理退款 · 询问订单号 · 退款发起
            ... 跟用户两轮对话 ...
            完事 · 转物流
            TRANSFER: shipping_agent

[shipping]  查物流 · 给出 ETA
            (no transfer · 自然终止)

我们的实现 · 简化版

backend/agent/multi_agent/handoff.py · 故意不用 langchain agent 抽象 · 纯 dict + while-loop · 100 行：

  • 每个 agent 的 system prompt 末尾 append "其它 agent 列表 + 转交语法"
  • parser 抓 trailing 行 TRANSFER: <name> · 没抓到 = 终止
  • max_hops 兜底防 ping-pong
  • 非法目标 / 自我转交 · 注入 system 提示让 agent 重写

失败模式

  • A↔B ping-pong · 解：max_hops + same-name 检测 · 还可加"距离上次同名 ≤ 1 hop 即拒"
  • Context bloat · message 一直传下去 · 解：在 transfer 时强制压缩历史 · 我们没做 · 生产建议加
  • 失忆 · agent A 收集的 memory 没传给 agent B · 解：用共享 state（见 §10）

§7 模式 6 · Reflection · Actor + Critic 循环

是什么

两篇姊妹论文：

  • Reflexion (Shinn et al. NeurIPS 2023) · agent 跑完一遍 task · 写 self-reflection · 下一轮带着反思再跑
  • Self-Refine (Madaan et al. 2023) · actor → critic 给 feedback → actor 改 · 循环

两者本质一样 · 都是actor + critic 内循环。我们实现里两个 agent 用不同 model （actor 用主模型 · critic 用 fast 模型）· 既省钱又减少 self-confirmation bias。

结构

round 1: ACTOR  ── draft ──▶ CRITIC ── score:6 + fixes ─▶
round 2: ACTOR  ── revise ─▶ CRITIC ── score:8 ──────── (≥ pass_score) STOP
final: 返回 actor 最后一版 draft

我们的实现 · 关键点

  • critic 强制输出 SCORE: N (1-10) + ISSUES: + FIXES: · 解析失败 fallback 到 score=5（保守）
  • actor 第二轮拿到 critique 必须 output FULL revised draft · 不是 diff（避免半成品）
  • 停止条件：score ≥ pass_score（默认 8）或 max_rounds（默认 3）· 用 max_rounds 兜底
  • critic 用 get_fast_llm() · cost 砍 5×

真正的应用场景

用

  • code generation · critic 跑 lint/test · actor 改
  • 长文写作 · critic 检查结构 / 论据 · actor 改
  • RFC / 设计文档 · critic 找缺失章节
  • SQL / config · critic 跑 dry-run · actor 改

别用

  • 对话 / 客服 · critic 没法判断对话质量
  • 事实问答 · 第一轮答对了再跑 critic 是浪费
  • 没有客观 rubric 的任务 · critic 没标准只会给 7 分凑合

失败模式 · 必须警惕

  • Self-confirmation · actor + critic 用同一 LLM 同 prompt · critic 永远说"good job · score 9" · 解：critic 用不同模型或更强 system prompt（"be brutal"）
  • Hallucinated fixes · critic 提的 fix 实际并不能改善 · actor 改了反而更差 · 解：跑实际 verifier（lint / test）替代 LLM critic
  • Score inflation · 第二轮 critic 看到自己上次的 critique 被采纳 · 倾向于打高分 · 解：critic 不带历史 · 每轮都是 fresh LLM call

面试时的 framework 问题

  • "Why LangGraph 而不是 AutoGen？" · 答：LangGraph 把状态显式做成图 · workflow 可枚举 · debug 好做。AutoGen 的 group chat 是涌现的 · 调试时很难复现。
  • "为什么 OpenAI Swarm 没火？" · 答：Swarm 官方 readme 写明"experimental, not for production"。它的价值是教学 · 让你理解 handoff 不需要 framework。
  • "CrewAI 适合什么？" · 答：非工程团队用 yaml 拼 demo · 但生产 agent 一般会被重写。

§9 我们的实现 · 一个 tool 包住 4 种模式

设计

我们没把 multi-agent 做成一个独立"模式开关" · 而是包成一个 harness tool 让 LLM 自己挑：

@tool
def multi_agent_run(
    pattern: Literal["debate", "vote", "handoff", "reflection"],
    task: str,
    n: int = 3,
    max_rounds: int = 3,
    pass_score: int = 8,
) -> str:
    """Run an inner multi-agent group on `task` and return the result."""

主 agent 在 plan 出 "this needs N-vote self-consistency" 时主动调 multi_agent_run("vote", task, n=5) · 拿到 JSON summary（final + meta + last 4 trace events）· 继续主循环。

4 个 pattern 的 1 行调用

from agent.multi_agent import (
    run_debate, run_vote, run_handoff, run_reflection, AgentSpec,
)

# debate · 默认 3 个 persona
r = run_debate("Should we deprecate API v1 in Q3?", max_rounds=3)

# vote · self-consistency
r = run_vote("What is 487 × 312?", n=5)

# handoff · 自定义 agents
r = run_handoff(
    "I want a refund and to track another order",
    [AgentSpec("triage", "..."), AgentSpec("refund", "..."), AgentSpec("ship", "...")],
    entry="triage",
)

# reflection · code review style
r = run_reflection(
    "Write a python function to merge k sorted lists in O(n log k).",
    max_rounds=3, pass_score=9,
)

print(r.final)        # 最终输出
print(r.meta)         # 模式特定 stats
print(r.trace[:3])    # 关键 events

统一 Result 形状

所有 4 个 pattern 都返回 multi_agent.Result：

@dataclass
class Result:
    final: str                       # 最终答案 · 给用户/上层 agent
    trace: list[dict]                # [{agent, round, text}] · 调试用
    meta: dict                       # rounds / votes / handoff path

统一形状的好处：

  • 外层 wrapper 不用为每个 pattern 写不同序列化
  • trace 可以直接扔进 data/traces/*.jsonl · 复用现有 replay 系统
  • meta 是 pattern-specific 但对调用方透明 · 调用方按需读

§10 4 大共性难题 · 状态 · 通信 · 终止 · 评估

不论选哪个 framework / pattern · 这 4 个问题你都要回答。面试时被问到 multi-agent 深问 · 90% 落在这里。

难题 1 · 共享 state

Agent 之间要不要共享内存？

  • 不共享（debate / vote）· 每个 agent 独立 ctx · 简单 · 但"昨天同事讨论的结论"不能复用
  • 全共享（handoff）· message history 整段传 · 上下文爆炸风险高
  • 结构化共享（LangGraph TypedDict）· 显式声明哪些字段在哪些 agent 之间流动 · 工程量大但可控
  • 外存共享（blackboard）· agent 都读写同一个外部 KV / vector DB · 适合长寿命 agent

实战经验：能不共享就不共享。需要时优先用 LangGraph 的 channel · 最后才考虑 blackboard。

难题 2 · 通信协议

Agent 之间怎么"说话"？三种主流：

  1. Natural language（debate / handoff）· 简单 · 但解析脆弱（容易遇到"agent 不按格式输出"）
  2. Structured tool call（hub-and-spoke）· 主 agent 用 tool_call 派 sub-agent · 强类型 · 但 sub-agent 输出仍是 NL
  3. Message bus / event stream（高级）· agent 之间发 typed events · 适合长寿命 service · 工程极重

建议：从 NL 起步 · 遇到解析问题改 structured · 不要一开始就上 message bus。

难题 3 · 终止

什么时候停？

  • 显式 final tool（hub-and-spoke）· LLM 决定 done
  • 共识检测（debate）· LLM judge 判 "all agents agree"
  • 分数阈值（reflection）· critic score ≥ pass_score
  • 不再 transfer（handoff）· agent 返回普通 text
  • Hard cap（所有模式）· max_rounds / max_hops 兜底 · 必须有

反模式：没有 hard cap。任何 multi-agent 都必须有最后兜底 · 否则一个无限 loop bug 烧爆账单。

难题 4 · 评估

怎么知道我的 multi-agent 系统真的比单 agent 好？

  1. A/B 评估：用 Book 14 的 eval framework · 同一组 golden cases 跑单 agent vs multi · 看 LLM-judge 分
  2. 成本对比：multi-agent 通常 3-6× cost · 看 cost-adjusted accuracy
  3. 失败案例分析：multi-agent 失败模式跟单 agent 不一样（self-confirmation / debate 不收敛 / handoff ping-pong）· 单独分类统计
  4. Trace 复盘：用 multi_agent 的 emit events 回放 · 看哪一步最先偏

面试金句

"Multi-agent 不是更厉害的 agent，是更复杂的 agent。复杂度的代价必须用 可衡量的提升来 justify。如果 A/B test 显示单 agent 不输，那就回去用单 agent。 我做过的真实 production multi-agent 系统都是被 single-agent 替换过 ≥ 1 次之后才决定上的。"
