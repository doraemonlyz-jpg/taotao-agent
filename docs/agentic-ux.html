§1 6 大新约束 · agent vs 传统 SaaS UX

维度传统 SaaS UIagent UI设计含义
响应时间200ms 内2s-30min 不等必须 streaming + progress + cancellation
结果可预测"button 必返回 X""可能返回 X · Y · 错的"必须显示 confidence / 来源 / 让 user 验
副作用边界明确（write 才会改）读和写界限模糊高风险动作必须 plan-confirm
错误信号红框 + "invalid"静默错误（幻觉）自检 + cite + critic 节点
状态可逆edit 后 ctrl-z跑过的 tool 不可逆plan + dry-run + 操作前预览
cost 透明flat 月费每次都消耗 tokencost meter · 预算上限 · 提示

"信任曲线" · agent 产品的核心设计目标

每一次 user 用 agent 都是一次信任投票。曲线大概长这样：

  • 第 1 次惊艳 · trust + 50
  • 第 2 次出错没解释 · trust - 80（不对称 · 一次失败抵 5 次成功）
  • 第 3 次出错有解释 + 能 undo · trust - 5（透明 + 可逆 = 几乎不掉信任）

所以本书 12 个模式可以归为 2 大原则：(1) 让用户看见过程（streaming · trace · cite · cost）+ (2) 让用户能介入或撤回（plan-confirm · interrupt · diff · undo · share / fork）。前者建透明 · 后者建可逆。

面试信号

被问 "agent UX 跟普通 SaaS 区别在哪"，低分回答："chat-style 会话"。 高分回答：响应不可预测 + 不可逆 + cost 透明 → UI 必须做透明 + 可逆 · 给 streaming · trace · plan-confirm · undo 4 个具体例子。

§2 模式 1 · Streaming · 比"加载圈"早 1.5s 给反馈

① Streaming token-by-tokenalways

用户认知：第一个字 200ms 出现 → "agent 听见了" · 实现：SSE / WebSocket · 难点：reconnect / 错乱顺序

什么时候必须用

所有 user-facing chat。哪怕模型 100ms 给完，也要走 streaming —— UX 一致性 比性能值钱。

什么时候不用

background agent · CLI 跑批 · webhook 触发 · 用户没在等的场景。

实现要点（本仓库 SSE 用法）

# backend · FastAPI SSE
async def stream():
    async for chunk in graph.astream_events(state, version="v2"):
        yield f"data: {json.dumps(chunk)}\n\n"
    yield "event: done\ndata: {}\n\n"

return StreamingResponse(stream(), media_type="text/event-stream")
// frontend · fetch + ReadableStream + 行解析
const r = await fetch(url, { method: "POST", body });
const reader = r.body.getReader();
while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  for (const event of parseSSE(value)) push(event);
}
第一个 token < 500ms · 持续 flush · 收到 done event 后再 commit final
先 buffer 完所有 token 再 render · 假 streaming 用 setInterval 切字

§3 模式 2 · Trace 透明化 · 把黑盒变玻璃盒

② Trace panelpower-user

用户认知："它在做什么？" 答案随时可看 · 实现：right rail · 节点 + tool call 流式 · 本仓库：frontend/src/components/TracePanel.tsx

分几层信息

  • Always visible：当前 node 名 + tool 名 + 已用 tokens
  • Click to expand：tool input/output · LLM prompt（可选 mask 长 system）· token 用量
  • Hidden by default：full message history · 仅 power user 开

为什么不只 "完成后给 trace"

失败时用户看不到 = 走开。trace实时显示 = 用户能在第 3 个 node 出错时立刻发现 · 主动 stop · 不浪费 30s 才看到红字。

权衡

  • Pro：信任 ↑↑ · debug 自助 · 用户帮你发现 hallucination
  • Con：占屏 · 移动端不好放 · 系统 prompt 可能泄露
默认 collapsed · 一键展开 · 节点名带 emoji 让非工程也能看懂
把 trace 当 debug log 倒给用户 · 一堆 JSON · 没层次

§4 模式 3 · Plan-then-confirm · 高风险操作

③ Plan + confirmhigh-stakes

用户认知："我先看你想干什么 · 满意才放行" · 来源：Claude Code plan mode · 本仓库：industrial.html §4 提过

什么时候用

  • 多步 + 不可逆（删文件 · 跑 migration · 发邮件）
  • 跨多个工具（自动决策风险叠加）
  • 大额（> 一定 cost / token / API call）

UI 形态

  1. 用户输入 → agent 进入 plan mode（不真调工具）
  2. agent 回："我打算做 1 / 2 / 3 · 预计耗时 / cost"
  3. 用户：edit plan / approve / reject
  4. approve → agent 真跑 · 每一步对照 plan 显示进度

实现要点

# 在 system prompt 加：
PLAN_MODE = """When in plan mode:
1. Don't actually call any tools
2. Output a plan with format: STEP 1: ... STEP 2: ...
3. End with "Proceed? [y/n]"
4. Only execute when user confirms"""
plan 用结构化（带步骤号 + 预估 cost / 影响）· UI 给单个 approve 按钮
默认所有 query 都先 plan · 5 字 query 也要确认 · 用户烦走

§5 模式 4 · Interrupt / cancel · stop 按钮的设计

④ Interruptalways

用户认知："我看出走偏了 · 让我喊停" · 客户端：AbortController · 服务端：cancellation token / disconnect handling

3 档 stop 语义 · 选错一档体验差

档含义用户期待
Soft停 stream · 已经跑的 tool 跑完 · cost 照算"你别再说了 · 但已花的不退"
Hard立即 abort · 进行中的 tool kill · checkpoint 不写"全部撤销"
Pause停在当前 step · 可改 prompt 后 resume"我想插一句话再继续"

大多数 chat agent 用 soft 就够。Cursor BA / Devin 这种长跑必须给 hard。 Claude Code 的 plan mode 类似 pause（plan 完等用户决定）。

本仓库实现（soft）

// frontend
const controller = new AbortController();
fetch(url, { signal: controller.signal });
// 按 stop 按钮:
controller.abort();
# backend · FastAPI 自动检测 disconnect
async def stream(req: Request):
    async for chunk in graph.astream_events(...):
        if await req.is_disconnected():
            break  # graceful
        yield ...
stop 按钮一直在 · 立刻反馈"已停" · 已收到的内容保留可见
stop 后清空 chat / 静默无反馈 / 5s 后才生效

§6 模式 5 · Citation · 跟着流式同步出引文

⑤ Inline citationRAG / search

用户认知："这是哪来的 · 我能查证" · 典型：Perplexity 把 [1] [2] 边写边出 · 实现：tool result → 带 url 列表 → LLM 输出时引用 → 前端解析 [n] 转链接

3 种 cite UI 形态

形态例适合
Inline number"... 是 X [1][2]"长文 / report
Hover popup悬停显示 source title + snippet桌面 chat
Right-rail list引文数字单独一栏移动 / 学术

关键：cite 要在同一 stream 出

错误做法：先输出 answer · 答完再单独给 sources。用户读到 "X 是 Y" 时第一反应是"靠谱吗"， 答完才给来源等于晚了 5s。正确：在每个事实后立刻 [n]，前端边收边渲染。

对幻觉的克制

规则："任何声明必须 cite · 否则不输出"。在 system prompt 强约束 · 模型实际服从率 70-90%。 再加一道 critic 节点 · 无 cite 的 sentence 触发 fail。

number cite + 鼠标 hover 显示 source title + snippet · 点击新 tab 打开
把所有 source 在最末尾 dump 一段 URL 列表 · 用户对不上是哪句话来的

§7 模式 6 · Diff preview · 改之前先看

⑥ Diff previewcode / config

用户认知："让我看你要改哪几行" · 典型：Cursor inline diff · GitHub PR 风 · 关键：颜色 + 行号 + 一键 apply / discard

什么时候必须用

  • 修改文件（code / yaml / config）
  • 修改 DB 行 (UPDATE / DELETE)
  • 编辑 doc · CMS · spreadsheet

颗粒度选项

逐 hunk apply 比"all in one button" 体验好 10 倍。Cursor 让用户对每个 hunk 单独 accept/reject · 甚至 split hunk 部分接受。

实现要点

  • 用 unified diff 格式（+/- 行）让 LLM 输出 · 别让 LLM 输出"全文新版"（cost + bug）
  • 前端用 monaco-editor diff view 或 react-diff-viewer-continued
  • apply 时校验 file checksum 没变 · 否则告诉用户 "文件被改过 · 要 re-diff"
行级红绿 + 一键 accept/reject 每个 hunk · 失败 fallback 到手动 copy
直接覆盖文件 · 给 toast "已修改"·没法回滚

§8 模式 7 · Undo / time-travel · 后悔药

⑦ Undo / branchtrust builder

用户认知："万一不对我能退" · 实现层：LangGraph checkpointer 自带 history · UI 给 timeline + "fork from this point"

2 种"撤回"语义

  1. Undo last turn：删除最后一组 user/assistant 消息 · 回到之前 state · 简单
  2. Time travel：选某个历史 checkpoint · 改 prompt · 从那点 fork 出新分支 · 强大但 UI 复杂

谁需要谁

  • 普通 chat · undo last turn 够（ChatGPT 的"重新生成"就是这）
  • code agent / research agent · time travel 必须（Devin / Cursor BA 都给）

注意 side-effect

undo 只回退 LLM 状态 · 已经发出去的邮件 / 已经写到 DB 的行 · 不会撤。 UI 必须明确："已撤销对话 · 但已执行的操作不可逆"。要真撤销 → saga / compensating transaction 设计 · 不在 UI 层。

undo 按钮 + 显示影响（哪些消息被删 · 哪些 side-effect 不可逆）
undo 默默回退 · 用户以为外发邮件也撤了 · 后果严重

§9 模式 8 · Cost meter · 让用户感知钱

⑧ Cost meterSaaS / paid

用户认知："这次花了多少 / 还剩多少额度" · 本仓库：TokenMeter.tsx + /usage/me

3 个时间窗口

窗口显示放在哪
per-turn本轮 tokens / $ / latency消息下方 inline
per-session本对话总计 + 预算条顶部 chip 或 sidebar
per-day / month用户额度剩余profile / settings

设计哲学

"Numerical 而不是 emoji"。用户付费产品 · 他们想看 $0.0042 而不是 "🟢 cheap"。 数字给得越精确 · 信任越强。但同时给一个 % bar · 一目了然没烧爆。

"软上限" 而不是 hard fail

到 80% 弹"今日额度即将用尽"·100% 给 "降级到便宜模型 / 升级套餐 / 等明天" 三个选项 · 不要直接 429 给个 "rate limit" 让用户莫名其妙。

实时 token / $ 数 · 80% 时温和提醒 · 100% 给 3 个 next step
隐藏 cost 直到爆账单后 email 通知 · 用户失控感拉满

§10 模式 9 · Suggestions · 引导用户问下一句

⑨ Suggested follow-upsretention

用户认知："我下一步该问啥？给我灵感" · 典型：ChatGPT / Perplexity 答完给 3 个延伸问 · 本仓库：可加在每条 assistant 后

2 种生成方式

  1. Inline by main LLM：在 final answer 后用 special token 段输出 3 个 suggestion · cheap
  2. Async by fast LLM：answer 完后用 Haiku 跑一次 · 解析 user / answer 出 3 个

什么时候不建议

  • 已经在做某个明确任务（生成完代码 · 用户要 review · 别打扰）
  • 用户是 power user (cmd+k · 不需要被引导)

chat 空状态版（onboarding）

把 suggestion 当 demo prompt 给：分类 tab + 卡片网格。本仓库 frontend/src/components/ExamplePrompts.tsx + frontend/src/prompts.ts 14 个 curated prompts。

3 个 · 短 · 跟当前对话相关 · 一条点击直接进 input 框
5+ · 长句 · 跟当前对话无关 · 像 ad 一样塞

§11 模式 10 · Empty state / onboarding

⑩ Empty stateactivation

用户认知："这玩意能干啥？" · 本仓库：ExamplePrompts tab + 卡片

3 件 must-have

  1. 1 句话定位："Ask me about your data, codebase, schedule"
  2. 3-12 个 example prompt·按能力分类 · 卡片可点击直接 send
  3. 1 个"看 demo"·video / GIF / 实例 chat replay

反例 · agent 产品最常 underdeliver

"Hello! How can I help you today?" + 输入框。用户：不知道你能做什么·关页面。 activation rate 低 30-50% 起。

更高阶 · personalised empty state

对老用户 · empty state 显示 "Continue last session: 阿里 ADS 退款" + "Resume: SWE-bench debug"。 Cursor 的 sidebar 就这么干。

分类 tab (tools / memory / code / RAG) + 14 卡片 + 双语 + engine 推荐
空白页 + 一个输入框 + "我能帮你什么"

§12 模式 11 · Error recovery · 失败也要友好

⑪ Error recoverytrust

用户认知："失败了我能干嘛 · 不要让我重新打字" · 典型：retry · regenerate · simplify · open ticket

错误分类 · 不同 fix 路径

类例UI
Transient5xx / rate limit / timeout"出错了 · 已自动重试 1 次 · 还是失败"·按钮 retry
Quota429 / 余额不足给具体数字 · 升级套餐 / 等 reset 时间 / 切便宜模型
Tool failedweb_search 0 hits显示工具名 + 简短原因 · 让 user 改写 query
Hallucination report用户标"这答案错了"thumbs down + 一段 textarea + 自动转人工
Permissiontool 不在 scope"我没权限 X · 联系管理员 · 可点链接"

"actionable" 是最大原则

错误信息必须告诉用户下一步能做什么 · 不只是 "Internal Server Error"。哪怕你 server 真挂了 · 也写 "Something went wrong on our end · Click to report or try again in 30s"。

error 卡片 = 一句话原因 + 一个 retry / regenerate / report 按钮
弹 toast "Error" 然后 chat 卡死 · 用户除了 refresh 没招

§13 模式 12 · Share / replay / fork · 协作维度

⑫ Share / replay / forkteam / virality

用户认知："给同事看 · 让 ta 接着改" · 典型：ChatGPT share link · Cursor BA 的 PR · Devin 的 timeline 公开

3 种"分享"语义

语义能做什么典型实现
Read-only share看消息 + trace · 不能继续public URL · markdown export
Replay从某个 state 复刻一份给别人共享 thread_id snapshot
Fork从 share 链接开始一个新 threadcopy-on-write thread

对你产品意味着什么

  • Virality：share link 是免费 marketing
  • 支持：用户给 share link 报 bug 比纯文字描述清晰 10×
  • 团队协作：B2B 需要"团队成员能看到/继续别人的 chat"

必做的 privacy 设计

  • share link 默认不含 PII（output filter 跑一遍）
  • 有效期可设（24h / 7d / 永久）
  • share 后 owner 还能撤销（generate-token 模式而非 hash 模式）
  • 明示分享者："你正在分享 · 包含 X · 不包含 Y"
share 前 preview + 显示 PII redact + 有效期可选 + 一键撤销
share = 直接 dump 整个 thread 含 system prompt + 私信 · 永不过期
