§1.42026 年 AI Infra 5 个最热话题

  1. Disaggregated Prefill / Decode Serving——把 LLM 推理的 prefill 阶段（计算密集）和 decode 阶段（带宽密集）分到不同 GPU pool 上跑，整体 throughput 提升 2-5x。代表系统：Mooncake (Moonshot) / DistServe / Splitwise (MS)。

  3. Speculative Decoding 普及——MTP (Multi-Token Prediction)、Medusa、EAGLE-2/3，主推理速度提升 1.5-3x，成 vLLM/SGLang 标配。
  4. Long Context 推理优化——StreamingLLM、Ring Attention、KV cache offload to CPU/SSD、prefix sharing —— 100k-1M context window 的工程化。
  5. MoE Inference 工程——expert parallelism 怎么做、capacity factor 怎么调、稀疏激活怎么避免 GPU idle。Mixtral / DeepSeek / Qwen 的 MoE 都需要专门优化。

Chapter 1.5 · LLM 基础速查

30 道必会 LLM 内部题 · attention / KV cache / GQA / spec decoding / 采样 / tokenizer

很多人面 senior agent eng 卡在 "讲一下 attention 内部计算" 这种基础题——本章把 6 大块凑成一份 30 题速查，配 4 个考前必看的 callout

§1.5.1Attention 内部 · 5 题

面试官最爱第一刀就切 self-attention 公式，不能背诵 = 直接被打到 mid。以下 5 题答案要 30 秒内说出来：

  1. Self-attention 公式：Attention(Q,K,V) = softmax(QKᵀ / √d_k) · V。√d_k 是为防 softmax 输入方差随 d_k 增长 → 梯度消失。
  2. multi-head 怎么拆：把 d_model 切 h 份独立做 attention 再 concat。每 head 学习不同关注模式（一些专 syntactic、一些 semantic）。
  3. O(N²) 来自哪：QKᵀ 是 N×N 矩阵 · seq len 翻倍计算量 4×。这是所有长 ctx 优化（FlashAttention / Ring Attention）的动机。
  4. causal mask 在哪一步：在 QKᵀ / √d_k 之后、softmax 之前 · 把上三角设 -inf。
  5. RoPE / ALiBi 解决什么：原 transformer 的位置编码是绝对的 · 长度推广差。RoPE 把位置编进 Q/K 旋转、ALiBi 加距离 bias，都让模型 train 短 infer 长。

§1.5.2KV cache · 6 题

  1. 为什么需要 KV cache：自回归 decode 每生成 1 token 重算所有历史 K/V 是 O(N²·N) = O(N³)。Cache 后 incremental 算 = O(N) per token。
  2. KV cache 大小公式：2 · L · n_heads · d_head · seq · bytes。Llama-3-8B (L=32, h=32, d=128, bf16) seq=8K → ~4 GB · 这就是"为什么 8B 在 24GB 卡跑得动 train 但 inference 长 ctx 也吃紧"的原因。
  3. PagedAttention 解决什么：传统 KV cache 必须 contiguous · 不同 seq 长 padding 浪费 60–80%。Paged 把 KV 切 block · 像 OS 虚拟内存 · 利用率 95%+。
  4. prefix cache：多个 request 共享同 system prompt → KV 只算一次。SGLang 的 RadixAttention / Anthropic prompt caching / OpenAI cached_input 都是这思路。
  5. cache 失效条件：任何 prefix 字节级改变 → 整个 cache miss。所以system prompt 不要插入时间戳 / 随机 ID。
  6. prefill vs decode 的 KV cache 区别：prefill 阶段一次性算所有 prompt 的 KV（compute-bound · 高 FLOPs 利用率）· decode 阶段每步只算 1 个新 token（memory-bound · 主要在读 KV）。所以两阶段拆开部署更优 (disaggregation)。

§1.5.3MHA / MQA / GQA · 4 题

  1. MHA → MQA → GQA 演进：MHA 每 head 各自 K/V → KV cache 大；MQA 所有 head 共享 1 套 K/V → KV 减 h 倍但精度掉；GQA (Llama-2 之后) 折中，每 g 个 head 共享一套。Llama-3 70B g=8 · KV 减 8 倍。
  2. GQA 对推理 throughput 影响：KV cache 减小 → 同显存能跑更长 ctx + 更大 batch · 推理 throughput +30–60%。
  3. 训练时 MQA/GQA 跟 MHA 哪个收敛快：MHA 信号最丰富收敛最快 · MQA 牺牲精度 · GQA 几乎打平 MHA。所以工业上 base model 都用 GQA。
  4. MLA (DeepSeek-V2/V3)：进一步把 K/V latent 投影成更小维度 · KV cache 比 GQA 再小 4–8 倍。代价是多一次 matmul，但 KV 节省值。

§1.5.4Speculative Decoding · 5 题

  1. 核心思想：用一个小 draft model 一次猜 K 个 token · 大 target model 一次性 verify · 接受率 ≈ 60–80% → 平均每步出 2–3 token。
  2. 为什么能加速：decode 是 memory-bound · 一次跑 1 token 还是 5 token 时间几乎一样（GPU 瓶颈在读 weights 不是算）。
  3. draft model 选什么：同 family 的小版（Llama 70B + Llama 1B）· 或 Medusa（在 target 上加 K 个 head 自己当 draft）· 或 EAGLE (训一个 feature-level draft)。
  4. 典型加速比：1.5–3×（取决于接受率 + draft cost）。代码 / 数学这种确定性强的接受率高 · 创意写作低。
  5. 缺点：① 显存多占（多一个 draft model）· ② 实现复杂 · ③ 跟 sampling temp > 0 协调有 tricky 数学（OpenAI rejection sampling 论文）。

§1.5.5采样 · 5 题

  1. greedy / temperature / top-k / top-p / min-p 各做什么：greedy 取 argmax（确定）；temp 缩放 logits（<1 更保守 · >1 更狂野）；top-k 只在 top-k 里采；top-p / nucleus 只在累计概率 <p 的子集里采；min-p 按"最大概率乘以 p"动态阈值。
  2. top-k 跟 top-p 区别：top-k 固定个数 · 不管分布陡峭；top-p 按概率质量 · 自适应。2024 起 min-p 在长输出场景被证更稳。
  3. repetition penalty / no_repeat_ngram：防模型陷入"我我我"循环。代价是创造性下降。
  4. logit bias：直接调某 token 的 logit · 强制 / 禁止特定词。OpenAI / vLLM 都有这 API。
  5. structured decoding：用 grammar / regex / JSON schema 在 decode 时屏蔽不合法 token。Outlines / xgrammar / JSON mode 都是这思路。tool calling 高准确率离不开它。

§1.5.6Tokenizer · 5 题

  1. BPE / WordPiece / SentencePiece 区别：BPE (GPT/Llama) 基于字节频率合并；WordPiece (BERT) 类似但用 likelihood；SentencePiece 把 raw text 当字节流不预 tokenize · 适合多语言。
  2. vocab size 对 model 影响：Llama-2 32K · Llama-3 128K · Qwen-2 152K。vocab 大 → 单 token 携带更多字 (中文从 ~3 char/token 到 ~1.3) · 同 ctx 装更多内容 · 但 embedding 矩阵也大 (vocab × d_model)。
  3. tokenization 引发的"陷阱"：① "strawberry" 数 r 数错 (token 不切到字母粒度)；② 数字 12345 可能拆成 [123, 45] 或 [1, 234, 5] 视 tokenizer 而定；③ 中文 / 代码 token 效率差异巨大。
  4. special tokens：BOS / EOS / SEP / PAD · 训练时 loss 通常 mask BOS 不算。Hermes 引入的 <tool_call> 是一个 special prompt pattern · 不一定是 single token。
  5. chat template：把 messages list 渲染成 plain text 的模板（如 ChatML / Llama-3 Instruct format）· 训练时格式必须跟推理一致 · 否则掉点严重。
考前必看 · 4 个 callout
  • 1. KV cache 大小公式 · 见过 Llama-3-8B 的具体数字 → 任何"为什么 ctx 不能再长"问题都能秒答。
  • 2. GQA group size · Llama-3 70B 是 8 · 这点工业知道的人不多 · 答出来分高。
  • 3. spec decoding 接受率 60–80% / 加速 1.5–3× · 这两个具体数字别给区间过宽。
  • 4. structured decoding 是 tool calling 高准确率的系统级支撑 · 不是 prompt 技巧 · 这个观点 senior 才答得出。

§2.2GPU 内部架构 · SM / Tensor Core / HBM 三件套

无论哪代卡，结构都是"很多 SM + Tensor Core 加速 + 三层 memory hierarchy"：

面试经典题"什么是 memory-bound 什么是 compute-bound"——本质就是上图。 Decode 阶段每个 token 只算一遍 → 大量 HBM 读 KV cache → memory-bound； Prefill 阶段一次算很多 token → 计算密度高 → compute-bound。这就是为什么要 disaggregate（§3.6）。

§2.3NVLink / NVSwitch · 单机多卡互联

1 台 8 卡机内部 GPU 怎么互联？答案：NVLink + NVSwitch。每张 H100 有 18 条 NVLink4 链路（25GB/s × 18 = 900GB/s）， 接到 4 个 NVSwitch 上，形成full-mesh——任意两张 GPU 之间都有 900GB/s 直连。

§2.4InfiniBand / RoCE / RDMA / GPUDirect · 跨机互联术语扫盲

跨机训练就要走网络，几个术语必须分清：

  • InfiniBand (IB)——专用网络协议（不是以太网），低延迟、高带宽。代表厂商 NVIDIA Mellanox。
  • RoCE (RDMA over Converged Ethernet)——把 RDMA 跑在以太网上，省钱替代品。RoCEv2 是事实标准。
  • RDMA (Remote Direct Memory Access)——一种能力，CPU 不参与的内存到内存传输。IB 和 RoCE 都支持。
  • GPUDirect RDMA——更进一步：跨机 GPU 显存可直接互传（不经过 CPU 也不经过 host RAM）。LLM 训练必备。
  • NCCL——NVIDIA 集合通信库（all-reduce / all-gather / broadcast）。所有分布式训练框架（PyTorch DDP / FSDP / DeepSpeed / Megatron）底层都用它。

§2.5集群拓扑 · Fat-tree vs Rail-optimized

大规模 GPU 集群（> 100 节点）的网络拓扑有两种主流设计：

  • Fat-tree (Clos)——传统 HPC 设计，所有节点等价对称，bisection bandwidth 高。Meta 的 Research SuperCluster 用这个。
  • Rail-optimized——把同 rank 的 GPU（如所有节点的 GPU0）放在同一个 rail 上，跨节点 all-reduce 只走自己的 rail。NVIDIA / Microsoft / xAI 的 H100/B200 集群基本都用这个。

Rail-optimized 的优势：大幅降低 all-reduce 的"碰撞"，对 TP+DP 训练特别友好。 面试如果你能说出"我们集群是 rail-optimized 拓扑，单 pod 1024 张 H100，pod 间走 IB NDR"——立刻显示是真做过大集群的。

§2.6单卡显存装得下吗？ · 一个必背公式

推理显存（1 batch）

params × bytes/param + KV_cache

其中 KV_cache = 2 × num_layers × num_heads × head_dim × seq_len × batch × bytes

例：Llama 3 70B FP16 推理，seq=4k, batch=1：

  • 权重：70B × 2 = 140 GB（显然 1 张 H100 80GB 装不下，需要 2-4 张 TP）
  • KV cache：2 × 80 × 64 × 128 × 4096 × 1 × 2 = ~10.7 GB（注意 GQA 把 KV head 数减到 8 了，否则更大）

这就是为什么 70B+ 模型必须tensor parallel 切到多卡。能现场算这个公式，infra 面试基本过半了。

训练显存（粗略）

params × (2 weights + 2 grads + 8 optimizer state for Adam) + activations

BF16 训练 70B：70B × 12 = 840 GB 仅 model state。Adam 8 字节是 fp32 momentum/variance + master weights。

这就是 ZeRO 的来源——Adam state 是大头，把它 shard 掉是 ZeRO-1 的核心。

Chapter 3 · 推理引擎

vLLM / SGLang / TGI / TRT-LLM 内部

面试 inference infra 80% 的题在这一章——每个核心技术单独讲透

§3.1LLM 推理两阶段 · Prefill vs Decode

每次 LLM 推理本质分两步：prefill（一次处理全部 prompt）+ decode（一个个吐 token）。 这两步的计算特征截然相反，搞清楚这点是后面所有优化的前提。

阶段做什么计算特征瓶颈优化方向
Prefill处理 prompt 全部 N 个 token大矩阵乘 · 计算密集 · arithmetic intensity 高compute-bound更高 FLOPs · FlashAttention · 算子融合
Decode生成第 N+1, N+2, … 个 token每步只算 1 个 token · 大量读 KV cache · 矩阵×向量memory-bound (HBM 带宽)大 batch · KV cache 优化 · speculative decoding
关键指标 TTFT vs TPOT
  • TTFT (Time To First Token)——首 token 延迟，由 prefill 决定。用户感知"反应快不快"
  • TPOT (Time Per Output Token)——后续每 token 间隔，由 decode 决定。用户感知"打字流不流畅"
  • ITL (Inter-Token Latency)——同 TPOT
  • Goodput——实际有用 throughput（满足 SLO 的 token/s），跟 throughput 不同

§3.2PagedAttention · vLLM 的核心创新

2023 年 vLLM 论文提出，把 KV cache 借鉴 OS 虚拟内存的思路分页管理。

问题：传统实现给每个 request 预分配 max_seq_len 的 KV cache 显存——大部分浪费（80%+ 显存碎片化或预留不用）。

方案：把 KV cache 切成固定大小 block（默认 16 token/block），用 block table 映射逻辑 token 到物理 block，类似 OS page table。

面试题 · 必答

Q: PagedAttention 解决什么问题？跟传统 KV cache 区别？

A: 解决 KV cache 显存碎片化问题。传统实现按 max_seq_len 预分配（浪费 60-80% 显存）， PagedAttention 借鉴 OS 虚拟内存把 KV cache 切成 16-token 固定大小 block，用 block table 映射。 显存利用率 20% → 96%+，throughput 提升 14-24x（vLLM 论文）。 副作用：每次 attention 多一次 indirect 查找，开销可忽略。 额外收益：block 自然支持 prefix sharing（多 request 同 system prompt 共享 block）、parallel sampling、beam search。

§3.3Continuous Batching · static batching 的革命

传统 static batching：组一个 batch (e.g., 8 个 request)，所有 request 算到全部完成才能开下一 batch。 问题：长度不等 → 短的等长的 → GPU 大量 idle。

Continuous batching（也叫 iteration-level scheduling）：每个 step 重新组 batch。 谁结束了立刻走人，新来的 request 立刻插队。GPU 几乎不 idle。

Continuous batching 来自 2022 Orca 论文，2023 年被 vLLM / TGI / TensorRT-LLM 全都实现。2026 年是行业标配，没有的就不算 production 推理引擎。

§3.4Speculative Decoding · 让 decode 跑得快

Decode 是 memory-bound（每 token 都要读全部 KV cache + weights），核心 bottleneck 是 HBM 带宽。怎么破？ 让 GPU 在等带宽的同时多算几个候选 token——用一个小模型猜 N 个 token，大模型一次性 verify。如果命中就赚 N-1 步，错了也只浪费这一步。

方法怎么猜命中率速度提升
Vanilla Speculative用小模型（如 Llama-3 8B 给 70B 当 drafter）40-60%1.5-2x
Medusa主模型加多个 prediction head 直接预测后续 token50-65%1.8-2.5x
EAGLE / EAGLE-2 / EAGLE-3更精巧的 drafter（用 hidden state 预测）60-80%2.5-3.5x
MTP (Multi-Token Prediction)训练时直接学多 token 预测（DeepSeek-V3 用）~85% acceptance1.8x

2026 年 vLLM 默认支持 EAGLE / Medusa；SGLang 实现得最早最全。如果你做 inference infra 不知道这些，面试基本挂。

§3.5Prefix Sharing / RadixAttention · SGLang 的杀招

多个 request 共享 system prompt（agent / chat 场景常见）——能不能让它们共享 KV cache？答案：能。

SGLang 的 RadixAttention：用 radix tree 索引所有历史请求的 prefix。 新 request 来了先查 tree，命中就直接复用 KV cache（连 prefill 都省了）。多轮对话场景能省 50-90% prefill 计算。

vLLM 也有类似的 Automatic Prefix Caching (APC)，2024 后默认开启。

§3.6Disaggregated Prefill / Decode · 2024-2026 最大架构变革

洞察：prefill 是 compute-bound，decode 是 memory-bound——为什么把它俩跑在同一组 GPU上？

方案：把 prefill 和 decode 分到不同 GPU pool，KV cache 通过 NVLink/RDMA 从 prefill GPU 传到 decode GPU。 代表系统：

  • Mooncake（月之暗面 2024）——Kimi 在用，全球第一个生产规模 disaggregated 推理系统
  • DistServe（北大 2024）——学术界开源参考实现
  • Splitwise（Microsoft 2024）——Azure 部署
  • vLLM 1.0+ 实验性支持

量化与压缩 · FP8 / GPTQ / AWQ / GGUF

把 FP16 模型变 INT4 不掉点的工程艺术——面试必考的 4 个算法

§4.1为什么要量化？

3 个目的，按重要性排序：

  1. 省显存——70B FP16 = 140GB，INT4 = 35GB，能塞进 1 张 A100/H100
  2. 提速——decode 阶段是 memory-bound，读权重的字节数 ↓ → throughput ↑
  3. 降本——同 GPU 装更多模型 / batch，单位成本 ↓

§4.34 个必会的量化算法

① GPTQ · 逐列量化 + 误差补偿（2022）

把 weight 矩阵每列做量化时，用剩余未量化的列吸收误差。 基于 OBS (Optimal Brain Surgeon) 思想。算法 1-pass，几小时能量化 70B。4-bit 推理 SOTA 之一。

② AWQ · 激活感知量化（2023）

关键洞察：不是所有 weight 都同等重要——对应大 activation 的那些 weight 量化误差影响巨大。 AWQ 用 calibration data 找出"重要 weight"，给它们更高精度（per-channel scaling）。 跟 GPTQ 同精度，速度更快、不需要 backprop。当前 vLLM/SGLang 默认推荐。

③ SmoothQuant · 把 outlier 从 activation 挪到 weight（2022）

LLM 量化最大的坑：activation 有 outlier（个别维度值很大），导致 INT8 量化掉点严重。 SmoothQuant 引入 scale 把 outlier 从 activation 转到 weight 上（weight 容易处理 outlier）， 使 W8A8 (weight INT8 + activation INT8) 可行。

④ GGUF · 边缘部署的 quantization 容器

不是算法是文件格式——llama.cpp 出品，把模型 + 量化 + tokenizer 打包成一个文件， 支持 Q2_K / Q4_K_M / Q5_K_M / Q8_0 等多种 mixed precision。 主流量化算法都能 export 成 GGUF。 个人本地跑 LLM 几乎都用 GGUF（Ollama 底层就是 llama.cpp）。

§4.5FP8 训练 · DeepSeek-V3 印证可行

2024-12 DeepSeek-V3 技术报告把 FP8 训练带火——671B MoE 全程 FP8 训练，相比 BF16 显存省一半、速度快 2x，最终质量超过 GPT-4-class。

关键技术：tile-wise scaling（不是 per-tensor 一个 scale，而是按 tile 给不同 scale）+ FP32 master weight + FP8 forward/backward + 多种 fp8 variant 混用（E4M3 forward / E5M2 grad）。

面试题 · 必答

Q: AWQ 和 GPTQ 哪个更好？

A: 精度差不多（INT4 都能保持 95%+ 任务表现），但AWQ 在工程上更优：① 不需要 backprop 因此更快、内存更省；② 对 outlier 处理更系统（per-channel scaling）；③ 推理时实现的算子（如 vLLM W4A16 kernel）针对 AWQ 优化得更早。 生产 LLM 推理 2024 年起 AWQ 已经成默认推荐，GPTQ 主要在 HuggingFace 老 checkpoint 兼容性上还有优势。 如果追求极低位宽（INT2/INT3），现在还看 GGUF k-quant 系列。

Chapter 5 · 分布式训练

§5.1为什么单卡不够？回到显存账

训练 70B 模型 BF16 + Adam 显存账：

  • 权重：70B × 2B = 140 GB
  • 梯度：70B × 2B = 140 GB
  • Adam state（fp32 momentum + variance + master weight）：70B × 8B = 560 GB
  • Activation（大约）：100-300 GB（取决于 seq_len + batch + recompute 策略）
  • 合计：1 TB+，1 张 H100 80GB 装得下个零头

所以必须多卡分布式。怎么分有 7 种策略：

  • ZeRO-3：weight 也 shard、显存极致省但通信最多 —— 单卡装不下时才开

§5.5Tensor Parallel · Megatron 的核心

TP 把单层内部矩阵乘法切到多 GPU。最经典是 Megatron-LM 的 column/row split：

  • QKV projection 按列切 → 4 GPU 各算 1/4 个 head
  • Attention output projection 按行切 → 跨 GPU all-reduce 一次
  • FFN up_proj 按列切，down_proj 按行切 → 同样 1 次 all-reduce

每 transformer layer 通信 2 次 all-reduce activation。NVLink 内部很快但跨节点会很慢——所以 TP 一般限在同机 8 卡内部。

§5.6Pipeline Parallel · 跨节点的选择

把 transformer 80 层按 layer 切，每 GPU 拿 N 层。问题："bubble"——前向时 GPU 1 算完才轮到 GPU 2，前面 idle。

1F1B (1-Forward-1-Backward) schedule（GPipe / Megatron）：交替前向反向跑，把 bubble 压到 ~10%。 Interleaved schedule：每 GPU 不只拿连续 N 层，而是分散拿几段，进一步缩小 bubble。

§5.73D Parallelism · 实战组合

训 70B+ 模型必须组合：TP × PP × DP。典型配置：

  • TP=8：单机内 NVLink，切层内
  • PP=8：跨节点 IB，切层间
  • DP=N/64：剩下做数据并行（N 是总 GPU 数）

训 GPT-3 175B 在 1024 GPU 上，配置就是 TP=8, PP=16, DP=8。能现场算这个分解面试就过了。

§5.8Sequence Parallel + Ring Attention · 长 context 救星

SP：Megatron 的扩展，把 LayerNorm / Dropout 这些 elementwise 算子按 seq 维度切 → 进一步降 activation 显存。

Ring Attention（UC Berkeley 2023）：把 attention 计算按 seq 切到多个 GPU，KV 在 ring 上传，能算1M+ context。 Gemini 1.5 Pro 1M context 用的就是这个。

§5.9Activation Recomputation · 显存换计算

反向传播需要 forward 的 activation——不存而是反向时重算一遍就能省大量显存。 Selective recompute（Megatron）：只重算 attention（最贵的部分），其他存。 典型省 50% activation 显存代价 25-30% 多算时间。70B+ 训练几乎必开。

面试题 · 高频

Q: Tensor Parallel 跟 Pipeline Parallel 各适合什么场景？

A: TP 通信频繁（每层 2 次 all-reduce activation），必须放在 NVLink 内（同机 8 卡），跨 IB 就慢死。 PP 通信少（只在层边界传 activation），能跨节点，但有 bubble。 实战组合：TP=8（机内）× PP=N（跨节点）× DP=剩余。 选择规则：单层放不下用 TP，层数多放不下用 PP，剩下用 DP/ZeRO。

L2 算子层 · 写过 kernel 的工程师在面试天然加分

§6.1FlashAttention · 一篇论文改变 LLM

2022 年 Tri Dao 的 FlashAttention 论文是近 5 年 deep learning kernel 最重要的工作。 解决一个看似无解的问题：attention 计算 O(N²) 显存怎么破。

问题：标准 attention 实现要把 N×N 的 attention matrix 写到 HBM——seq_len=4096 时单层就 64MB，且大部分时间耗在 HBM 读写。

核心洞察：attention 是 memory-bound 而不是 compute-bound——瓶颈在 HBM 带宽不在 FLOPs。 解法：把 Q/K/V 切 tile（block），用 online softmax 算法在 SRAM 内一次算完一个 block 的 softmax+matmul，never materialize 完整 N×N 矩阵。

版本年份关键改进对比基线提升

§6.3Triton · 写 GPU kernel 的 Python DSL

OpenAI 出品，用类 Python 语法写 CUDA kernel。比 raw CUDA 简单 10 倍，性能能打到手写 CUDA 的 80-95%。 FlashAttention 2/3 都有 Triton 实现，vLLM 大量自定义算子用 Triton。 面试加分：你写过哪个 Triton kernel——能现场画 grid / block 切分的就是真懂。

# 一个最简单的 Triton kernel · 向量加
import triton
import triton.language as tl

@triton.jit
def add_kernel(x_ptr, y_ptr, output_ptr, n, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offsets < n
    x = tl.load(x_ptr + offsets, mask=mask)
    y = tl.load(y_ptr + offsets, mask=mask)
    tl.store(output_ptr + offsets, x + y, mask=mask)

§6.4CUDA Graph · 减少 launch overhead

每个 CUDA kernel launch 有 ~5-10μs CPU 开销。decode 时一步只算几个小算子—— launch overhead 占比能到 30-50%！

CUDA Graph：把整个 forward pass 录制成一个 graph，一次 launch 全部跑完。 vLLM / TRT-LLM 推理 decode 阶段几乎都用 CUDA Graph。典型加速 30-80%。

限制：graph 录制时形状必须固定（dynamic shape 不友好）→ 需要预录多个 batch_size 的 graph。

典型 LayerNorm + Linear + GELU 三层连续算子，朴素实现要 3 次 HBM 读写（每个算子读输入写输出）。 融合成一个 kernel 后只 1 次读 1 次写，省 67% HBM I/O —— memory-bound 场景直接 3x 加速。 torch.compile / TorchInductor / Triton 自动做这件事。

§7.3MoE Infra 4 大挑战

  1. 负载不均——某些 expert 被频繁路由 → GPU 满 / 其他 GPU 闲。需要 load balancing loss（auxiliary loss 让 router 均衡）+ capacity factor（每 expert 最多收 C 个 token，超出丢弃 / 溢出）
  2. All-to-All 通信——expert 在不同 GPU，token 要 all-to-all 路由过去再路由回来。这是 MoE 训/推的主要瓶颈。
  3. 显存大但激活小——推理要装全部 expert（DeepSeek 671B → 1.3TB FP16），需要多卡 expert parallel
  4. 稀疏激活批不到一起——decode 时每 token 走的 expert 不同，无法像 dense 那样大 batch 算

§7.4Expert Parallelism · MoE 必备

把 N 个 expert 切到 N 张 GPU 上，每张 GPU 只装 1 个（或几个）expert。 forward 时：token 经过 router 选中 top-k expert → all-to-all 把 token 发到目标 GPU → expert 计算 → all-to-all 把结果发回。 跟 TP 组合：每个 expert 内部还可以 tensor parallel。 跟 DP 组合：DP 维度复制 expert（如 EP=8 + DP=4，就 32 GPU 上每个 expert 4 副本）。

§7.52026 MoE 推理工程关键词

  • Expert offloading——把不常用的 expert 放 CPU RAM，按需载入 GPU
  • Speculative routing——猜测下一 token 会路由到哪些 expert，提前预热
  • Batch within expert——同 expert 上的不同 token 一起算（grouped GEMM）
  • Capacity-aware scheduling——调度器感知每 expert 当前队列，避免热点

Chapter 9 · 向量库

HNSW / IVF / DiskANN / PQ · 向量库内部

RAG 时代的"OLAP 数据库"——Vector DB 子方向必学

§9.1问题定义 · ANN 是什么

给一个 query 向量，从 N 个向量库中找出最相似的 K 个（典型 N=100M-10B，K=10-100）。 暴力算 cosine 相似度 O(N) 太慢——必须近似（ANN, Approximate Nearest Neighbor）。 工业界 4 大算法：HNSW / IVF / DiskANN / ScaNN，每个都有自己的 trade-off。

§9.2HNSW · 当前最流行

Hierarchical Navigable Small World——多层图，每层稀疏度递减。 搜索从顶层开始 greedy walk → 跳到下层细化 → 直到底层。O(log N) 的实际复杂度。

关键参数：

  • M——每个节点最多连边数（典型 16-64）
  • ef_construction——建图时探索的候选数（影响建图质量，典型 200-400）
  • ef_search——查询时探索的候选数（影响 recall vs latency，典型 50-300）

优势：高 recall + 快 query · 劣势：建图慢 + 全 in-memory · 删除复杂

§9.3IVF · 倒排文件

Inverted File Index——先用 K-means 把所有向量聚 N 个 cluster（centroid）， query 来了先找最近的 nprobe 个 cluster，再在这些 cluster 内 brute force。 Faiss 默认实现 IVF + PQ 是经典组合。

§9.4PQ · 把向量压小

Product Quantization——把 768 维向量切成 8 段（每段 96 维），每段独立用 K-means 量化成 8 bit code。 768 × 4B = 3KB → 8B，压缩 384 倍。距离计算用查表（precomputed centroid distances）非常快。

变种：OPQ（旋转后再 PQ，提高质量）、SQ (Scalar Quantization, 简单但效果差)。

§9.5DiskANN · 大数据量首选

Microsoft 2019。当向量量大到 装不进 RAM（> 1B 向量），HNSW 就废了——DiskANN 把图主体放SSD， 用Vamana 算法构图（比 HNSW 更紧凑），加 PQ 把 in-memory 部分压缩到能放 RAM。 10B 向量单机搜索 P99 < 50ms。Microsoft Bing / OpenAI / 字节都用类似方案。

§10.1训练核心指标 · MFU / HFU

MFU (Model FLOPs Utilization)

MFU = 实测 throughput × 每 token 的 model FLOPs / 硬件峰值 FLOPs

例：H100 BF16 峰值 989 TFLOPS。Llama-3 8B 训练实测 200 TFLOPS/GPU → MFU = 200/989 ≈ 20.2%

典型 MFU 范围：

  • 30-50%：优秀（GPT-3 论文 21.3% · Llama-3 论文 38-43%）
  • 15-30%：合格
  • < 10%：有大问题（通信瓶颈 / 算子没融合 / batch 太小）

HFU (Hardware FLOPs Utilization)：把 activation recompute 多算的也计入分子，比 MFU 高一些。 Meta 报论文都用 MFU，因为它反映"真实有效产出"。

§10.5容量规划 · 简单公式

QPS = (8 × 60) / 2000 = 0.24 → 平均 4.2s/req · 高峰要排队

如果加 spec decoding 2x + AWQ INT4 + disaggregated 2x → 实际 throughput 4-8x → QPS 可达 1+

以下题目按 8 个模块分组——前 4 个是 L1-L3 硬核技术（必会），后 4 个是 L4-L5 系统/方法论（加分）。 每道题给关键答题点而不是完整答案，需要你结合自己经历用 STAR (Situation / Task / Action / Result) 包装。

§AGPU / 网络 / 硬件（10 题）

A1 · 讲讲 GPU 的 memory hierarchy

关键点：register / SMEM / L2 / HBM 4 层 · 容量越下越大、带宽越下越慢、cycle 越下越多。 H100 数字：SMEM ~256KB/SM · L2 50MB · HBM 80GB/3.35TB/s。 能讲出"FlashAttention 就是要让数据尽量留在 SMEM 不去 HBM"就过。

A2 · NVLink 跟 PCIe 跟 InfiniBand 区别

关键点：NVLink = 同机 GPU 互联，900GB/s（H100），是 PCIe 5（120GB/s）的 ~7x； InfiniBand = 跨机网络，NDR 400Gb=50GB/s，比 NVLink 慢 18x； 所以 TP 必须放 NVLink 内、DP/PP 才能跨节点。

A3 · GPUDirect 是什么 · 为什么必要

跨机 GPU 显存可直接互传不经 CPU 也不经 host RAM。 没有它的话 NCCL 通信要走 GPU→host RAM→网卡→对端 host RAM→GPU，多 2 次拷贝带宽减半。

A4 · H100 vs B200 主要差别

FLOPs ~2x · HBM 80→192GB · HBM 带宽 3.35→8 TB/s · NVLink5 双倍 · FP8 性能 2x · FP4 原生支持。 重点在显存带宽翻倍——decode memory-bound 阶段直接受益。

A5 · 推理 70B 模型显存怎么算？

权重 70B × 2B = 140GB · KV cache = 2 × layer × heads × head_dim × seq × batch × bytes。 1 张 80GB H100 装不下 → TP=2 切。能现场算这个就过了。

A6 · Memory-bound vs Compute-bound 怎么判断？

看 arithmetic intensity（FLOPs / bytes accessed）。 高于硬件 ridge point（H100 BF16 ≈ 200）就 compute-bound，否则 memory-bound。 典型：matrix-matrix multiply (高) compute-bound · matrix-vector (低) memory-bound · LLM decode 是后者。

A7 · NCCL 4 个核心原语

all-reduce / all-gather / reduce-scatter / broadcast。 能讲清"all-reduce = reduce-scatter + all-gather"以及"ring vs tree" topology 区别加分。

A8 · MIG 是什么 · 什么场景用

Multi-Instance GPU。H100 可硬切成 7 个独立 instance（计算/显存/带宽都隔离）， 适合多租户、小模型推理共享 GPU。不适合大训练（不能跨 instance）。

A9 · 训练 Llama-70B 单 step 的瓶颈通常是什么

三个候选：① attention（FlashAttention 解）② all-reduce gradient（ZeRO/拓扑解）③ activation 显存（recompute 解）。 典型大集群训练，跨节点通信就是大头，所以 rail-optimized 拓扑那么重要。

A10 · Rail-optimized 拓扑是什么

把同 rank 的 GPU（每节点 GPU0、每节点 GPU1...）放同一 rail（同一组 IB switch）， all-reduce 几乎不撞包。NVIDIA SuperPOD / xAI Colossus / 字节大集群都用。

§BLLM 推理引擎（15 题）

B1 · PagedAttention 解决什么问题

见 §3.2 完整答案。必须 30 秒讲完。

B2 · Continuous batching 跟 static batching 区别

见 §3.3。关键词：iteration-level scheduling · 短请求不等长请求 · throughput 5-10x。

B3 · Prefill 和 Decode 计算特征区别

见 §3.1。Prefill = matrix-matrix = compute-bound · Decode = matrix-vector = memory-bound。

B4 · Speculative decoding 原理

用小 drafter 模型猜未来 N 个 token · 大模型一次性 verify · 命中赚 N-1 步、错了浪费 1 步。 因为 verify 阶段是 batched matrix multiply（compute-bound 不是 memory-bound）， 相当于用 GPU 闲置算力换掉 HBM 带宽。能讲到这一层就是真懂。

B5 · 为什么 spec decoding 对 decode 加速但对 prefill 没用

Prefill 已经是 compute-bound，没多少 GPU 闲置算力可换。Decode 是 memory-bound，GPU 大量空算力，spec 才划算。

B6 · KV cache 怎么算大小 · 怎么省

大小：2 × L × H × D × seq × batch × bytes（2=K和V）。 省的方法：① GQA/MQA（减 H）② 量化 KV（fp8/int4）③ KV offload to CPU/SSD（StreamingLLM）④ prefix sharing。

B7 · GQA / MQA 区别

MHA: Q heads = KV heads。MQA: 所有 Q 共享 1 组 KV（KV cache 缩 H 倍但质量明显掉）。 GQA: Q heads 分成 G 组，每组共享 KV（折中，质量几乎不掉）。Llama 系列从 Llama-2 70B 开始用 GQA(8)。

B8 · vLLM / SGLang / TRT-LLM 主要区别

见 §3.7 表格。

B9 · Disaggregated serving 是什么 · 什么时候值得

把 prefill / decode 分到不同 GPU pool。值得的场景：长 prompt + 短 response（典型 RAG）。 代价：架构复杂 + KV cache 跨 pool 传（要 RDMA）。

B10 · Prefix caching / RadixAttention 怎么工作

用 radix tree 索引历史 KV cache 的 prefix。新 request 进来先 longest-prefix-match，命中段直接复用。 多轮对话 / agent / 共 system prompt 场景特别有效。

B11 · Chunked prefill 是什么

把超长 prompt 的 prefill 切成 chunk，跟 decode token 一起 batch。 解决"长 prefill 阻塞短 decode"的 head-of-line blocking。 vLLM / SGLang 默认开。

B12 · TTFT 突然变高怎么排查

① queue depth 上升（请求堆积）② prefix cache miss 率突变 ③ KV cache 占满触发 evict ④ 某个 request 超长 prompt 占满 prefill window。 看 vLLM/SGLang metric 即可定位。

B13 · Long context (1M) 推理怎么做

① attention：用 FlashAttention 或 Ring Attention ② KV cache：太大要 offload (CPU/SSD) ③ position：YaRN / NTK-aware RoPE ④ 工程：streaming output 让用户感知不到等待。

B14 · 推理时如何同时支持 OpenAI / Anthropic / 自部署后端

用 LiteLLM / OpenRouter 这种 router，统一 OpenAI 兼容协议，按需路由。 SGLang / vLLM serve 也都暴露 OpenAI 兼容 API。

B15 · 评估推理引擎选型 · 你看哪些指标

① throughput (tok/s) under SLO · ② TTFT/TPOT P50/P95 · ③ 模型支持广度 · ④ 是否支持 spec / quant / disaggregated · ⑤ 社区活跃度（PR 频率、release cadence） · ⑥ ops 友好度（metrics、log、debug）。

§C分布式训练（15 题）

C1 · ZeRO 1/2/3 区别

见 §5.3。

C2 · FSDP 跟 ZeRO-3 是同一回事吗

本质相同。FSDP 是 PyTorch 原生实现，跟 TP / DTensor 集成更好。新项目首选 FSDP2。

C3 · TP / PP / DP 各自什么时候用

见 §5.2 表格 + §5.7 实战组合。能现场为 70B / 175B / 671B 设计 3D 并行配置就过了。

C4 · TP 为什么不能跨节点

每层 2 次 all-reduce activation · 跨节点 IB 比 NVLink 慢 18x · 整个训练就被通信拖住。

C5 · 1F1B vs interleaved pipeline schedule

1F1B：交替 forward/backward 跑，bubble ~10%。Interleaved：每 GPU 拿不连续的几段层，bubble 进一步缩小但通信增加。

C6 · Activation recompute 是什么 · 怎么权衡

反向时不存 activation 而是重算。典型省 50% 显存代价 25-30% 多算。Selective recompute 只重算最贵的（attention）。

C7 · Mixed precision 训练 · 为什么 master weight 用 fp32

BF16 forward / backward 节省带宽和显存。但 Adam state 累加要 fp32 精度（防止小梯度累加为 0）。这就是 master weight。

C8 · FP8 训练有哪些坑

① 动态范围窄 → 需要 per-tensor / per-tile scaling ② 梯度用 E5M2 forward 用 E4M3 ③ Loss scale 调整 ④ 某些 op (LayerNorm) 仍用 BF16/FP32。 DeepSeek-V3 报告是最详细 reference。

C9 · NCCL all-reduce 内部怎么实现

大消息：ring all-reduce（reduce-scatter + all-gather，O(2(N-1)/N × M) 通信）。 小消息：tree all-reduce（O(log N) 延迟）。NCCL 自动按消息大小选。

C10 · Gradient accumulation 是什么 · 怎么影响 throughput

显存装不下大 batch 时，分 N 个 micro-batch 跑，梯度累加后再 step。 副作用：通信量减少（all-reduce 频率 ÷ N），所以大 grad accum 反而提升 throughput（在通信瓶颈场景）。

C11 · 训练 OOM 怎么排查

顺序：① 减 batch ② 开 activation recompute ③ 开 ZeRO 2/3 ④ 开 FlashAttention ⑤ 切 TP ⑥ offload optimizer to CPU。

C12 · Loss spike 怎么处理

① 检查 grad norm（一般 spike 时 grad 爆炸）② 降 LR ③ 用 grad clip ④ skip 这个 batch（rewind to checkpoint）⑤ 永久解法：调 mixed precision / 加 z-loss。

C13 · 大集群 train 故障率 · 怎么 fault tolerance

见 §11.1-11.2。

C14 · Checkpoint 写得太慢怎么办

① 异步 checkpoint（PyTorch DCP）② sharded checkpoint（每 GPU 只写自己那份）③ 写到本地 SSD 再异步 sync 到对象存储 ④ 增量 checkpoint。

C15 · 怎么估算 70B 训练总成本

FLOPs ≈ 6 × params × tokens（Chinchilla 系数）。70B × 15T tokens = 6.3e24 FLOPs。 H100 BF16 989 TFLOPS × MFU 40% = 4e14 useful FLOPs/s/GPU。 GPU-hours = 6.3e24 / 4e14 / 3600 ≈ 4.4M GPU-hours。$2/h × 4.4M = $8.8M。

§D算子 / 量化 / MoE（15 题）

D1 · FlashAttention 解决什么问题 · 怎么解

见 §6.1。关键词：HBM I/O 从 O(N²) 降到 O(N) · tiling + online softmax · never materialize N×N matrix。

D2 · FlashAttn 1/2/3 演进

见 §6.2。FA2: 更好 work partition + causal mask 优化。FA3: H100 async + WGMMA + FP8。

D3 · 为什么 attention 是 memory-bound

arithmetic intensity 低 · 大量 HBM 读 KV · ridge point 之下。这是大部分场景，但 prefill 长 seq 时因为 N² 关系会变 compute-bound。

D4 · Triton 跟 CUDA 区别

Triton 是 Python DSL · 自动 tile + memory coalescing · 不需手写 PTX · 写 1/10 代码达 80-95% CUDA 性能。 CUDA 给极致控制（warp 级原语）但学习曲线陡。

D5 · 算子融合为什么能省时间

见 §6.6。多算子合一 → 减少 HBM 读写次数 → memory-bound 场景直接 N-1 倍加速。

D6 · CUDA Graph 是什么 · 什么场景值得

见 §6.4。launch overhead 占比高时（decode 小算子）值得。

D7 · GPTQ vs AWQ 哪个好

见 §4.5 面试题。

D8 · INT4 量化为什么不掉点很多

① 权重分布通常对称 + 集中 ② 用 group-wise scale（每 64-128 元素一个 scale）保留分辨率 ③ 重要 weight (high-activation) 保持高精度（AWQ）。 代价主要在极端 outlier 任务（数学 / 代码长链推理）会有 1-3 分掉。

D9 · FP8 训练为什么可行 · 难点是什么

见 §4.6 + DeepSeek-V3。

D10 · MoE 是什么 · 跟 dense 的 trade-off

稀疏激活 = 总参数大但每 token FLOPs 小 = 训练快 + 推理便宜，但显存巨大 + all-to-all 通信瓶颈。

D11 · MoE 负载不均怎么处理

① auxiliary loss 鼓励 router 均衡 ② capacity factor 限流 ③ 监控每 expert 流量、调路由温度 ④ 部署时 EP 副本数 / placement 优化。

D12 · MoE 推理为什么比训练更难

训练 batch 大，每 expert 都能拿到足够 token；推理 decode 时每 token 走的 expert 不同，无法 batch 起来 → expert 频繁 idle / launch overhead 高。 解：grouped GEMM / spec routing。

D13 · DeepSeek-V3 为什么这么便宜

① MoE 671B/37B 激活 ② FP8 训练省 2x compute ③ MTP 推理加速 1.8x ④ Multi-head Latent Attention 减 KV cache ⑤ 工程极致优化（NVLink + IB + 自研 NCCL 替代 DeepEP）。

D14 · Speculative routing 是什么

见 §7.5。猜下一 token 路由到哪些 expert，提前预热（载入 weight / 准备 batch）。

D15 · 你写过最复杂的 Triton kernel 是什么

这是"判断你真假"的题。如果没写过，老老实实说"读过 vLLM 的 attention kernel + 跟过 Triton tutorial 实现 flash attention 简化版"。

§E平台 / MLOps（10 题）

E1 · K8s vs Slurm 训练

见 §8.1。

E2 · K8s 上调度多机多卡训练 · 怎么避免半 pod 等死

用 gang scheduling（Volcano / Kueue）。要么全部 pod 起、要么都不起。

E3 · Topology-aware scheduling 是什么

调度器感知 NVLink / 节点 / rail 亲和度，把一个 job 的 GPU 放在同一 NVLink island 里。

E4 · LLM 推理 autoscaling 难点

见 §8.5。冷启动 5-10min vs 用户耐心 5s。

E5 · scale-to-zero 推理可行吗

大模型几乎不行（冷启动太慢）。小模型（< 7B）+ pre-warmed image + 模型 cache disk 可以做到 < 30s。

E6 · 多租户 GPU · 怎么隔离

① MIG（硬隔离）② time-slicing（软共享）③ MPS（多进程共享）。 MIG 适合稳定多租户、time-slicing 适合开发、MPS 适合同 user 多 job。

E7 · Ray 跟 K8s 关系

Ray 是distributed Python framework，K8s 是infra orchestrator。 Ray 通常跑在 K8s 上（KubeRay operator）。Ray 提供 task/actor 抽象，K8s 提供节点/资源管理。

E8 · 模型怎么从训练交付到推理

典型流水线：训练 done → 写 model registry（W&B / MLflow）→ 转换 + 量化（trtllm-build / vllm convert）→ 打包 docker → 上推理平台（vLLM Serve / KServe）→ canary → 全量。

E9 · GPU 监控你看哪些指标

见 §10.3 DCGM。GPU util / SM occupancy / mem util / NVLink BW / power / temp / XID errors。

E10 · 你怎么设计公司内部 LLM gateway

① 统一 OpenAI 兼容 API ② 多 provider 路由（自部署 + 云）③ 配额 + cost tracking ④ prompt cache（语义级）⑤ guardrail（PII / toxicity）⑥ observability（log + trace）。 能讲出这 6 层就稳。

§F向量库 / RAG infra（8 题）

F1 · HNSW 怎么工作

见 §9.2。多层 small-world graph + greedy walk + O(log N) 检索。

F2 · IVF + PQ 组合工作流

建索引：① K-means 聚 N 个 cluster ② 每 cluster 内向量做 PQ 量化。 查询：① 找 top nprobe cluster ② 在这些 cluster 内用 PQ 加速距离计算 ③ rerank top-k 用原始向量。

F3 · DiskANN 跟 HNSW 区别 · 什么场景用

见 §9.5。> 1B 向量装不进 RAM 时用 DiskANN。否则 HNSW 更快。

F4 · 向量库怎么支持过滤（metadata filtering）

3 种思路：① pre-filter（先按 metadata 筛子集再 ANN，子集大时快）② post-filter（先 ANN 再筛，质量差） ③ integrated（边搜边过滤，HNSW 改造，Qdrant / Milvus 支持）。

F5 · 增量更新怎么处理

HNSW 删除是软删除（标记 + 跳过），周期性 rebuild。 IVF 增量：直接加到对应 cluster，定期重 cluster。 高频写场景一般用 LSM-style：MemTable + SSTable 多层。

F6 · Hybrid search · 怎么 fuse BM25 + vector

① RRF (Reciprocal Rank Fusion) — 简单且效果好，score = sum(1/(k+rank_i)) ② 加权和（normalized score 加权） ③ 学一个 reranker（cross-encoder / 模型）做最终排序。

F7 · 向量库怎么做分布式

① Sharding by hash / range ② 每 shard 独立 HNSW/IVF index ③ query fan-out + 聚合 top-k ④ replication for HA。Milvus 是这套架构的开源标准实现。

F8 · Embedding model 选型 · 你看什么

① MTEB benchmark看任务相关榜（retrieval / clustering） ② 维度（小维度便宜大维度更准，384/768/1024 是常见） ③ 多语言支持 ④ 是否支持 instruction-aware（query / doc 用不同 prompt） ⑤ 自部署 vs API 成本。
