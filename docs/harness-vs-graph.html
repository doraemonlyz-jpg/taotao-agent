这不是论坛吵架——taotao-agent 同时支持 POST /chat（graph）和 POST /chat/v2（harness）， 本手册把 5 类典型 query 在两个 endpoint 跑一遍，给出真实 trace · token 数 · cost · latency · 答案质量对比。 每条 query 都标了"该选哪个 + 为什么"。读完你能用证据决策 graph vs harness · 不再凭感觉。

📌 测试条件

  • Model: anthropic:claude-sonnet-4-5-20250929 (main) · anthropic:claude-haiku-4-5 (fast)
  • Tools: 完全相同的 11 + harness 多 2 个 (final_answer, dispatch_subagent)
  • Memory / profile: 空 · 每个 query 起新 session
  • Prompt caching: 开
  • 每个 query 跑 3 次取中位数，避免 LLM 抖动

关键 takeaway

graph 在 trivial query 上烧了 5 次 LLM call · harness 烧了 1 次。15× cost · 6× latency。所有"以防万一"的节点（planner / critic / extractor）在简单问题上都是纯浪费。这是 graph 风的结构性税——没法避免 · 除非你 graph 里加 "if trivial then skip planner" 之类的 routing · 但那又复杂化了。

Harness 的执行轨迹

step 0: LLM 看到 query · 决定先 search
        → tool_call: web_search(query="PagedAttention paper vLLM")
step 1: 看到 search 结果 · 决定算内存
        → tool_call: calculator(expr="4096 * 32 * 4096 * 32 * 2 * 2 / 1e9")  
                                          # token * layer * hidden * head * (K+V) * fp16
step 2: 看到计算结果 = 0.137 GB · 决定答
        → 直接生成最终答案 · 无 tool_call
        → loop exits (no tool_call)
为什么 graph 多了那么多 LLM call

graph 版的 13 步包含：1 perception · 1 guardrail · 1 planner · 4 executor (ReAct loop 2 轮 · 每轮 1 个 LLM call + 1 个 tool 节点) · 1 critic · 1 extractor · 1 guardrail · 1 final · 中间还有几个 routing decision。每个 LLM call 都带 ~1500 token 的 system prompt + history · 累计 12.5k token。harness 因为 LLM 一次决定多步 · 加上没有 critic/extractor 的额外 call · 直接砍到 5.8k token。

为什么 harness 在复杂 query 上只略微赢

这种真正"该 multi-agent" 的 query · graph 的 supervisor 设计是其设计目标 · overhead 摊到了真实工作里 · 比例下降。但 harness 仍胜在：(1) 主 agent 天然知道整合的语境 · writer 模板化倾向小；(2) 没有 critic 节点的 conservatism（critic 经常 over-suggest 改回低风险方案）；(3) 主 agent 决定 dispatch sub-agent 跟它们结果整合 · 是一次 LLM call · 不需要额外的 supervisor 决策 LLM。

Query 5 · memory 修正

多轮 · 用户更新自己之前说过的话

两种 memory 范式碰到"撤销"语义 · 表现完全不同

场景

三轮对话：

  1. Turn 1: "我最喜欢的编辑器是 helix · 帮我记下来"
  2. Turn 2: "其实刚才说的 helix 是开玩笑 · 我用 vscode"
  3. Turn 3: "我用什么编辑器？"

Graph 版（auto-extract memory）

Turn 1: extractor 自动 LLM call · 抽出 fact "用户喜欢 helix" · 写 memory
Turn 2: extractor 看到这句 · 但它只能写 · 不能更新 · 写了一条"用户开玩笑说 helix"
        → 长期 memory 现在有 2 条互相冲突的 fact
Turn 3: perception recall memory · top-k 拿回 2 条 · LLM 看到矛盾 · 50% 概率说错
        实测结果："你喜欢 helix" ❌

Harness 版（LLM-driven memory）

Turn 1: 主 LLM 看完 user msg · 决定 call remember(text="用户喜欢 helix", kind="preference")
Turn 2: 主 LLM 看到这条 + history · 决定 call recall(query="编辑器") · 看到老 fact
        → 决定 call remember(text="用户用 vscode · 之前说 helix 是开玩笑", kind="preference")
        → 还可能 call delete_memory(...) 如果你给了这个 tool
Turn 3: 主 LLM 决定 call recall(query="编辑器") · 拿到最新 fact
        实测结果："你用 vscode" ✓
关键差别

graph 的 auto-extract 是"写"模型——只追加 · 不更新 · 不删除。harness 的 LLM-driven 是"读+写+判断"模型——主 LLM 看到上下文 · 自己决策"该不该更新 / 该不该删"。这就是 chapter 7 讲的 "ownership" 差别 · 在 memory 修正这种 edge case 上特别明显。

分析

为什么差距是这样 · 3 个根因

不只是"少了几个节点" · 是结构性的

§1根因 1 · Fixed overhead vs proportional cost

Graph 版每个 query 都付出 固定的 planner + critic + extractor 三次 LLM call，无论 query 多简单。这是 fixed overhead。Harness 版的 cost 跟 query 复杂度成正比——简单 query 1 call · 复杂 query 多 call · 你付的钱跟拿到的价值匹配。

§2根因 2 · Decision 内嵌 vs Decision 独立

graph 把"该不该 supervisor / 该不该 critic / 该不该 extract" 都做成独立的 LLM call，每次决策都要传 context + 等响应。harness 把这些决策内嵌进主 agent 的 tool 选择——主 LLM 一次 decide 完所有事。所以 harness 的 LLM call 数大概是 graph 的 1/3。

§3根因 3 · 模型能力比框架结构重要

2024 年前 graph 设计有意义——模型不够聪明 · 你必须用 framework 强加结构。2024 年后 Claude Sonnet 4 / GPT-5 等模型的 tool calling + reasoning 已经够强 · framework 强加的结构反而是 over-engineering。当模型变聪明时 · framework 应该让出路 · 不是加更多节点。

Anthropic 在 Building Effective Agents 里讲得很直白："when in doubt · do less framework · do more model"。

实操

怎么自己跑一遍

花 10 分钟亲眼看见数据 · 比读手册有用

  1. Clone + setup： 
    git clone https://github.com/doraemonlyz-jpg/taotao-agent
cd taotao-agent
cp backend/.env.example backend/.env
# 编辑 .env · 填 ANTHROPIC_API_KEY
cd backend
uv venv .venv && source .venv/bin/activate
uv pip install -e .
uvicorn app:app --reload
  2. 跑 Q1 graph 版： 
    curl -N -X POST http://localhost:8000/chat \
     -H "Content-Type: application/json" \
     -d '{"message":"1+1=?"}'
    看 SSE 事件流 · 数 trace event 个数。
  3. 跑 Q1 harness 版： 
    curl -N -X POST http://localhost:8000/chat/v2 \
     -H "Content-Type: application/json" \
     -d '{"message":"1+1=?"}'
  4. 看 token / cost： 
    curl http://localhost:8000/usage    # 全局总数
# 或加 ?session_id=xxx 看单 session
  5. 把 5 个 query 都跑一遍 · 自己做表对比。模型抖动会让数据 ±15% · 跑 3 次取中位数。
真正掌握的标志

跑完后回答这 3 个问题 · 不查手册：(1) 为什么 graph Q1 烧了 5 次 LLM call？答得出名字（planner / critic / extractor / executor / output）。(2) harness Q4 的 dispatch_subagent 跑完后 · 主 agent 怎么知道 sub-agent 的结果？答 trace 里看 ToolMessage。(3) Q5 graph 出错的根本原因？答 auto-extract 只追加不更新。3 个都答得出来 = 你掌握了 harness 跟 graph 的本质。
