Book 18 · 行为面 + 项目深挖

每场 senior 面 都有 1–2 轮 不是技术题

技术轮答得再亮，行为面崩一轮也能把 offer 拖到中位数甚至下放到 mid 级。 这一本把"非技术"的两类题——项目深挖和行为题——按头部公司 （Anthropic / OpenAI / Stripe / Cursor / Anysphere / 字节 / Meta）真实流程拆开， 给 12 条 senior signals · 6 类 STAR 模板 · 30 道原题答法 · 用 taotao-agent 当例子的项目演练剧本。

8 章 · 30 真题（按公司）· 50 词术语 · 阅读 ~45 分钟

§ 1为什么 senior 一定有行为面

Mid → senior 的核心差异不是"会用更多框架"，是 能独立 ship、能带人、能扛模糊。 技术轮验证不了这些，所以每家公司都会安排 1–2 轮专门测：

公司行为轮数典型轮名谁面
Anthropic2"Past Project Deep Dive" + "Values"HM + cross-functional
OpenAI2"Project & Impact" + "Behavioral / Hiring Manager"HM + 隔级
Stripe1–2"Behavioral" + "Operating principles"HM + 隔级
Cursor / Anysphere1–2"Founder chat" + 团队 culture fitfounder + 1–2 IC
字节 / Meta1"Behavioral" / 总监终面L+2 级

硬规律

Anthropic 的"Project Deep Dive"普遍 60–75 分钟只挖 1 个项目， 细到 "your DB schema 第三张表的索引为什么这么建"。准备时选 1 个能扛得住 1 小时连珠炮的项目， 而不是简历上铺 5 个。

§ 212 条 senior signals · interviewer 在听什么

"senior" 这个标签由面试官打分给的 signals 拼出来。下面 12 条是头部公司 hiring rubric 的并集——回答任何题之前，心里先核对你给出去的信号有几条命中：

Signal面试官想听到的话术 / 例子
Trade-off awareness"我评估过 A / B / C 三条路 · 选 B 是因为... · A 的代价是... · 现在回头看 B 也有遗留..."
Cost / latency / quality 三角主动量化："P95 从 8s 降到 2.3s，月 token 成本 $4.2k → $1.8k，accuracy -0.7%"
Ownership用"我"而不是"我们"讲关键决策；说出失败的代价是你扛的
Bias for action"模糊到我没法等 spec 了，我先 ship 一个 stub 让 PM 改"——而不是"我等 spec"
Mentoring / multiplier"我把这个 pattern 写成了内部 doc · 团队 4 人现在都用 · 节省了..."
Cross-functional跟 PM / Design / Sales / Legal 沟通的具体例子（不是只跟 eng）
Production thinking主动提 SLO / on-call / postmortem / runbook / canary
Failure ownership讲一个真失败 + 你做错了什么 + 学到的具体 process 改动
Disagree & commit跟 senior / manager 意见不合、最后按对方走、但也讲清楚自己当时为啥反对
Strategic thinking能解释项目"为什么对公司重要"——不只是"PM 让我做"
Calm under ambiguity"requirements 是后来才稳定的 · 我先做了 X / Y 假设，跟 PM 周会校准"
Curiosity / depth偶尔抛一句"这个我后来 weekend 自己 reverse engineered 看了源码"

面试前 3 天

把这 12 条打印出来，对你准备的 1 主项目 + 2–3 副项目，每条都想一个 1 分钟故事。 不需要每场面试讲全 12 个，但需要"被点哪个有哪个"。

§ 3STAR 6 类常见问题模板

行为题 80% 落在这 6 类。每类给一个 STAR (Situation / Task / Action / Result) 骨架， 照着填你的真实例子就能答出 senior 水准。每个模板控制在 2–3 分钟口述——再长就成自我陶醉，再短就显得空。

3.1 失败 / 错误判断

Q: Tell me about a time you made a wrong technical decision.

S: 2024 Q3 我们在做 RAG 系统，规模从 1 万文档涨到 50 万，召回 P95 从 200ms 跳到 4s。
T: 我作为 tech lead 负责选向量库迁移方案。两条候选：
   ① 升级现有 Pinecone 实例（贵但 0 改动）
   ② 自建 Qdrant 集群（便宜但 3 周工期）
A: 我选了 ②，理由是省 60% 成本。但我低估了运维复杂度——
   上线第二周节点 OOM，两小时 SLO 违约。
   我做错的不是选 Qdrant，而是没要求做 1 周 shadow traffic 验证。
R: 事后我们补 shadow，迁移完成。我把这套流程写进了团队的 "infra migration runbook"，
   后续两个项目复用，再没出过类似事故。
   反思：在自己没运维过的系统上线前，shadow ≥ 1 周不能省。

3.2 跨职能冲突

Q: Tell me about a time you disagreed with a PM.

S: PM 想在 chat 产品里加 "Always-on agent，预测用户下一句话主动 reply"。
T: 我的判断：技术上 doable，但 cost 会涨 4x（每条用户消息额外 1 次 LLM call），
   且伪阳性体验差。
A: 我没直接说 "不行"。
   ① 先写了 1 页 RFC，列出 cost 模型 + 3 种交互对比 + 2 条 A/B 实验设计；
   ② 主动约 PM + Design + Data 一起 30 分钟过 RFC；
   ③ 提议先做窄场景实验（只在等待 > 30s 时主动 reply），把 cost 控在 1.3x。
R: PM 同意了窄版本，A/B 显示 retention +2.1%、cost +28%（在预算内）。
   关键：我没说 "no"，是说 "yes, but 给我数据决定 scope"。

3.3 模糊需求 / 缺信息

Q: How do you handle ambiguous requirements?

S: 老板让做 "an internal AI assistant for ops team"——一句话 spec。
T: 第一周我没动键盘，做了 3 件事：
A: ① 跟 5 个 ops 团队员工各 30 分钟访谈，记 "你们最痛的 3 件事"；
   ② 把痛点聚类：50% 都是"翻 Notion 找 runbook"——决定 v1 只做这一个；
   ③ 写 1 页 spec：scope / non-scope / success metric (P95 找到正确 runbook 的时间)，
      给老板 review 30 分钟，签字。
R: v1 三周上线 · 上线第一个月 ops 团队人均节省 45min/天 ·
   v2 才扩到 incident summarization。
   反思：模糊 spec 时不要 jump to coding，先做 user research + scope 收敛。

3.4 mentor 别人

Q: Tell me about mentoring a junior.

S: Q1 来了个 new grad，第一个月看 PR review 总是过不了第二轮。
T: 我作为他的 onboarding buddy。
A: ① 第一周：观察 5 次 PR · 列出 3 条 pattern（commit 太大、缺 test、命名抽象）；
   ② 给他1:1 doc 模板"每周 PR self-review 5 题"，让他提 PR 前自查；
   ③ 我自己的 PR 主动 cc 他做 reviewer，让他逆向学好 PR 长啥样；
   ④ 第 6 周开始他独立 review 我的 PR。
R: 3 个月后他成了团队 PR 速度最快的人之一。
   关键：mentor 不是"我教你"，是设计self-correction loop 让他自己修。

3.5 主导跨团队项目

Q: Drive a project across multiple teams.

S: 公司决定从 GPT-4 切到 Claude 3.5 + 自家 LoRA fine-tune，涉及 backend / data /
   ML platform / SRE 4 个团队。
T: 我被指定为 tech lead（不是 manager，没有汇报权）。
A: ① 第一周写 1 页 "迁移路线图"——明确 6 个里程碑 + 每个里程碑 owner + 验收标准；
   ② 设双周 sync（4 个 lead 必到，30 分钟，只过 risk + blocker，不讨论 detail）；
   ③ 关键节点用 shadow traffic + canary 切流，给 SRE 留回滚开关；
   ④ 跟 finance 算清月成本 swing，主动 cc CFO 团队。
R: 8 周完成迁移 · 0 incident · 月成本 -42% · accuracy +1.3%。
   关键：没有汇报权时靠白纸黑字 + 节奏 + 透明度。

3.6 抗压 / 紧急修 bug

Q: Time you handled a production incident.

S: 周五晚 10 点 P0：agent 开始 hallucinate 调用不存在的 tool，影响 12% 流量。
T: 我是 on-call。
A: ① 先止血不查根因——立刻把流量从 v3.2 切回 v3.1（提前准备过 feature flag）；
   ② 切完才开始 root cause：v3.2 引入新 tool schema 描述，跟 system prompt 里的
      example 冲突，模型 confused；
   ③ 周六上午写 fix + 加 "tool schema vs example consistency" 的 unit test；
   ④ 周一发 postmortem，列 5 条 process 改进。
R: SLO 没违约（5 分钟内回滚）· postmortem 里的 prompt diff CI 检查后来抓到 2 次类似问题。
   关键：on-call 第一反应永远是止血，不是查 root cause。

§ 4项目深挖 · 4 步框架

Anthropic / OpenAI / Stripe 都有专门一轮叫 "Project Deep Dive"，60–75 分钟只挖 1 个 项目。 下面这 4 步框架是被反复验证有效的：

  1. 选项目 (面试前 1 周)——选 1 个能撑得住 1 小时连珠炮的项目。条件：
    • 你是核心贡献者（不是 "团队的项目"）
    • 有量化结果（latency / cost / accuracy / retention 至少一项）
    • 有真实 trade-off——你做过取舍 + 后来反思过
    • 不超过 2 年——记忆度高
  2. 写一份 internal doc (面试前 3 天)——给自己写，不交给面试官，5 段：
    1. 问题陈述（业务 context · 不要先讲 tech stack）
    2. 关键约束（latency P95 / 成本上限 / 团队人数 / 工期）
    3. 3 条候选方案 · 每条 pros & cons · 你最后选了哪条 · 为什么
    4. 实施过程的 3 个关键决策点（每个 1 段）
    5. 结果数据 · 你事后会怎么改
  3. 5 分钟 elevator pitch (面试前 1 天)——把 doc 浓缩成 5 分钟讲完的版本， 练 3 遍直到不卡壳。开场 30 秒必须包含：业务问题 + 你的角色 + 1 个量化结果。
  4. 面试时——5 分钟讲完后主动停，等面试官提问。不要塞细节， 让面试官引导深度。常见 follow-up：
    • "why not X?" → 你 doc 第 3 段的 pros & cons 派上用场
    • "how did you measure success?" → doc 第 5 段
    • "what would you do differently?" → doc 第 5 段后半
    • "if traffic 10x'd, what breaks first?" → 现场推演
    • "who pushed back? how did you handle it?" → § 3.2 模板

陷阱

最常见的失败是开场就讲 architecture——你以为在 show off 技术， 面试官以为你不理解业务。永远先讲 "why this matters to the user/business"。

§ 5用 taotao-agent 当 demo 项目演练

没真实 ship 过 senior 级项目？拿这个 repo 当例子完全够用——它真实可跑、 有多种实现对比（graph vs harness）、有生产化加固（P1 五条）、 还有真 trade-off（每条 P1 都付出了 latency / 复杂度代价）。

5.1 一份能用的 deep-dive 剧本（5 分钟版）

问题陈述（45 秒）

我们想做一个能在生产环境跑的 agent demo——但市面上 demo 都活在 notebook 里，
真正放到 web 服务后面，5 个问题立刻冒出来：cost 不可控 / parallel tool 拖慢响应 /
memory 无限膨胀 / 前端跟后端 schema 漂移 / 用户切回历史 session 没法。
所以我决定：写两套实现 (graph + harness) 同 query 跑出来对比，再加一层 P1 加固。

我的角色（15 秒）
solo build，从 0 到 17 本配套手册 + ~6k 行 Python + 完整 frontend。

关键决策（3 个，每个 60 秒）

① graph 还是 harness？两边都做了。graph 适合 explicit workflow 可解释；
  harness 适合 open-ended，但 token 高 30–60%。给两个 endpoint 让用户选。

② memory decay 用什么算分？最简单是纯 recency，但 high-value 信息会被冲掉。
  最后用 recency × log1p(use_count) 复合分，配 365 天 half-life，pruning 时
  blast radius 限制 100 条/次，避免一次清掉用户主轴信息。

③ parallel tool execution？asyncio.gather + asyncio.to_thread 桥接同步 tool。
  关键是结果重排回输入顺序，否则 LLM 的 prompt cache hit 会暴跌。

结果（30 秒）

P1 加固后：单 session token 预算护栏防住了 3 次实测越界 (cost +0%) ·
parallel tool 让 web search × 2 + RAG × 1 的总耗时从 3.4s → 1.6s ·
memory prune 把 500 条 noise 清成 80 条 signal · OpenAPI codegen 让前端
schema drift 在 CI 阶段就 fail 不再上线。
   事后会做：cost guardrail 应该按 user 而不是 session，session 太短粒度。

5.2 deep-dive 必背的 5 条数据点

数据点数值原始来源（项目里位置）
graph vs harness 同 query token 比1.0× / 1.3–1.6×docs/harness-vs-graph.html
parallel tool 加速比~2.1× (3 tools)backend/agent/harness/loop.py
memory decay half-life365 天backend/agent/memory/long_term.py
cost guardrail 默认上限$0.30 / sessionbackend/agent/security.py
OpenAPI codegen 触发时机CI · `npm run gen:api` diff 非空 failfrontend/package.json

面试官的 follow-up 准备

  • "why decay 用 log1p 而不是 sqrt？" → log1p 对 use_count = 0 / 1 / 2 三档差异更大，sqrt 在小值附近平坦
  • "why 365 天？" → 拍的，但应该按 corpus 半衰期实测调（reading list 类长，工单类短）
  • "asyncio.gather 怎么处理一个 tool 抛错？" → return_exceptions=True 把 exception 当结果回写，让 LLM 自己决定重试或换路
  • "parallel 后怎么保证 prompt cache？" → results 按输入顺序重排再喂回 LLM，否则 KV-cache prefix 错位
  • "如果用户开 10 个 session 怎么办？" → 这是当前 cost guardrail 漏洞，session 级护栏防不住，应该补 user-level rate limit

§ 630 道头部公司原题 · 按公司分类

下面是 2025–2026 真面 senior agent eng / staff ML 候选人收集到的行为 + 项目题。 给提示不给标准答案——按 § 3 的 STAR 模板和你自己的故事填。

Anthropic (Project Deep Dive 风)

  1. 讲一个你 ship 过最复杂的 LLM 系统。提示：选 1 个项目讲 60 分钟 · 准备好 architecture 草图
  2. 这个系统里 cost 怎么控？P95 latency 怎么测？SLO 多少？提示：必须有数字 · 不知道就别选这个项目
  3. 当时不选另外那条路是为什么？现在回头看后悔吗？提示：每个决策都要有 trade-off · 别 PR 自己
  4. 这套东西如果流量 10×，第一个 break 的是什么？提示：现场推演 · 一般是 vector DB / KV cache / rate limit 三选一
  5. 你们团队怎么 eval 这个 agent？发现 regression 谁 own？提示：能讲到 LLM-as-judge calibration 加分
  6. 讲一个你 disagree 但最后按 manager 走的决定。提示：disagree & commit · 强调 commit 不抱怨
  7. 最近一次你被自己代码恶心到是什么时候？提示：诚实 · 别说 "我从不写烂代码"

OpenAI (Project & Impact)

  1. 你最 proud 的项目 · 它对公司带来什么 measurable impact？提示：必须有 $ / user 数 / NPS / latency 任何一项
  2. 过去 1 年最大的一个 mistake，怎么 recover 的？提示：选真错过的事 · 别选 "我太追求完美" 这种
  3. 讲一个你影响了别的团队 / 让公司方向改变的例子。提示：senior 信号 · 不只是写代码
  4. 给你 3 个月 + 不限预算，你会做什么 agent？提示：能展现 product taste · 不只是 "更大模型"

Stripe (Operating Principles)

  1. 讲一个 "Move with urgency and focus" 的例子。提示：Stripe Operating Principles 之一 · 提前看 stripe.com/jobs
  2. 讲一个 "Think rigorously" 的例子。提示：要有量化分析 · 不只 gut feeling
  3. 你 ship 过最大的一次 cost optimization？提示：Stripe 内部对 unit economics 敏感
  4. 怎么处理一个长期低绩效但 cooperative 的同事？提示：senior+ 才会被问

Cursor / Anysphere (Founder chat)

  1. 你为什么想来 Cursor？三个月内你能加什么 feature？提示：用过产品 · 列 3 个具体 issue + 你的 fix 思路
  2. 你怎么看 agent 未来 1–2 年的演进？提示：有 take · 不要骑墙
  3. 给你 1 周时间，你会怎么 prototype 一个新功能？提示：bias for shipping
  4. 你最不喜欢现在 Cursor 的哪个设计？提示：诚实 · 但要 constructive

字节 / Meta (总监终面)

  1. 你怎么衡量自己跟同 level 的差距？提示：自我认知 · 别说 "我没差距"
  2. 你带过最难带的人是谁？怎么处理的？提示：mentoring 信号
  3. 你跟你最近 1 个 manager 关系怎么样？为什么离职？提示：客观讲 · 别贬低
  4. 未来 5 年你想成为什么样的工程师？提示：不要说 "我想做 manager" 除非 JD 写明 EM 路线
  5. 你做过最孤独的一次决定是什么？提示：ownership 信号 · 顶住压力做的事

通用 wildcard (任何家都可能问)

  1. 你怎么 stay current？最近读的 paper / blog / repo 是？提示：能讲 1–2 个 specific · 别说 "twitter"
  2. 跟一个不懂 LLM 的人 30 秒解释 RAG。提示：测沟通能力
  3. 讲一个你反对了主流意见但事后被证明对的事。提示：independent thinking
  4. 你最近 6 个月学到的最反直觉的一件事是？提示：能讲一个具体技术细节加分（e.g. "prompt cache 比 model size 影响 cost 更大"）
  5. 给你一个 junior + 1 个 incident，前 30 分钟你做什么？提示：incident management · 先止血再 mentor
  6. 你对 senior 这个 title 的定义是什么？提示：自检 · 用 § 2 那 12 条信号
  7. 反向问问题环节：你想问我什么？提示：见 § 7

§ 7反向提问清单 · 6 类高质量问题

面试最后 5–10 分钟反向 Q&A 不是闲聊——问什么本身就是 signal。 下面 6 类避开"福利、远程政策"等 HR 题，每类挑 1–2 个你真关心的：

类别示例问题你能 read 出什么
团队节奏 "你们一个新 feature 从 idea 到生产平均多久？" 体感 ship velocity · 是不是 spec → review → ship 各一周 vs 当天上
技术深度 "过去 3 个月你们 production 出过最严重的 incident 是什么？root cause 是？" 面试官敢不敢讲真事 · 团队复盘文化
组里现在最大瓶颈 "如果你能给团队加一个能力，你会要哪种 senior？" 探出团队 gap · 也暗示你能不能填
面试官个人 "你来这家公司多久了？最 enjoy 的一件事是？" 看面试官眼睛有没有光 · 是 honest 还是 PR 答案
Career growth "过去两年从 IC4 升 IC5 的人 · 他们做对了什么？" 晋升节奏是否畅通 · 路径是否清晰
组里的 con "如果让你给加入这个团队的人一句忠告，会是什么？" 客气问 "缺点" · 答得越具体越好

心法

少问"团队 tech stack 是什么"（公开信息）。 多问"上次你们改 stack 是因为什么"（讲故事的问题）。

§ 8红黑榜 · 该说 / 不该说

✅ 该说

具体数字："P95 从 8s → 2.3s · cost 月省 $4.2k → $1.8k"

事后反思："如果重做，我会先...因为..."

承认不会："我没真做过 RLHF · 但我读过 paper · 你们组怎么用？"

用 "我"：关键决策用第一人称

讲到边界："这超出我当时 scope 的部分 PM owns"

主动量化：被问 "效果如何" 时立刻给数字

❌ 不该说

"我们"滥用——分不清你做了啥

抱怨前公司——任何形式 · 包括 "manager 不懂技术"

装懂——被问到完全不会的，承认；硬编一定翻车

"我太追求完美" 当缺点——面试官秒识别 PR 话术

讲项目只讲 architecture——business context 永远先

"那个不是我做的"——但又说不清谁做的

反问 "你们工作压力大吗"——闪退信号

最容易被忽略的 1 条

不要回避 "你的失败"。被问失败不答或答得很轻 = 不诚实 + 没成长。 准备 2 个真失败（一个技术 · 一个协作），讲清楚你做错了什么， 而不是甩锅给 "需求变了 / 时间紧"。
