§ 1VLM 一句话 · 它跟纯文本 LLM 差在哪

VLM (Vision-Language Model) = LLM 把"图像 token" 跟"文本 token" 拼在一起处理。 架构上几乎都是同一套 transformer，差别在前面多一个 vision encoder （ViT / SigLIP / CLIP-style），把图像切 patch、过 encoder、拿到 image tokens， 跟文本 token concat 进 LLM。 

image (224×224 / 336×336 / 1024×1024)
  └─ vision encoder (ViT) → patch embeddings (e.g. 576 tokens for 24×24 patch)
       └─ projector (MLP / Q-Former / Resampler) → 跟 LLM hidden_dim 对齐
            └─ concat with text tokens → 普通 transformer forward
                 └─ 输出文本 (autoregressive)

关键 trade-off：image tokens 数量决定能看清多少细节但也决定推理成本。 GPT-4V 默认每张图 ~1100 tokens · Claude 3.5 ~1600 · Gemini 1.5 Pro ~258（很激进的压缩）· Qwen-VL 用动态 patch（小图省 token，大图多 token）。

面试常考

"为什么 VLM 处理 1024×1024 图比 224×224 慢这么多？" 答：image tokens 数量 ≈ (width/patch)²，1024 / 14 patch → ~5300 tokens vs 224 / 14 → ~256 tokens · 多 20 倍 · 走 attention 是 N² · 推理时间 400 倍。 所以工业用法基本都是 动态 resize / multi-crop 平衡。

§ 3Screen-use / Computer-use · 怎么实现

Computer Use (Anthropic 2024-10) 让 Claude 能看 desktop 截图、移动鼠标、 敲键盘。Cursor / Devin / Open Interpreter 已经基于这套思路落地。看似魔法， 其实就是VLM + tool-call + screenshot loop： 

loop:
  screenshot = take_screenshot()          # tool: 截屏 -> bytes
  obs = vlm.run(
    system="你是 desktop agent · 看截图 · 决定 click/type/key/scroll · 输出 tool_call",
    images=[screenshot],
    messages=history,
  )
  if obs.tool_call:
    execute(obs.tool_call)                # tool: pyautogui.click(x,y) / type(text) / etc
    history += [obs.tool_call, "done"]
  else:
    return obs.message                    # 任务完成 · 输出给用户

3.1 关键 tool 集（Anthropic Computer Use 标准）

toolsignature关键参数
screenshot() → image压缩 / 分辨率
mouse_move(x, y) → okVLM 要直接吐 pixel coord
left_click(x, y) → ok常合并 mouse_move
type(text) → ok键盘输入 · 多语言
key(combo) → ok快捷键如 cmd+s
scroll(direction, amount) → ok页面浏览

3.2 三大坑

  • 坐标精度：VLM 直接预测 pixel 坐标常偏 10–30 px。修法：让 VLM 先输出"目标 element 描述"，再用视觉 grounding 模型（OmniParser / SeeClick）转 bbox。
  • screenshot 太大太贵：1080p 全屏 ≈ 2 MP · 1500+ image tokens。修法：先 detect 当前 active window 裁切 · 或缩到 768 px 长边。
  • 无限 loop：模型卡在某一步反复重试。修法：① 强制每 5 步问"任务完成度 0–100"；② 累计 20 步还没结束直接 abort + 给用户 fallback。

VLM + UI 的开源底座

  • OmniParser (Microsoft, 2024): 把任意截图解析成结构化 element tree
  • SeeClick (2024): 文本 "log in 按钮" → bounding box · 解决坐标偏问题
  • Anthropic Computer Use Demo: github.com/anthropics/anthropic-quickstarts
  • Open Interpreter: 开源的 desktop / terminal agent · 可以 self-host

§ 4语音 agent 三层栈 · STT → LLM → TTS

"voice agent" 不是 LLM 直接听说，而是三层 pipeline + 一堆 latency-killer 中间件： 

用户开口
   │
   │  ① VAD (Voice Activity Detection)  · 判断 user 在说话 / 已说完
   │
   ▼
   ② STT (Speech-to-Text)              · streaming 转文字
   │
   ▼
   ③ LLM (生成 response)                · 流式输出 token
   │
   ▼
   ④ TTS (Text-to-Speech)              · 流式合成音频
   │
   ▼
   ⑤ 音频送回用户耳朵

4.1 各层主流方案 (2026)

层主流 / 默认替代关键指标
VAD Silero VAD (开源 · 1MB · CPU 跑) WebRTC VAD · Picovoice Cobra 对静音灵敏度 / 假阳性率
STT Deepgram Nova-3 · OpenAI Whisper / gpt-4o-transcribe AssemblyAI · 自部署 Whisper-large-v3 WER · 首字延迟 (TTFB < 200ms)
LLM GPT-4o-mini · Claude Haiku · Llama-3.1-70B 同 SLM 生态 TTFT < 300ms · TPS > 50
TTS ElevenLabs Flash · Cartesia Sonic · OpenAI TTS Resemble · Coqui XTTS（自部署） 首字音频 < 100ms · 拟人度

4.2 端到端 latency budget · 把 1500ms 拆开

用户体感"自然对话" 的硬杠是 800–1200ms（电话级）； "fluent / 像真人"是 500ms 以下。budget 拆：

VAD detect end-of-speech 250–400ms (主要是 silence threshold)
STT 出最后一个 partial 100–150ms
LLM TTFT (首 token) 250–500ms
TTS 首字音频 80–120ms
网络 RTT (双向) 60–120ms
──────────────────────────────────
感知延迟（用户开口 → 听到 agent 第一个字）≈ 800–1200ms

4.3 latency 5 大优化招

  1. STT 出 partial 就喂 LLM（不等 final）· LLM 可以"预热" KV-cache
  2. "endpointing": 不等 800ms 静音才认为说完 · 用 STT 的 punctuation prediction · 听到 ?/. 立刻打断
  3. LLM 开 streaming + TTS 开 streaming: LLM 出第一句 / 第一个 punctuation 就喂 TTS · 不等整个回复
  4. 填充词 (filler) 抢时间: LLM 还在算时先 TTS 一句"嗯..."掩盖 · Vapi / Retell 默认行为
  5. 预热 + warm pool: 长连接（WebSocket）+ 预 fork worker 池 · 砍冷启动

§ 5Vapi / Retell / Pipecat · 三选一

不用从零搭三层栈——下面 3 个框架已经把 latency 优化做得很到位。横评：

方案定位典型 latencyvendor lock适合谁
Vapi SaaS · 全栈托管 · 自带 phone number ~700ms 高（绑定 SaaS） 需要电话 inbound/outbound · 创业公司起步
Retell AI SaaS · 主打企业电话场景 ~600ms 高 客服 / 预约 / 销售 outbound
Pipecat 开源 framework (Daily.co) 取决于自己组件 · 可压到 ~400ms 低 需要 deep customization · 自部署 · 接 WebRTC

5.1 Pipecat 核心设计 · 一段代码看完

from pipecat.pipeline.pipeline import Pipeline
from pipecat.services.deepgram import DeepgramSTTService
from pipecat.services.openai import OpenAILLMService
from pipecat.services.cartesia import CartesiaTTSService

pipeline = Pipeline([
    transport.input(),                  # 进 WebRTC frames
    DeepgramSTTService(api_key=...),    # frames -> partial text frames
    LLMUserResponseAggregator(),        # 把 partial 攒成整句再 ship
    OpenAILLMService(api_key=..., model="gpt-4o-mini"),
    CartesiaTTSService(api_key=..., voice_id="..."),
    transport.output(),                 # 出 WebRTC frames -> 用户耳朵
])

# 跑：
runner = PipelineRunner()
runner.run(PipelineTask(pipeline))

关键设计：每个 service 只处理 frames（音频 / 文本 / 控制）， 所有连接是异步流。LLM service 一边输出 token 一边 push 给 TTS service · 不等完。

谁用 Pipecat

ChatGPT Voice Mode 早期 demo 用了 Pipecat · Anysphere / Cursor 内部 voice 实验 · Daily.co 自家产品。2025 H2 起，企业自部署 voice agent 几乎都从 Pipecat 起步。

§ 6原生多模态模型 · GPT-4o-realtime / Gemini Live

上面四层栈是松耦合组合 · 灵活但延迟有下限（VAD + STT 串行）。 2024 H2 起出现"原生多模态"——audio in / audio out 一个模型搞定， 跳过 STT 和 TTS 两步。

模型架构typical latency定价 (per min)
GPT-4o Realtime API同一个 transformer 接受 audio token~320ms~$0.06 in / $0.24 out
Gemini 2 Live同上 · video in 也 native~400ms~$0.07
Moshi (Kyutai 开源)full-duplex (能边听边说)~200ms$0 自建

权衡：原生模型延迟低、表达力强（能模仿语气 / 笑 / 唱），但难调试 （没有中间 text，看不到模型在说什么）、价高、定制差（不能换 voice / STT 引擎）。 生产上 2026 大多数公司还是 松耦合栈 为主，原生只在高端体验场景用。

§ 74 真翻车复盘

7.1 翻车 A · VLM screenshot 分辨率太大爆 cost

背景：computer-use agent 跑客户 SaaS dashboard · 每步截 1440×900 全屏。

症状：100 个并发 session · 单月 vision API 账单 $48k · 比预期高 9 倍。

根因：1440×900 → ~1700 image tokens · 每步对 2 张图（before/after action） → 3400 image tokens / step · 平均 12 step / task · 40000 image tokens / task。

修法：① 先 detect active window 裁切（典型省 60%）· ② 长边缩到 1024 · ③ 只在"action 后" 截屏 · 不在 "before"。月成本降到 $11k。

7.2 翻车 B · 语音 agent 在嘈杂环境一直被打断

背景：客服语音 agent · 用户在地铁 / 开车环境调用。

症状：agent 才说半句就被自己背景噪音 trigger VAD 当作"用户在说话" → 自我打断 · 用户体验灾难。

修法：① 加 echo cancellation (Krisp / Daily AEC) · ② VAD threshold 动态根据 ambient noise 调 · ③ TTS 播放期间 VAD sensitivity 降低（仅检测显著语音事件）。

7.3 翻车 C · LLM 流式输出导致 TTS 句子拆得很奇怪

背景：为了低延迟 · 每收到 1 个 token 就尝试喂 TTS。

症状：TTS 出来"嗯。是的。它。能。帮你。" 像机器人。

修法：在 LLM stream 跟 TTS 之间加 sentence aggregator—— 累积到 punctuation (. ? ! 中文 。？！) 再 flush · 同时设 max_chars=80 兜底防长句卡住。

7.4 翻车 D · Computer-use agent 误删用户文件

背景：让 agent "整理一下 Downloads 文件夹"。

症状：agent 把整个 Downloads 拖进了垃圾桶并清空。

根因：① 模糊指令 + ② 没 dry-run + ③ 没要 user confirmation。

修法：① 任何 destructive action（rm / drag-to-trash / overwrite）必须 user confirm · ② sandbox 跑（一个空白 VM） · ③ 大动作前 take backup snapshot · ④ system prompt 加 "如果指令含糊 · 先复述给用户确认 · 不要假设"。
