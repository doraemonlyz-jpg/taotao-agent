Chapter 1 · 立场

RAG vs Long Context vs Fine-tune · 决策树

面试官第一问 · 答错就掉档

§1.2决策树

  • 知识 > 1M token · 必 RAG
  • 知识 ≤ 1M 但要省钱 / 低 latency · RAG
  • 知识 ≤ 200k · 一次性 query · 要最高准确度 · long context
  • 样式 / 风格 / 行为模式（不是事实知识）· fine-tune
  • 领域专有词汇 + behavior + 知识三合一 · fine-tune (LoRA) + RAG 混合

§1.3"Lost in the middle" · long context 致命弱点

Stanford 2023 paper 发现：在 long context 中 · LLM 对开头和结尾召回率高 · 中间召回率显著下降。即使是 GPT-4-Turbo 128k · 中间 30% 位置准确率掉 30%+。

结论：long context 不是 RAG 完美替代品 · 关键信息塞中间 = 模型不认。

§2.26 种 chunking 策略

1. Fixed-size · 最简单 (不推荐生产)

按 token / char 数硬切。优势：快。劣势：切断句子 / 段落 / 表格 · retrieval 召回质量差。

2. Recursive (LangChain default · 改良版)

from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", ".", " ", ""],  # 优先级
)

优先按段落切 · 段落太长退化按句子 · 还长按词。大多数场景 default。

3. Semantic (按语义边界)

用 embedding 算句子相邻相似度 · 相似度突降处切。优势：chunk 内语义内聚。劣势：embedding 慢 · 大文档跑 1 小时。

4. Structural (按文档结构)

按 markdown header / HTML tag / PDF outline 切。技术文档 / API 文档最佳。

5. Context-aware (Anthropic Contextual Retrieval)

2024 末 Anthropic 提：每个 chunk 前用 LLM 加 50-100 字 context（"这段在第 3 章 · 讲 X 服务的 API"）· 嵌入 + 检索都包含 context · failure rate -49% · prompt caching 让 cost 控制住。

CONTEXT_PROMPT = """
<document>{whole_doc}</document>
<chunk>{this_chunk}</chunk>

请用 50-100 字描述 chunk 在 document 中的位置 / 上下文 · 帮助检索。直接给描述 · 不加任何前缀。
"""
context = anthropic.messages.create(
    model="claude-3-5-haiku",
    extra_headers={"prompt-cache": "on"},   # 关键 · whole_doc cache
    ...
).content[0].text

augmented_chunk = context + "\n\n" + this_chunk  # 嵌这个

6. Parent-child / Small-to-Big

嵌入小 chunk（句子级）但检索后返回大 chunk（段落 / 章节）给 LLM。优势：检索精确 + LLM context 丰富。LangChain 有 ParentDocumentRetriever。

§2.3chunk size + overlap · 经验值

场景chunk sizeoverlap
FAQ / 短文档200-400 token0-30
技术文档500-80050-100
书 / 长文 / 法律800-1500100-200
code按 function 切 · 不切函数中间0
最常踩坑

overlap 太大（> 30%）→ vector store 膨胀 + 同一信息检索重复出现 · 浪费 LLM context。overlap 0 → 跨 chunk 信息断开。10-15% chunk_size 是甜区。

不要无脑跟 MTEB 排行榜 · 排行第一不一定适合你

§3.2选型 · 5 个判定

  1. 语种——纯英文 → OpenAI / Voyage；多语 → Cohere multilingual / BGE-M3
  2. 领域——法律 / finance / code → Voyage 有专精模型 + 微调
  3. OSS 强制（数据合规 / on-prem）→ BGE-M3 / Jina v3
  4. cost 极度敏感——Cohere 有 binary embedding (32x 压缩) · 适合 100M+ chunk
  5. 上下文长（要 embed 整段 chapter）→ Jina v3 (8k) / BGE-M3 (8k)

§3.3MTEB 排行的 3 个局限

  • 分布偏差——MTEB 重学术语料 · 你的客服 / 法律 / 内部 wiki 分布完全不同
  • language bias——很多任务英文为主 · 中文任务少且偏新闻
  • train leakage——开源模型可能训过 MTEB · 排行虚高

正确做法：用你自己的 200-500 条 golden query-doc pair 跑 recall@10 / NDCG@10 · 决定哪个 embedding 模型。

§3.4Matryoshka embedding · 2024 流行

OpenAI text-embedding-3 + voyage-3 + Cohere v3 都支持 truncate dim——3072 维裁到 256 维 · 准确率仅掉 5% · 但 vector DB 体积 / 检索速度 12x。

# OpenAI 直接支持
emb = client.embeddings.create(model="text-embedding-3-large",
                                input=texts, dimensions=256).data
# 256 维 · 占用 1/12 · 召回掉 ~3-5%

§4.16 步选型决策

  1. 已经用 Postgres? → pgvector first（少一个服务 · 多花一行 SQL · 80% 场景够）
  2. SaaS · 0 维护 · 不在意贵 → Pinecone
  3. OSS + 高性能 + 复杂 metadata filter → Qdrant
  4. OSS + GraphQL + multi-modal → Weaviate

§4.2HNSW vs IVF · 索引选型

  • HNSW（默认）——召回准 · 内存大 · build 慢；适合 < 100M chunk · 优先选
  • IVF + PQ——内存小 · 召回略低；适合 100M+ · 内存有限
  • DiskANN（Microsoft）——磁盘存 · 内存极少；适合 1B+ · 单机省内存

§4.3Metadata filter · 别忽视

典型例：用户只能搜自己 tenant 的文档 · 必须先 filter tenant_id=X 再 ANN。 错：先 ANN top-100 再 filter（filter 后可能 0 个 · recall 暴跌）。 对：用 vector DB 的 pre-filter 能力（Qdrant / Weaviate / Milvus 都强 · pgvector 也支持但慢）。

§4.4用 quantization · 内存省 4-32x

  • scalar (int8)——4x 压缩 · 召回 -1%；首选
  • binary——32x 压缩 · 召回 -5-10%；超大规模 + 配 rerank
  • matryoshka——见 §3.4
Chapter 5 · Retrieval

Retrieval · dense / sparse / hybrid / multi-vector

单 dense 在 2026 已过时 · hybrid 是 default

§5.14 种检索方式

方式原理强项弱项
Dense (vector)embedding cosine语义相似 · 同义词强精确关键词 / 数字 / 名字弱
Sparse (BM25)词频 + IDF关键词精确 · 数字 / SKU / 人名同义词 / 概念查询差
Hybriddense + sparse 融合两边优势叠加稍贵 · 需要 RRF 融合

§5.2Hybrid · 用 RRF 融合 (Reciprocal Rank Fusion)

def rrf(rank_lists, k=60):
    """
    rank_lists: [[doc1, doc2, ...], [...]]  来自不同 retriever
    返回融合后排序
    """
    scores = {}
    for rl in rank_lists:
        for rank, doc in enumerate(rl, 1):
            scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank)
    return sorted(scores, key=scores.get, reverse=True)

dense_top   = vector_db.search(emb_q, k=50)
sparse_top  = bm25_search(q, k=50)
final = rrf([dense_top, sparse_top])[:10]   # top 10 给 LLM

§5.3Multi-vector · ColBERT 是啥

传统 dense：1 doc → 1 vector。ColBERT：1 doc → N 个 token-level vector · query 跟 doc 算 max-sim 矩阵。召回准确度高 5-15% · 但存储是 N 倍 · 计算贵 5-10x。

2025 实战：ColBERTv2 / PLAID / answer.ai 的 ColBERT-small · 已经 production-ready · 配 Vespa / RAGatouille 库。

§5.4Top-k 怎么定

常见错误：top_k = 5 喂 LLM 完事。正确：

  1. retrieve top 50（dense + sparse 各 50）
  2. RRF 融合 → top 50
  3. rerank → top 5-10
  4. 给 LLM

1 步 retrieve top 5 vs 多步 50 → 5 → recall 差 30%+。

Chapter 6 · Rerank

Reranking · 3 流派对比

不加 rerank 的 RAG · top-k 准确率掉 20-40%

§6.1Why rerank

embedding 召回是近似·top-50 里有 5-15 个真相关 · 但顺序往往不对。rerank = 用更贵但更准的 model 给 top-50 重新打分。

§6.23 流派

流派代表原理delay价
API rerankCohere Rerank · Voyage Rerank · Jina RerankSaaS · 给 query+50doc 返打分~200ms$1-2/1k req
Cross-encoderBAAI/bge-reranker-large · ms-marco-MiniLM本地 · query+doc 拼接进 BERT 二分类~50-200ms (GPU)OSS · 自托管
LLM rerankRankGPT · GPT-4 / Claude 排LLM 输出 ranking1-3s$$ · 慢但准

§6.3Cohere Rerank 实战

import cohere
co = cohere.Client(api_key=...)

results = co.rerank(
    model="rerank-english-v3.0",
    query=user_q,
    documents=[d.page_content for d in top_50],
    top_n=8,
).results

reranked = [top_50[r.index] for r in results]

§6.4什么时候不该 rerank

  • top-k ≤ 5 · 文档很短——rerank 收益 < 5% · 不值 latency
  • 用户对 latency 敏感（语音 / Realtime）—— rerank 加 200-2000ms
  • retriever 已经召回精准（精确关键词 + filter 后只剩 5 条）
Chapter 7 · 高级

Advanced · HyDE / CRAG / Self-RAG / Adaptive RAG

naive RAG 的 4 个升级方向 · senior 必懂

§7.1HyDE (Hypothetical Document Embeddings)

问题：query "如何配置 SSL?" 跟 doc "用 OpenSSL 工具创建证书 · 配置 nginx ssl_certificate ..." 在 embedding 上未必近——query 短 / 抽象 · doc 长 / 具体。

方案：让 LLM 先假装写一段答案 · 用假答案的 embedding 去查。

hypo = llm.invoke(f"假装回答这个问题（即使不确定也写）：{q}").content
hypo_emb = embed(hypo)
docs = vector_db.search(hypo_emb, k=10)

RAG benchmarks 上召回 +5-15%。劣势：每 query 多一次 LLM 调用。

§7.2CRAG (Corrective RAG)

2024 paper · 核心：retrieval 后先评估质量 · 不好就 fallback web search。

  1. retrieve → top-10
  2. 用 lightweight evaluator (微调小 model) 给每 doc 打 relevance（high / mid / low）
  3. 都 high → 直接给 LLM
  4. 有 low → 触发 web search 补
  5. 都 low → 完全靠 web search

这是第一个引入"质量自评"的 RAG 模式 · agentic RAG 雏形。

§7.3Self-RAG

UW 2023 paper · 把 RAG 决策本身交给 LLM——用特殊 token：

  • [Retrieve]——LLM 决定要不要 retrieve
  • [IsRel]——retrieve 后 LLM 评估 relevance
  • [IsSup]——LLM 评估 output 是否被 retrieved 支持
  • [IsUse]——LLM 评估 output 对用户有用

需要微调一个会输出这些 token 的 LLM。生产用得少 · 概念被 Adaptive RAG / Agentic RAG 继承。

§7.4Adaptive RAG

2024 paper · 一个 router 先分类 query 的"难度"：

  • 简单事实 → no retrieval · 直接 LLM
  • 单跳 retrieval → 标准 RAG
  • 多跳 / 复杂 → multi-step RAG (iterate)

意思是不是每个 query 都该走 RAG。生产实战：用 LangGraph router node + LLM 分类 + 3 条不同 path。

Chapter 8 · Agentic

Agentic RAG · iterative / multi-hop · 跟传统 RAG 区别

2025-2026 主流形态 · "RAG 已死" 死的是 naive · 复活的是这个

§8.14 种区别

Naive RAGAgentic RAG
retrieval 次数1 次多次 · LLM 决定
query 改写无LLM 把 user query 拆 / 改写成多个 sub-query
retrieval 失败处理硬塞 LLMLLM 看到不够 → 改 query 再 retrieve
多源整合1 个 vector DBvector + SQL + web + API · 用 tool 调

§8.2Agentic RAG 实现 · LangGraph 模板

from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode

@tool
def vector_search(query: str) -> list[str]:
    return vector_db.search(query, k=8)

@tool
def web_search(query: str) -> list[str]:
    return tavily.search(query, k=5)

@tool
def sql_search(question: str) -> str:
    sql = nl2sql(question)
    return str(db.execute(sql).fetchall())

agent_llm = llm.bind_tools([vector_search, web_search, sql_search])

def agent_node(state):
    return {"messages": [agent_llm.invoke(state["messages"])]}

def should_continue(state):
    return "tools" if state["messages"][-1].tool_calls else END

g = StateGraph(...)
g.add_node("agent", agent_node)
g.add_node("tools", ToolNode([vector_search, web_search, sql_search]))
g.add_conditional_edges("agent", should_continue, ["tools", END])
g.add_edge("tools", "agent")
g.add_edge(START, "agent")

LLM 看到 query → 决定调哪个 tool · 看 result → 决定够不够 · 不够再调。这就是 Agentic RAG。

§8.3Multi-hop RAG · 真复杂查询

例："小米成立时 CEO 现在在哪家公司" 需要：

  1. 查"小米成立年份" → 2010
  2. 查"2010 小米 CEO" → 雷军
  3. 查"雷军 现任公司" → 仍小米

naive RAG 一步 retrieve 完全答不出。Agentic RAG 自然多次 retrieve。

Chapter 9 · 变体

GraphRAG / Multi-modal RAG / SQL RAG

3 个高频场景的特化 RAG · 面试经常被追问

§9.1GraphRAG (Microsoft 2024)

问题：传统 RAG 不能回答summary 类 / global 类 query · "这本书的整体主题是什么"。

方案：

  1. indexing 时用 LLM 抽实体 + 关系 → 建知识图
  2. 对图做community detection（Leiden algorithm）· 每社区用 LLM 生成 summary
  3. query 时检索 community summaries · 回答 global question
  4. local question 仍用传统 vector retrieval

开源：microsoft/graphrag + LightRAG (轻量替代)。

劣势：indexing 贵（~$10-50/100k 字）· 主要用在 long doc 集合 / 内部 wiki / 法律。

§9.2Multi-modal RAG

文档含图 / table / chart 怎么办？

  • 方案 A · Caption + text——LLM (GPT-4V) 给图生 caption · 把 caption 当 text 走传统 RAG。简单 · 准确度中。
  • 方案 B · Multi-modal embedding——CLIP / SigLIP / Cohere embed v3 多模态 · 文本 + 图同空间 embed。统一空间 · 准确度高。
  • 方案 C · ColPali / ColQwen——直接 embed PDF 页面图（不 OCR）· "把 PDF 当图"。2024 末新方案 · 视觉重的文档（PPT / 报告 / 论文）效果最好。

§9.3SQL RAG (Text2SQL with retrieval)

问题：query "上月 GMV 增长多少" · 数据在 1000+ 表 schema 中。

方案：

  1. 所有 table schema + column desc + 例 query embed 入 vector DB
  2. query → retrieve top-K 相关 table schema
  3. LLM 看相关 schema + query → 生成 SQL
  4. execute SQL · 把结果 + SQL 回 LLM 生最终 answer

开源：Vanna AI · DB-GPT。

Faithfulnessanswer 是否忠于 retrieved context · 没幻觉LLM 抽 answer 中 claims · 验每个 claim 是否在 context 中
Answer Relevanceanswer 是否回答了 questionLLM 反向给 answer 生 N 个 question · 算跟原 question 相似度
Context Precisionretrieved chunk 中真有用的比例LLM 评每 chunk 是否对 answer 有贡献
Context Recall需要的信息是否被 retrieved需要 ground truth · LLM 评 GT 中 statement 是否在 context 中

这 4 个把 RAG pipeline 拆开评——retrieval 锅 vs LLM 锅一目了然。

§10.2Ragas 实战

from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall

dataset = Dataset.from_dict({
    "question":        ["如何配置 SSL?", ...],
    "answer":          [agent_outputs, ...],
    "contexts":        [[chunk1, chunk2, ...], ...],
    "ground_truth":    ["用 OpenSSL 创建...", ...],
})

result = evaluate(dataset, metrics=[faithfulness, answer_relevancy,
                                     context_precision, context_recall])
print(result.to_pandas())

§10.3retriever 单独评 · recall@k / NDCG@k

不要等到 RAG end-to-end 才发现 retrieval 烂。独立 eval retriever：

  • 50-200 query + ground-truth doc id
  • retrieve top-k · 算 hit rate / recall@k / MRR / NDCG@k
  • recall@10 < 0.8 → 不要谈 LLM · 先修 retrieval

生产 8 大坑 + 真实复盘

这章是别人血换的

  1. chunk overlap 太大 → vector DB 膨胀 + LLM context 重复——10-15% 是甜区
  2. 没用 metadata pre-filter → tenant 串号 / 检索很慢——必须 vector DB 原生 filter
  3. rerank 放在错误位置——应该 retrieve 50 → rerank → 5；不是 retrieve 5 → rerank（rerank 5 个收益微）
  4. 没监控 retriever recall · 只盯 E2E——retrieval 烂 LLM 再强没用 · 必须独立 SLO
  5. embedding 模型升级没重 embed——新旧 embedding 不可比 · 必须全库重 embed (灰度)
  6. prompt 没引用 source · LLM 编——必须强制 LLM "Citation: [chunk_id]" · post-hoc 验证
  7. 数据更新不增量 / 全量重 embed 太贵——按 doc_id 增量 + soft delete · 用 ETL 调度
  8. 对长文档没 parent-child · LLM context 碎——小 chunk embed · 大 chunk feed

§11.1真实复盘 · 法律检索 RAG · 引用错位

背景：法律 RAG · 律师查 "合同条款类似案例" · 用户反馈 "你给的案例 ID 是真的 · 但案件描述跟引用合不上"。

挖根因：parent-child 模式 · child chunk 携带的 metadata 是错的 parent_id——chunking 时 parent_id 用了文件名 · 后来文件改名 metadata 没更。LLM 引用的是 child 的 metadata · UI 跳转拉的是新文件 · 内容对不上。

fix：

  1. parent_id 用 hash(content) 不用文件名
  2. 每次 ingest 校验 parent-child 链路完整
  3. UI 引用跳转时显示 chunk 内容 · 用户能立刻发现错位

教训：metadata 设计跟稳定 ID 一样要 careful · 用 immutable hash 不用易变字段

决策 (5)
