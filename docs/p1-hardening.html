Chapter 1 · 总览

为什么这 5 件 · 而不是别的 50 件

P1 不是新功能 · 是把现有功能"撑得住"

Agent 项目从 demo 到产品的常见死法 —— 不是"功能不够"，是"用着用着就烂"： memory 越查越脏、tool 串行卡住整个对话、长会话烧空预算、前端跟后端的字段对不上、用户找不到自己昨天聊过的内容。 我们这次 sweep 选的 5 件，都不是新增能力，全是给已有能力加底裤。

§1.2P0 / P1 / P2 怎么排

我自己内部的优先级框架（推荐 senior 用）："如果不做，多久会爆"。

  • P0 · 不做 = 上线一周内必爆。例：没有测试、没有 CI、没有 stop button、没有真流式。
  • P1 · 不做 = 上线一个月内开始烂。例：memory decay、cost guard、parallel exec、契约自动化、多会话 UI。
  • P2 · 不做 = 上线半年才显出来。例：多模型路由、暗色模式、WebSocket 替代 SSE、trace 时间轴可视化。

这本盯的是 P1。P0 还有空缺（没有 pytest、没有 CI、没有 graph 侧 token-by-token 流），下一轮 sweep 会补。

概念 · "Production-shape" 不是状态 · 是方向

没有"我的服务现在是 production-ready 的"这种说法 —— 只有"我离 production 还差几步"。每次 sweep 把当前最痛的 N 件解决掉，让掉档时间从一周推到一个月、再推到半年。 这是渐进式硬化（progressive hardening），是 SRE 圈一个非常 senior 的姿态。

Chapter 2 · 记忆衰减

Memory decay · 不会忘的系统是个烂系统

人脑每天忘 90%，是 feature 不是 bug · 我们的 memory 也得

Long-term memory（我们用 Chroma 存 vector）的最常见翻车 —— 跑得久了就变垃圾。 用户随口说一句"记下我喜欢深色主题"，agent 写了；明天用户改主意"还是浅色吧"，agent 又写一条；一个月后 vector store 里一堆互相矛盾的 fact，recall 出来全是噪音。

§2.1痛点 · 三种"脏"

类型例子后果
过时2 年前用户喜欢 Vue · 现在改 Reactrecall 出旧偏好 · 答错
低价值"今天天气真好" 这种闲聊也写进去稀释相关结果 · 召回率下降
重复"我在字节" / "我是字节员工" / "字节是我公司"3 倍占位 · 1 倍信息

我们已经有 remember_if_new() 用 cosine similarity 在写入时去近重（threshold 0.92）。但 dedup 只解决"重复"，不解决"过时"和"低价值"。需要一套主动遗忘机制。

§2.24 种主流 forgetting 策略 · 我们选哪个

策略原理优点缺点
TTL固定过期时间简单、可证明一刀切 · 重要的 fact 也会被砍
LRU最久没用的先扔好实现 · 跟 cache 一脉相承新存的总输给旧的（即使重要）
LFU用得最少的先扔"重要 = 常用" 直觉新 fact 永远 use_count=0 · 一存就被砍
Recency × Usage 复合时间衰减 + 调用频率 加权同时考虑"新鲜度"和"价值"需要两个 metadata 字段 · 两个权重要调

我们选了第 4 个。公式：

# 越高 = 越值得留
score = 0.7 * recency + 0.3 * usage

recency = exp(-age_days / half_life_days)            # 半衰期衰减
usage   = log1p(use_count) / log1p(max_use_count)    # 对数归一化

这两条公式背后的 senior 取舍：

  • recency 用指数衰减不是线性：新 fact 价值下降快、老 fact 已经稳定了不该急着扔。 exp(-age/14) 表示 14 天后还剩 37% 价值、28 天剩 14%、56 天剩 2%。这就是半衰期（half-life）的物理直觉。
  • usage 用 log1p 归一化：use_count 是长尾分布（大多数 fact 是 1-2 次，少数热门是 50 次）。 直接除会让热门 fact 把所有冷门压到 0。log1p 把 50 压到 ~3.9、1 压到 ~0.7，差距合理。
  • 权重 7:3 偏向 recency：因为"用户偏好会变"是 agent 场景的最大 risk。如果做 docs RAG 反过来 3:7（旧文档反而权威）。

§2.3实现 · 60 行讲完

三个改动点，每个都有意思：

① 写入时多两个字段

def remember(self, text, kind="fact", session_id=None):
    now = _utc_now_iso()
    self.collection.add(
        ids=[mem_id], documents=[text],
        metadatas=[{
            "kind": kind, "session_id": session_id or "",
            "ts": now,
            "last_used_at": now,    # ← 新增
            "use_count": 0,        # ← 新增
        }],
    )

② recall 时 bump 这两个字段

def recall(self, query, k=4):
    result = self.collection.query(query_texts=[query], n_results=k,
                                  include=["documents", "metadatas"])
    ids = (result.get("ids") or [[]])[0]
    docs = (result.get("documents") or [[]])[0]
    self._bump_usage(ids)              # ← 新增
    return docs

def _bump_usage(self, ids):
    current = self.collection.get(ids=ids, include=["metadatas"])
    new_metas = []
    for meta in current["metadatas"]:
        m = dict(meta or {})
        m["use_count"] = int(m.get("use_count", 0) or 0) + 1
        m["last_used_at"] = _utc_now_iso()
        new_metas.append(m)
    self.collection.update(ids=current["ids"], metadatas=new_metas)

③ prune · 复合分排序后砍底部

def prune(self, *, max_keep=None, drop_fraction=None,
          half_life_days=None, dry_run=False):
    out = self.collection.get(include=["metadatas"])
    ids, metas = out["ids"], out["metadatas"]
    max_use = max((int(m.get("use_count", 0)) for m in metas), default=1)

    scored = sorted(
        zip(ids, metas),
        key=lambda p: self._score(p[1], max_use, half_life_days),
    )

    n_drop = total - max_keep if max_keep else int(total * drop_fraction)
    n_drop = min(n_drop, total // 2)         # ← 安全 floor

    victims = [pid for pid, _ in scored[:n_drop]]
    if not dry_run:
        self.collection.delete(ids=victims)
    return {"total": total, "kept": total - len(victims), ...}
细节 · "永远不删超过 50%" 的安全 floor

很容易写出 prune(max_keep=10) 但忘记现在有 1000 条 —— 一刀砍 990。 我们硬编码 n_drop = min(n_drop, total // 2) 永远不允许单次调用删超过一半。 这是 blast radius limit（爆炸半径限制）—— SRE 圈的标准做法。同样的思路在数据库迁移、batch delete、feature flag rollout 里到处出现。

§2.4非显然的工程坑

坑 1 · 时区 naive vs aware

老 memory 是 datetime.utcnow().isoformat() 写进去的（tz-naive）。新代码用 datetime.now(timezone.utc)（tz-aware）。 第一次跑 prune 直接 TypeError: can't subtract offset-naive and offset-aware datetimes。 修复：parse 完之后 if last.tzinfo is None: last = last.replace(tzinfo=timezone.utc)。 升级数据格式时永远要兼容老数据，这是 schema migration 的基本功。

坑 2 · dry_run 必须先实现

prune 是不可逆操作 —— 写完了一定先支持 dry_run=true 让用户看 victim 列表再决定。 我们 endpoint 直接暴露这个开关。这跟 terraform plan / kubectl --dry-run 的设计哲学一致。

坑 3 · 不要在 recall 路径里同步写

我们的 _bump_usage 是同步的，紧贴在 recall() 后面。当前实现可以接受，因为 Chroma 是本地 SQLite + ONNX，update 微秒级。 但如果换成远程 vector DB（Pinecone / Qdrant）就要改成异步（fire-and-forget queue），不然每次 recall 多 100ms 网络延迟很难看。这是"读路径不该同步写"的经典原则。

§2.5怎么用

# dry-run · 看砍谁
curl -X POST :8000/memory/prune -d '{"dry_run":true,"max_keep":50}'

# 真砍 · 留 50 条
curl -X POST :8000/memory/prune -d '{"max_keep":50}'

# 调半衰期 · 短期项目用 7 天 · 长期项目用 90 天
curl -X POST :8000/memory/prune -d '{"half_life_days":7,"max_keep":100}'

生产建议挂 cron · 每周日凌晨 3 点跑一次 {"max_keep": 500}，配 Slack webhook 报告战果。

面试题 · M.1

"你的 agent 长期 memory 怎么避免变垃圾？"
L4 答："加 TTL 过期"——只解决过时，不解决低价值，掉档。
L5 答："写入时去重 + 定期 LRU 清理"——比 L4 好，但 LRU 让新 fact 永远输给旧的。
L6 答：分写入和读取两条路径。写入用 cosine similarity 去近重；读取每次 bump last_used_at + use_count；定期 prune 用 0.7×exp(-age/half_life) + 0.3×log1p(use_count)/log1p(max) 复合分。半衰期按业务调（用户偏好 14 天、技术文档 90 天）。dry_run 必须支持。永远不一次删超过 50%。

Chapter 3 · 并发

Parallel tool exec · 三行代码 3x 加速

改对一行 · 比换模型省钱多了

Harness 主循环（backend/agent/harness/loop.py）以前长这样： 

for tc in tool_calls:
    content = await asyncio.to_thread(safe_run_tool, tool, tc["args"])
    messages.append(ToolMessage(content=content, ...))

看上去没毛病——asyncio.to_thread 把 sync tool 扔到线程跑，主 event loop 不阻塞。 但这个循环是顺序的。同一轮 LLM 一次发了 3 个 tool call（典型场景：web_search("X") + web_search("Y") + read_file("Z")），它们一个接一个跑。 每个 web_search 1.5 秒，3 个就是 4.5 秒。LLM 干等。

§3.1原理 · 为什么 I/O 密集型并行收益巨大

对 web_search、file_ops、MCP-client tool 这种等网络 / 等磁盘的工具，CPU 99% 时间空着。 线程模型（asyncio + thread pool）下，N 个并发任务的总时长 ≈ 单个最慢的那个，不是 N 倍。

等的时间

§3.2实现 · 三行 diff

# 旧 · 顺序 for-loop
for tc in tool_calls:
    content = await asyncio.to_thread(safe_run_tool, tool, tc["args"])
    messages.append(...)

# 新 · gather 并发
async def _run_one(tc):
    name = tc["name"]
    tool = tools_by_name.get(name)
    if tool is None:
        return tc, f"[error] unknown tool {name!r}"
    content = await asyncio.to_thread(safe_run_tool, tool, tc.get("args", {}))
    return tc, content

results = await asyncio.gather(*[_run_one(tc) for tc in tool_calls])

# 关键 · 按原顺序 append · 不按完成顺序
for tc, content in results:
    messages.append(ToolMessage(content=content, name=tc["name"],
                                tool_call_id=tc["id"]))
细节 · 为什么"按原顺序 append" 那么重要

三个 tool 完成时间不同 —— 如果你按完成顺序 append，每次同一组 tool call 拼出来的 message 序列就不一样。 这会破坏 prompt cache（Anthropic / OpenAI / Gemini 都按 prefix hash 命中缓存），让你白白多付钱。 asyncio.gather 返回值本身就是按入参顺序，所以我们用 (tc, content) 元组配对，遍历时就是确定性序列。

§3.3关于 GIL · senior 必懂的细节

有人会问：Python 不是有 GIL 吗，开多线程不也是假的并行？

答案：I/O 释放 GIL，CPU 不释放。 requests.get()、open(f).read()、socket 读写、Chroma 查 embedding —— 这些进入 C 层的 I/O 调用会主动释放 GIL，让别的线程跑。 所以 N 个 web_search 真能 N 倍并发。

如果是纯 Python CPU 任务（比如 numpy 不参与的循环），GIL 确实让多线程退化成串行。这种情况要用 multiprocessing 或者 asyncio.to_process_pool。 Agent 里的 tool 99% 是 I/O，所以 thread pool 是合适的选择。

§3.4提示 LLM "可以并发" · 改 docstring

模型不知道"我们的 harness 能并发"——它只看 tool docstring。 所以我们在 dispatch_subagent 的 docstring 里加了： 

"""...
PARALLELISM · IMPORTANT.
The harness runs all your tool calls in a single turn CONCURRENTLY, so
if you have 3 independent research questions, emit 3 dispatch_subagent
calls in the SAME assistant turn · they will run side-by-side instead
of sequentially.  Don't fan them out one per turn.
..."""

效果：模型学会一次发多个 call。一个真实 query 例子： 

"对比 LangGraph 0.6 / FlashAttention 3 / vLLM 最新进展，每个三句话总结"

# 旧 · 一轮一个 sub-agent · 总耗时 = 3 × (research + LLM)
# 新 · 一轮 3 个 sub-agent 并发 · 总耗时 ≈ 1 × (research + LLM)

§3.5取舍 · 什么时候不要并发

  • tool 之间有数据依赖：先 list_files() 再 read_file(picked)，模型已经知道分两轮发，不会混在一轮。OK。
  • tool 写共享状态：5 个 remember() 同时写 Chroma 可能竞争 SQLite 锁。 我们的 safe_run_tool 用 ThreadPoolExecutor(max_workers=8)，并发度有上限，问题不大；但如果 tool 是非线程安全的（rare），要么改 tool 要么按名字 group 串行执行。
  • 限流敏感：5 个 web_search() 同时打 DDG 容易 429。我们 web_search 已经有 retry-with-backoff，自动处理。但如果你接的 API 有 hard QPS，要在 tool 内部加 semaphore。
面试题 · P.1

"agent 一轮发多个 tool · 怎么并发？"
关键点：(1) asyncio.gather 而不是 for-loop；(2) 每个 tool 用 asyncio.to_thread 进线程池（兼容 sync tool）；(3) 结果按入参顺序回写保证 prompt cache 命中；(4) GIL 在 I/O 时释放所以真并发；(5) 在 tool docstring 里教 LLM "可以一轮多发"。能讲到第 3 点 = senior 信号。

Chapter 4 · 预算护栏

Cost guardrail · fail-fast 之美

烧光额度才发现 = 一次 PR 直接被 P0 召回

有了 demo 后第一周大家随便玩，第二周收到账单 $500，老板黑脸。这是 agent 项目的典型黑天鹅。 防御方式分三层：监控（看到）、告警（被通知）、护栏（直接被拒）。 前两个我们之前已有（Prometheus + cost_per_session 在 /usage 里）。这次补的是第三层。

§4.1原理 · "状态共享" 的设计美感

我们已经有 UsageCallback 在每次 LLM 调用后更新一个全局 _per_session 字典： 

# observability/usage.py
_per_session: dict[str, dict[str, int]] = defaultdict(lambda: dict(_ZERO))

def add(usage_metadata, session_id):
    with _lock:
        _add_locked(_global, ...)
        if session_id:
            _add_locked(_per_session[session_id], ...)

关键洞察：guardrail 不需要新数据源。它只需要在请求进来的时候、在 LLM 跑之前， 读一下 _per_session[sid] 跟预算比一比。这就是"single source of truth" 的实践 —— 监控、告警、护栏共享同一份状态。

§4.2实现 · FastAPI dependency · 不是 middleware

async def enforce_session_budget(payload_session_id: str | None = None) -> None:
    if not payload_session_id:
        return  # 新 session 还没花钱
    from .observability import usage as usage_tracker
    snap = usage_tracker.snapshot(payload_session_id)
    if snap.get("over_budget"):
        raise HTTPException(
            status_code=402,
            detail={
                "error": "session budget exhausted",
                "session_id": payload_session_id,
                "spent_usd": snap["session"]["cost_usd"],
                "budget_usd": snap["budget_usd"],
                "hint": "Start a new session_id, raise AGENT_SESSION_BUDGET_USD, or =0 to disable.",
            },
        )

# app.py · 在 /chat 里手动调（dependency 链限制 · 见 §4.3）
@app.post("/chat", dependencies=[Depends(require_api_key)])
async def chat(request: Request, payload: ChatIn):
    await enforce_session_budget(payload.session_id)
    ...
细节 · 为什么是 dependency 而不是 middleware

Middleware 在路由解析之前跑，拿不到 parsed body（payload.session_id）。 要么手动 await request.json() 然后消费掉 body stream（导致后面 endpoint 拿不到 body），要么 buffer 后回灌（复杂）。 FastAPI dependency 在路由解析之后跑，直接拿到 typed payload，最干净。

代价：dependency 不能跨 endpoint 自动应用（middleware 可以），所以每个 chat-like 端点要手动调一次。我们目前只有 /chat 和 /chat/v2，手动 OK。如果有 10 个就考虑写一个 router-level dependency。

§4.3状态码选 402 不是 429

code语义什么时候用
429 Too Many Requests调用频率太高 · 等会儿再试rate limit · 请求来太快
402 Payment Required需要付费 · 等不会有用budget exhausted · 钱花完了
403 Forbidden无权访问 · 跟身份相关auth 失败 · 403 不是 budget 场景

402 是 HTTP 标准里专门留给"钱"的状态码，2025 年开始因为 Stripe 和 Anthropic API 用它代表 quota 问题，业界终于在 agent 场景达成共识。用 402 客户端能立刻分辨"是付费问题"而不是"是 rate limit 等等就好"。

§4.4护栏的执行顺序 · auth → rate-limit → budget

如果把 budget 放在 auth 之前，还没认证就告诉攻击者"这个 session 钱花完了"——信息泄露。 如果把 rate-limit 放在 budget 之后，恶意攻击会先消耗你的 LLM 调用配额（因为 budget 检查需要 lookup 状态，本身比 IP 计数贵）。 顺序 = 检查成本递增 是这类设计的通用原则。

§4.5本地模型 = 0 成本 · 怎么处理

Ollama / LM Studio / vLLM 跑本地模型，token 不计费。我们 usage.py 用前缀匹配判断： 

_LOCAL_PREFIXES = ("ollama:", "lmstudio:", "local:", "llamacpp:", "mlx:")
def _is_local(model):
    return any(model.startswith(p) for p in _LOCAL_PREFIXES)

local model 永远 cost_usd=0，永远不 over_budget，guardrail 自动失效。 这是"feature 默认无害"的实践 —— 用本地模型的 dev 不会被一个他没碰过的预算护栏卡住。

面试题 · C.1

"agent 服务怎么防止单 session 烧爆 LLM 预算？"
关键点：(1) 共享同一个 usage 状态 · 监控/告警/护栏不要各开各的 metric；(2) 用 FastAPI dependency 而不是 middleware，因为要看 parsed body；(3) 状态码选 402，让客户端能分辨是 quota 不是 rate limit；(4) 检查顺序 auth → rate-limit → budget → LLM，按"检查成本递增"排；(5) local model 自动 zero-cost，护栏对 dev 透明。能讲到第 4 点 = 系统设计 senior 信号。

Chapter 5 · 契约

OpenAPI codegen · 让契约会自己骂人

后端改字段 · 前端编译就报红 · 不用等 QA

前后端分离的最大失败模式：静默漂移（silent drift）。后端 PM 上去把 session_id 改成 sessionId，前端 PR 没人提 review，CI 没炸（因为前端不知道后端改了），上线之后用户发 message 直接 422，回滚。 类似的故事每个有真实用户的 service 都有过 1 次以上。

§5.14 种解法 · 谁更稳

方案原理稳定度
共识 + reviewer 把关每次后端改 API · 找前端 review★ · 累、容易漏
手写 contract teste2e 测每个端点★★ · 跑得慢、easily out of date
Protobuf / gRPC共享 schema · 各语言生成 client★★★★ · 但要换协议
OpenAPI codegenFastAPI 自动生成 OpenAPI · 前端 codegen TypeScript★★★★ · 工作量小

FastAPI 因为用了 Pydantic，"自动生成 OpenAPI" 本来就是免费功能（访问 /openapi.json 就有）。 我们要做的是把那个 schema 喂进 openapi-typescript，让前端 TypeScript 类型跟后端 Pydantic 同源。

§5.2实现 · 一个 npm script

// frontend/package.json
"scripts": {
  "gen:api": "cd ../backend && uv run python -c \"import json; from app import app; open('../frontend/openapi.json','w').write(json.dumps(app.openapi(), indent=2))\" && openapi-typescript ./openapi.json -o ./src/api/schema.gen.ts && echo '✓ schema.gen.ts updated'"
}

这一行干了三件事：

  1. 用 backend 的 venv 跑一行 Python，把 FastAPI 的 app.openapi() dump 到 frontend/openapi.json
  2. 调 openapi-typescript 把它转成 1100 行的 schema.gen.ts
  3. echo 提示完成

§5.4真正的收益 · 三个例子

例 1 · 后端改字段名

后端 PM 把 ChatIn.session_id 改成 ChatIn.sessionId。

  1. 跑 npm run gen:api，schema.gen.ts 立刻更新
  2. 前端所有用 payload.session_id 的地方编译失败
  3. 修过来 = 字段名重命名安全

例 2 · 后端加新端点

  1. 跑 npm run gen:api
  2. Schemas["PruneIn"] 自动出现，TypeScript 智能提示有 max_keep / drop_fraction 等字段
  3. 不需要查文档

例 3 · 后端改字段类型

后端把 budget_usd 从 float 改成 Decimal。

  1. OpenAPI 里类型从 number 变成 string（JSON 不支持 Decimal）
  2. 前端拿 budget_usd 当数字用的地方编译失败 · 提醒你加 Number(...) 或者 BigDecimal lib
细节 · response 类型为什么我们没自动生成

OpenAPI codegen 只在后端 endpoint 用 response_model=... 注解时才能精确出 response 类型。 我们大多数 endpoint 返回 raw dict（图省事），所以 schema.gen.ts 里 response 类型是 unknown / 含糊的 union。 短期：response 类型继续手写，注释里标明跟哪个端点对齐。长期：给关键 endpoint 加 response_model=，下一次 codegen 就自动收紧。

§5.5什么时候不要 codegen

  • API 极不稳定 · 每天改几次 schema · 生成的文件 review 比手写还累
  • 多 client 用不同 schema 子集 · codegen 一刀切的全量类型反而碍事
  • 已经有 BFF（backend-for-frontend）层做翻译 · 那一层负责契约
面试题 · O.1

"前后端分离怎么避免 API 字段对不上的 silent drift？"
L4："共识 + 联调"——掉档。
L5："写 e2e contract test" —— OK 但慢。
L6：用schema-driven开发。后端用 Pydantic / Protobuf 定义 schema，自动生成 OpenAPI / .proto；前端 codegen 出 TypeScript 类型；任何字段变更触发前端编译失败。Trade-off：codegen 文件不要手改、要进 git；不稳的 API 早期可能不适合（每次重生 review 累）；response 类型要后端写 response_model= 才能精确。

Chapter 6 · 会话

Multi-session UI · "切会话" 是个错觉

HTTP 是 stateless · 真正的状态在 checkpoint 里 · UI 只是过滤器

加完前面几件，最缺的就是用户感知 —— "我昨天聊到哪了？"。 Sidebar 加一个会话列表，看上去是个 UI 改动。但理解它在 stateless HTTP 下到底意味着什么，是面试题里被低估的考点。

§6.1三层"会话"概念 · 别混

层是什么谁管
HTTP sessioncookie / token 标识身份auth 层（我们没用，用 X-API-Key）
Conversation session一段连续对话 · 有 session_idbackend checkpoint（LangGraph SQLite）
UI session用户在前端"打开的对话"frontend state · 只是个 filter

关键洞察：backend 永远不知道"用户切了 session"。 backend 只知道每个请求带的 session_id 是什么。如果两个请求带同一个 sid，它们共享 checkpoint；如果带不同 sid，就是两条独立对话。

所以"切 session" 在 frontend 就是把 React state 里的 sessionId 字段换一个值。下一次 /chat 带新 sid，backend 自动从 checkpointer 里拉对应的 message history。

概念 · stateless backend + stateful storage

这是 web service 的黄金范式。HTTP handler 本身不持有任何 in-memory user state（可以随时重启、scale-out 加 replica）， 所有状态外置到 SQLite / Redis / S3。frontend 持有的不是"状态"，是"指向状态的指针"（session_id）。

切换 UI session = 换个指针 = backend 从存储拉新的状态。这跟 Vim buffer / 浏览器 tab / IDE workspace 都是同一个 pattern。

§6.2列表数据从哪来

Backend 早就有 /chat/replay/sessions endpoint —— 扫 data/traces.jsonl，按 session_id group，每个返回 (sid, first_user_message, last_ts)： 

@app.get("/chat/replay/sessions")
def list_replayable_sessions(limit: int = 50) -> list[dict]:
    path = get_settings().trace_file
    seen: dict[str, dict] = {}
    for line in path.read_text().splitlines():
        evt = json.loads(line)
        sid = evt.get("session_id") or (evt.get("payload") or {}).get("session_id")
        if not sid: continue
        info = seen.setdefault(sid, {"session_id": sid, "first_user": None, "last_ts": 0})
        info["last_ts"] = max(info["last_ts"], float(evt.get("ts", 0)))
        if info["first_user"] is None:
            txt = _trace_payload_text(evt.get("payload"))
            if txt:
                info["first_user"] = txt[:160]
    return sorted(seen.values(), key=lambda x: x["last_ts"], reverse=True)[:limit]

Trace JSONL 是真理 —— 因为所有事件都通过 tracer.emit() 写进去，不会有"漏掉一条 session" 的可能。 没有专门的"sessions table"，避免了双写不一致。

§6.3前端 · 4 行 state · 1 个 effect

// Sidebar.tsx
const [sessions, setSessions] = useState<SessionSummary[]>([]);

useEffect(() => {
    listSessions(50).then(setSessions);
}, [refreshTick]);              // 每次 turn 完都刷新

// 用户点哪一行 → 把 sessionId 换成它
<button onClick={() => onSwitchSession(s.session_id)}>...</button>

// + 新对话按钮 = 把 sessionId 设成 null · 下一条消息后端会生成新 uuid
<button onClick={() => onSwitchSession(null)}>+ new</button>

App.tsx 持有 sessionId state，onSwitchSession 同时清空 trace 面板（旧事件属于旧 session，留着会误导）。就这么简单。

§6.4三个非显然的 UX 决策

  • "+ 新对话" 不是"清空当前"。两者语义不同 —— 新对话是开启新 session_id（旧会话保留在历史），清空当前是删除当前 session 的 messages（破坏性）。我们只提供前者，避免误删。
  • 列表用第一句用户消息当标题，不是"对话 #14"。原因：人类靠内容回忆，不靠序号。这跟 Cursor / ChatGPT / Claude 的 sidebar 设计一致。
  • active 标记用红色 badge，跟 Ghibli 主色一致。点别的会话不需要确认弹窗 —— 因为切换是非破坏的，犯错代价低。

§6.5留给以后的坑

  • Trace 文件会涨。每个 turn 写 30+ 事件，每条事件几百字节，1000 个 session 后 trace.jsonl 会到 10MB 级。需要做 rotation（按周分文件）+ 一个 sessions index（专门一行/session 的总结表，省扫描）。
  • Multi-user 隔离没做。当前所有 user 共享同一个 trace 文件。生产前必须给每个 trace 事件加 user_id，按 user filter。
  • 没有"删除会话"。需要的话写一个 DELETE /chat/sessions/{sid} · 同时清 trace 行 + checkpoint 行 + harness store 文件。GDPR 场景必须有。
面试题 · S.1

"怎么实现 ChatGPT 那种 sidebar 历史会话？"
关键点：(1) backend stateless，状态全在 checkpointer / 存储 · session_id 是指针；(2) 列表数据来源不要双写 · 从 trace 或 checkpoint 派生；(3) "切" session 在前端就是改 React state，下一个请求带新 sid 即可；(4) "+ 新对话" 跟 "清空当前" 语义不同 · 不要混；(5) multi-user 必须在 trace 事件里加 user_id，否则数据串号。

Chapter 7 · 哲学

5 件事的共同主题 · senior 工程的 DNA

把每个改动单独看是 feature · 一起看是姿态

回头看这 5 个 P1 改动，背后其实是同一套工程姿态。把它说清楚，比把任何单个 feature 讲清楚都重要 —— 因为姿态可以迁移，feature 不行。

§7.1三条贯穿的原则

原则 1 · 默认安全 · 显式开放

每个新能力都默认不会破坏老用法：

  • Memory decay 默认不自动 prune · 必须手动调端点
  • Parallel tool exec 默认所有 tool 都安全（safe_run_tool 已在线程池）
  • Cost guardrail 默认 budget=0（不限）· 本地 Ollama 永远 zero-cost
  • OpenAPI codegen 默认手写类型不动 · 你想用就用 Schemas["..."]
  • Multi-session UI 默认显示当前 session · 切换是用户主动行为

反过来：每个能力都显式提供"开到底"的开关：MCP_EXPOSED_TOOLS=*、AGENT_SESSION_BUDGET_USD=10、npm run gen:api。 SRE 圈这叫"safe defaults, easy escalation"。

原则 2 · 状态共享 · 不开新孔

新功能尽量复用已有数据源：

  • Cost guardrail 复用 UsageCallback 的 _per_session 状态 · 不另开计数
  • Multi-session UI 复用 traces.jsonl · 不专门写 sessions table
  • Memory decay 复用 Chroma metadata · 不另开计数表

这叫"single source of truth"。每多一个数据源就多一处可能不一致的地方。

原则 3 · Fail fast · 在最便宜的层失败

auth → rate-limit → budget → LLM 顺序不是随便排的 —— 是按"检查成本递增"。 dry_run 必须在 prune 之前提供。tsc 编译失败 必须在 runtime 之前发生。

每一层都问自己一个问题："这个错误能不能在更便宜的地方发现？" 能 = 把检查往前挪一层。这是 senior 工程师的肌肉记忆。

Memory · M.1 · 长期 memory 怎么避免变垃圾？

见 §2.5 末尾。

Memory · M.2 · 为什么 prune 不能用纯 LRU？

纯 LRU = 按 last_used_at 排序砍最老的。问题：刚写入的新fact use_count 也是 0、last_used_at 跟时间戳一样新；老的高价值 fact（被反复 recall）use_count 高但 last_used_at 也新。纯 LRU 没法区分"新但没用过" 和"老但常用"。 正解：复合分把 recency 和 usage 都算进去（recency 主导，usage 调味）。

Parallel · P.1 · agent 一轮发多个 tool · 怎么并发？

见 §3.5 末尾。

Parallel · P.2 · Python GIL 不是会让多线程退化成串行吗？

只对纯 Python CPU 任务退化。I/O 调用（socket / 文件 / C extension）会主动释放 GIL，让别的线程跑。 Agent tool 99% 都是 I/O，所以 thread pool 真并发。CPU 重活（嵌入计算、解码）走 numpy/torch 时也会释放 GIL（这些库内部跳出 Python 解释器）。 真正中招的是纯 Python 循环（做 JSON 反序列化大文件之类），那种要么换 multiprocessing 要么换 PyO3 / Cython。

Cost · C.1 · 怎么防 LLM 单 session 烧爆？

见 §4.5 末尾。

Cost · C.2 · 为什么用 dependency 而不是 middleware？

Middleware 在路由解析之前跑，拿不到 parsed body（payload.session_id），要么手动 read body stream（破坏后续 endpoint），要么 buffer + 回灌（复杂易错）。 FastAPI dependency 在路由解析之后跑，直接收到 typed payload，最干净。代价：dependency 不能跨 endpoint 自动应用，需要每个端点手动挂；endpoint 数量多了考虑 router-level dependency。

OpenAPI · O.1 · 前后端怎么避免 silent drift？

见 §5.5 末尾。

OpenAPI · O.2 · response 类型为什么 codegen 出来不准？

FastAPI 只在 endpoint 显式标 response_model=SomeModel 时才把 response schema 写进 OpenAPI。 返回 raw dict 的端点 OpenAPI 里 response 是 {}（任意 object），codegen 出来就是 unknown / 含糊 union。 修法：给关键 endpoint 加 response_model=。短期权宜：response 类型继续手写，注释里标对齐哪个端点。

Session · S.1 · 怎么实现 ChatGPT 风格 sidebar 历史会话？

见 §6.5 末尾。

Session · S.2 · session_id 应该谁生成 · 前端还是后端？

后端生成 + 前端缓存。前端第一次 POST /chat 不带 session_id，后端 uuid.uuid4() 生成并通过 SSE first event 返回，前端拿到后续请求都带它。 不要前端生成原因：(1) uuid 算法不一致、碰撞概率难证；(2) 前端 bug 可能让两个 tab 用同一个 sid 串号；(3) 后端可以加签名/校验防伪。

综合 · X.1 · 这 5 个改动有什么共同点？

见第 7 章。三条原则：默认安全显式开放、状态共享不开新孔、fail-fast 在最便宜层失败。

综合 · X.2 · 如果只能挑 2 个先做，挑哪 2 个？为什么？

Cost guardrail + Memory decay。原因：

  • Cost guardrail 防的是"突发性灾难"——一次脚本失误 burn 几百刀，没护栏没救。
  • Memory decay 防的是"慢性病"——每天烂一点，三个月后 memory 完全无用。

Parallel exec 是性能优化、OpenAPI / Multi-session UI 是开发体验，可以晚一周。两类："会死人"先做、"会变烂" 紧跟、"会慢" 和"会累"放后面。

Chapter 8 · Cost & Latency Engineering

5 个独立 case study · token 预算 / prompt 压缩 / semantic cache / model routing / streaming UX

"agent 太贵 / 太慢" 是 senior 面 / 1:1 / 季度 OKR 永恒的话题。这一章不讲理论，5 个 case study 每个 30 行讲完：背景 → 优化 → 数字 → 反思

§8.1Case 1 · Token 预算切分 · 把 32K ctx 砍到 8K 不掉精度

背景：客服 agent · 每次喂 system prompt (4K) + 完整对话 (10–28K) + tool descriptions (3K) ≈ 17–35K token。月 token bill $42K。

优化 4 招：

  1. Tool descriptions 按需注入——系统 prompt 里只列 tool 名 + 一行 summary（300 token）· 模型决定调哪个时再注入完整 schema。砍 2.7K/turn。
  2. 对话窗口——只保留最近 10 turn · 早期 turn 用 LLM summarize 成 200-token 摘要塞 system。砍平均 12K → 4K。
  3. System prompt 拆 static + dynamic——80% 内容（角色 / 准则）走 prompt cache（OpenAI 50% / Anthropic 90% 折扣）· 20% 动态（用户名 / 时区）每次重发。
  4. Tokenize-aware 截断——别在字符层截 · 用 tiktoken 算到 token 边界 · 不会半截字 corrupt 后面。

结果：平均 ctx 17K → 6.5K · accuracy holdout -0.3%（统计内噪声）· 月 token bill $42K → $14K。

反思：tool description 按需注入有1 turn 延迟——第一次调用某 tool 时才下发 schema · 模型有时第一次输出会 schema 错。修法：summary 里多带 1 行参数样例。

§8.2Case 2 · Prompt 压缩 · LLMLingua 把指令砍 50%

背景：长 system prompt（4.2K token）· few-shot 5 个 example (每个 600 token) · 总指令 ~7K。

优化：

  • 用 LLMLingua-2（小 BERT 模型）给每个 token 打"重要度"分 · 删低分 token · 保留 50%。
  • 关键技巧：不动 instruction · 只压 example 和 background description。
  • fallback：跑 holdout 100 题 · 任何一题 accuracy drop >5% 就 disable 这一条。

结果：7K → 3.6K · 整体 accuracy -1.2% (可接受) · cost / latency 各砍 ~40%。

反思：压缩的 prompt 对 prompt cache 是 hit miss——压缩本身改了字节 · 整段 cache 失效。所以只在从来没 cache hit 的 prompt（动态生成）上做。静态 system prompt 优先走 cache 不要压。

§8.3Case 3 · Semantic cache · Q 重复率 30% 的免费午餐

背景：FAQ-style 客服 · 用户每周问"how do I refund?" 这种 50 次。Exact match cache hit 仅 8%（用户问法太多变）。

优化 (semantic cache)：

  1. 每条 query embed (small model, e.g. text-embedding-3-small) · 存 (embedding, response, ttl=24h) 进 Redis vector index。
  2. 新 query 来 · embed · top-1 检索 · cosine similarity > 0.92 直接返回缓存 response · > 0.85 + 同 user 直接返回 · 否则走完整 LLM。
  3. 关键防坑：threshold 必须按 domain 校准 · 通用 0.92 在医疗 / 法律必须升到 0.97（差 1 个词意义就完全不同）。

结果：cache hit 8% → 38% · LLM call -32% · embedding cost +1.2% · 月成本 -30%。

反思：semantic cache 跟 personalization 冲突——同一个 "refund 怎么操作" 但用户 A 是 EU (GDPR 流程不同) 用户 B 是 US。修法：cache key 加 user_attributes hash。或者 semantic cache 只用在明确无个性化的 query 子集。

§8.4Case 4 · Model routing · Haiku 做 80% · Sonnet 兜底

背景：所有请求走 Claude 3.5 Sonnet · 月 $58K。但 abuse-tracking 数据显示 70% query 是简单分类 / 路由 / 提取等任务。

优化 (cascading model routing)：

  1. 第一步走 Claude Haiku（便宜 12 倍）· 加 system prompt: "回答完后输出 confidence 0–1"。
  2. confidence ≥ 0.8 直接返回（占 70%）。
  3. confidence < 0.8 · 调用 Sonnet 重做（占 30%）。
  4. 规则护栏：detect 到 "tool calls / 多步推理 / 长 response" 直接 skip Haiku 走 Sonnet（避免低质量 retry 反而更慢）。

结果：月 cost $58K → $19K · P95 latency -22%（Haiku 更快）· accuracy hold-out -0.8%。

反思：① 模型自报 confidence 有 length / vocab bias · 几个月后漂移 · 需要 quarterly recalibrate；② 不要把 router 跟 worker 混在一段 prompt · 应该显式两次 LLM call 才能监控分别成本 / latency。

§8.5Case 5 · Streaming UX · "感知" P50 从 4s → 0.3s

背景：聊天 UI · LLM TTFT (Time-To-First-Token) 大概 1.5s · 全部输出 4–8s · 用户看到 spinner 4 秒一直点 retry。

优化 4 个 UX 招：

  • SSE / WebSocket 流式渲染 token——感知首字 ~300ms。
  • 第一秒 LLM 还没出 · UI 显示 tool call placeholder（"Searching docs..."）· 让用户知道在动。
  • tool call 阶段 stream tool name · 不只 stream response token——长 tool call 用户能 cancel。
  • 客户端做 optimistic echo——用户敲 enter 那一刻就把 user msg 渲染上去 · 不等 server 200。

结果：客观 latency 没变（still 4–8s 总输出）· 感知 P50 从 4s 跌到 0.3s · NPS +12 · "agent 太慢" 投诉 -73%。

反思：流式 + cancel 需要后端能 abort 中途 · 一定要在 LLM client 设 timeout + 把 cancellation 传到 LLM API（OpenAI 用 abort signal · Anthropic SDK 用 stream context manager）· 否则成本继续烧。

5 个 case 共同的方法论
  1. 先量化：每个 case 都是从 "$X / 月 + Pn / latency" 出发的——不要"感觉慢"就动手。
  2. 分层不耦合：5 招互不依赖 · 任意停一招产品仍跑——最大化可逆性。
  3. cache vs 压缩 是死敌：压缩破坏 prefix · 全栈一定要先选定哪个优先（一般 cache 优先）。
  4. 有 holdout 才有 license：accuracy 不可见，任何"省钱"优化都会被怀疑——先建评估闭环。
  5. UX latency > engine latency：用户不在意 token/s · 只在意"agent 在不在动"。
