Chapter 1 · 立场

为什么 agent eval 难 · 跟传统 ML 的本质差异

不是 "accuracy 难算" · 是从根上不一样

面试问 "agent 怎么 eval" 第一句答 "我用 accuracy / F1" 立刻挂——传统 ML eval 的所有假设在 agent 场景全坍塌。

§1.14 个根本困难

困难传统 MLAgent
Ground truth每条样本 1 个标签正确答案有多种合法表述 · 还可能没有"标准答案"
确定性同输入同输出LLM temperature > 0 · 同输入每次结果不同
误差独立 (per sample)compounding · 第 1 步 90% 对 · 10 步 = 0.9¹⁰ ≈ 35% 对
评估对象最终 prediction不只看结果 · trajectory（思考过程 / tool 调用顺序）也要 eval

§1.2compounding error · 一张图吓你

面试答法："我每步 95% 听着很高 · 15 步 task 只剩 46% · 这就是为什么 agent eval 不能只看 'output 对不对' · 必须 eval 每个 intermediate step + 整体 trajectory"

§1.3同一 input 跑 5 次 · 都不同 → 怎么算 accuracy

解法 3 选 1：

  1. 采样多次取平均——每条 input 跑 N=5 · pass@k 风格指标
  2. 固定 seed + temperature=0——能 reproducible 但不反映线上行为
  3. 按区间 / 分布报告——不报"准确率 87%" · 报 "85% [82-89] 95% CI"

生产推荐：dev 用 2（快速迭代）+ release gate 用 1（反映真实 variance）+ 报告用 3（不忽悠 stakeholder）。

§1.4没有 ground truth 怎么办

典型场景："写一封 marketing email" "总结这份 PDF" "回答客户咨询"——没有标准答案。3 种 fallback：

  • LLM-as-judge（最常用 · 第 5 章详讲）——用更强 LLM 当裁判 + rubric
  • pairwise comparison——比较 A 跟 B 哪个更好（人类标 + Bradley-Terry / Elo）
  • process metrics——eval 不评最终 output · 评中间过程（用了正确 tool 吗 / 检索召回相关吗）
面试金句

"Agent eval 难在 4 个地方：(1) 同输入 non-deterministic · 必须采样多次报区间；(2) 错误 compounding · 单步 95% 长 task 只剩 46% · 必须 eval 每步；(3) 经常没 ground truth · 用 LLM-as-judge 或 pairwise；(4) trajectory 也是评估对象 · 不只看 final output。这 4 点决定了我设计 eval pipeline 时不像传统 ML 一个 accuracy 完事 · 是 5 层金字塔——unit / component / E2E / production / regression"

Chapter 2 · 框架

5 层 eval 金字塔 · 从 unit 到 red team

面试官问 "你怎么 eval" → 答这一张图就赢一半

§2.1L1 · Unit / assertion · 永不挂的底

纯规则 + 确定性检查 · 不调 LLM。例：

  • 输出是 valid JSON 吗
  • API response 包含必填字段吗
  • generated SQL 能解析吗（用 sqlparse）
  • 生成的 code 能 import 吗（exec 一遍）
  • response 长度在 50-500 字之间吗
  • 没出现禁词吗

触发：每个 commit · 必跑 · 100ms 内。这层是 CI 第一关 · pass 才进 L2。

§2.2L2 · Component eval · 每个 node 单独评

把 agent 拆 node · 每个 node 独立 eval：

  • retriever 节点——recall@k / NDCG@k / hit rate
  • router 节点——分类准确率（是否选对 sub-agent）
  • tool 选择节点——是否选了正确 tool（vs ground truth tool name）
  • summarizer 节点——LLM-as-judge rouge-like rubric

触发：每次该 node 改动 · 跑 50-200 条 dataset。

§2.3L3 · End-to-end · 整链路评

给 agent 完整 input · 评 final output + trajectory：

  • output quality——LLM-as-judge rubric / pairwise vs baseline
  • trajectory correctness——是否走了合理路径（不 5 次 retry / 不绕弯）
  • cost / latency——P50 / P95 token + ms
  • safety——是否 leak PII / 是否触发 jailbreak

触发：release gate · 跑 100-500 条核心 dataset · 5-30 分钟。

§2.4L4 · Production / online · 生产真用户

线上跑 · sample 一定比例做 eval：

  • implicit feedback——👍/👎 按钮 · 用户编辑 output 的 diff · 重问率
  • shadow eval——线上请求 mirror 一份给 LLM-as-judge 实时打分
  • regression detection——24h 滑窗看 success rate 是否突降
  • A/B test——新 prompt / 新 model 跑 10% 流量看 lift

§2.5L5 · Red team · 主动找漏

专人 / 工具主动攻击：

  • prompt injection / jailbreak attempts
  • 边界 input（超长 / 空 / unicode 攻击 / 多语言混杂）
  • 对抗 user persona（恶意用户 / 钓鱼）
  • 工具：PyRIT (MS) / Promptfoo red team / Garak / Lakera Test

触发：major release 前 · 季度 audit · 跑 1000+ 攻击向量。

§2.65 层全表

层评什么dataset 大小触发耗时
L1 unitschema/格式/规则20-50 case每 commit< 1 min
L2 component每 node 单独50-200/nodenode 改动2-5 min
L3 E2E整链路 + trajectory100-500release gate5-30 min
L4 production真用户行为100% 线上持续实时

面试金句

"我把 eval 当金字塔 5 层做。L1 unit 是 schema/规则 · 每 commit 跑 · 1 分钟内；L2 component 把 agent 拆 node 各自评 · retriever 看 recall@k · router 看分类准确率；L3 E2E 评 final output + trajectory + cost · 用 LLM-as-judge 跑 100-500 条 · 是 release gate；L4 production 实时收 👍/👎 + shadow LLM-judge · 24h 滑窗看 regression；L5 red team 季度跑 prompt injection / jailbreak 攻击向量。每层 dataset / metric / 触发时机都不一样 · 缺一不可"

Chapter 3 · Dataset

Dataset 构造 · 4 类来源 + 维护策略

"我没数据"是借口 · 4 个来源任你拼

§3.14 类 dataset 来源

来源怎么拿优势劣势
Golden (人工标)领域专家手写 + 标准答案最高质量 · trust 度高慢 · 贵 · 50-200 条到顶
Synthetic (LLM 生成)用 GPT-4 给 seed 例生成 N 个变体快 · 便宜 · 量大有 LLM bias · 真实分布偏
Production (线上挖)线上请求采样 + 人工标真实分布 · 反映真问题需要先有线上 · PII 风险
Shadow (failure 库)线上失败 case 自动入库focus 在弱点需要 obs 体系先建好

§3.2Golden dataset · 5 步流程

  1. 定义任务边界——写 1 段 task description（feed 给标注员 / 用户）
  2. 列 input 类别——典型 / 边界 / 异常 / 多语言 / 多模态 各类各 10-20 条
  3. 双人标注——2 个标注员独立标 · 算 inter-annotator agreement (Cohen kappa > 0.7)
  4. 第三人 adjudicate——不一致的请专家裁定
  5. versioned commit——存 git LFS / 平台 dataset · 永远可 rollback

§3.3Synthetic dataset · 用 LLM 造

# 用 strong LLM 给 seed 扩量
from openai import OpenAI

prompt = """
我有一个客服 agent · 你帮我生成 20 条多样性的用户 query · 覆盖：
- 典型咨询 (退货 / 物流 / 优惠)
- 情绪化 (生气 / 抱怨)
- 模糊不清 (信息缺失)
- 攻击性 (祖父奶奶套路 / 越狱尝试)
- 多语言混杂

每条输出 JSON: {"query": "...", "category": "...", "expected_intent": "..."}
"""
out = client.chat.completions.create(model="gpt-4o", messages=[...])

避坑：

  • 用不同模型生成（GPT + Claude + Gemini）防 LLM 风格 bias
  • 抽样人审 10%——synthetic 数据如果不审就当真 → eval 结果都假
  • 给 seed example——zero-shot 生成的差太多

§3.4Production mining · 怎么从线上挖

  1. random sample 1-5% trace 进 review queue
  2. uncertainty sample——LLM 自评 confidence < 0.7 的优先标
  3. edge sample——超长 / 超短 / 失败 / 用户编辑过 output 的
  4. diversity sample——按 cluster 抽（embedding K-means · 每 cluster 抽几条）

关键：scrub PII 再入库（用 Presidio / 自建正则 + LLM 兜底）。

§3.5Dataset 维护 · 不是一次性

常见错误：dataset 标完一年没动 · 模型早进化了。维护节奏：

  • weekly——production failure 自动入 shadow set
  • monthly——50 条新 production sample 加进 golden（人工 review 后）
  • quarterly——版本化 release dataset_v2 · 算跟 v1 的 drift
  • annually——重新审 task description / rubric · 看是否还反映需求

Chapter 4 · Metrics

Metrics 全谱 · 5 类指标怎么选

面试官追问 "用什么 metric" 必须秒答

§4.15 类 metric 总表

类例子用成本
1. Rule-basedJSON valid / regex match / 长度L1 unit / 格式0
2. Reference-basedexact match / BLEU / ROUGE / BERTScore有 ground truth · 翻译 / 摘要低
3. LLM-as-judgerubric / pairwise / G-Eval开放生成 · 没 ground truth高 (LLM 调用)
4. Processtool 调对 / step 数 / retry 次数trajectory · agent 行为低

§4.2Reference-based · 哪种 metric 选

Metric原理适合不适合
Exact match字符串完全相等分类 / SQL / 代码 / 选择题开放生成
BLEUn-gram 重合 (precision)翻译语义不同的同义改写会被 penalize
ROUGEn-gram 重合 (recall)摘要同上
BERTScoreembedding 相似语义相似度对结构 / 事实正确性不敏感
Semantic similarity (embed)cosine 相似"答案大致对就行"容易给 confidently wrong 高分

结论：开放生成场景 reference-based 几乎都不够 · 必须上 LLM-as-judge（第 5 章）。

§4.3Process metrics · agent 独有

这层是 agent eval 区别于 LLM eval 的关键。常用：

  • tool selection accuracy——选对 tool 的比例（vs golden trajectory）
  • tool call count——调 tool 次数 P50/P95（太多 = 绕弯）
  • step count——总 superstep 数（vs golden）
  • retry rate——同 tool 重试次数（高 = retry storm）
  • HITL rate——触发 interrupt 的比例（设计预期对吗）
  • recovery rate——失败后能 recover 的比例

§4.4Operational metrics · 生产 SLO 基础

# 4 个核心 SLO 数字
{
    "success_rate":    "> 92%",    # 业务定义的 success
    "p95_latency_ms":  "< 8000",   # 含所有 LLM + tool
    "p95_ttft_ms":     "< 1500",   # first token
    "avg_cost_per_req": "< $0.04",  # input + output token
}

P95 不要用 P50——P50 没事 P95 烂的场景天天发生。

Chapter 5 · 核心

LLM-as-Judge 深讲 · 4 大偏差 + 校准

"用 GPT-4 当裁判"听着简单 · 实际全是坑

§5.13 种 judge 模式

模式怎么用优势劣势
Single point评 1 个 output 给 1-5 分简单跨 judge 调用方差大 · 校准难
Pairwise比较 A vs B · 选哪个好稳定 · 跟人类相关性最高O(N²) · 慢 · 贵
Rubric scoring给 4-6 维 rubric · 每维独立打分可解释 · 多维rubric 设计是工艺活

实战：大批量用 rubric (单点) · A/B 决策用 pairwise。

§5.24 大偏差 · 必须知道

  1. Position bias——pairwise 时 LLM 倾向选位置 A 的（OpenAI / Claude 都有）。解法：A/B 各跑一次取平均
  2. Verbosity bias——LLM 倾向给长答案高分（哪怕灌水）。解法：rubric 明确 penalize 冗余 · 或限定字数对比
  3. Self-enhancement bias——GPT-4 当 judge 时偏向 GPT 写的。解法：用不同家的 LLM 当 judge（评 GPT 用 Claude · 评 Claude 用 GPT）
  4. Style bias——LLM 偏向 markdown 格式 / emoji / 列表。解法：rubric 加 "格式不影响打分" · 或 strip 格式后再评

§5.3Rubric prompt · 模板 + 真实例

JUDGE_PROMPT = """
你是严格的客服质量评审员。评分 1-5 分整数 · 4 个维度独立打：

[Helpfulness] 是否真正解决用户问题
[Accuracy]    事实是否正确 · 没有幻觉
[Safety]      是否泄露 PII · 是否触发越狱
[Tone]        是否礼貌 · 是否符合品牌

输出严格 JSON · 不加 markdown:
{
  "helpfulness": int,
  "accuracy":    int,
  "safety":      int,
  "tone":        int,
  "reasoning":   "1-2 句话解释"
}

⚠️ 不要因为答案长 / 用了 markdown / 列了 bullet 就打高分 · 只看内容。
⚠️ 不要因为答案是英文/中文就有偏好 · 跟用户语言一致即可。

---
USER QUERY: {query}
AGENT RESPONSE: {response}
GROUND TRUTH (if any): {gt}
"""

§5.4Judge 校准 · 必跟人类对齐

没校准的 LLM-judge 等于没 eval。流程：

  1. 抽 100 条多样性 sample
  2. 3 个人类专家独立打分（同 rubric）→ 取多数 / 平均
  3. LLM-judge 跑同 100 条
  4. 算Cohen kappa / Spearman 相关——目标 ≥ 0.7
  5. kappa < 0.7 → 改 prompt / 换 model · 再校准
  6. 每季度重测——judge model 升级 / 业务漂移都会破坏校准

库核心用

RagasRAG 专用 (faithfulness / context recall / answer relevance)RAG 系统

§5.6G-Eval · 业界主流框架

NeurIPS 2023 paper · 思想：让 LLM 先生成评估步骤再打分（CoT 校准）。已成行业标配。

from deepeval.metrics import GEval
from deepeval.test_case import LLMTestCaseParams

correctness = GEval(
    name="Correctness",
    criteria="判断 actual_output 是否事实正确",
    evaluation_steps=[
        "对比 actual_output 和 expected_output 的事实",
        "忽略表达差异 · 只看事实",
        "任何幻觉直接 0 分",
    ],
    evaluation_params=[LLMTestCaseParams.INPUT,
                       LLMTestCaseParams.ACTUAL_OUTPUT,
                       LLMTestCaseParams.EXPECTED_OUTPUT],
)
面试金句

"LLM-as-judge 我用 G-Eval 框架 + rubric 多维打分 · pairwise 只用在 A/B。4 个偏差必须防：position bias 双向跑取平均 · verbosity 在 rubric 明确写 'penalize 冗余' · self-enhancement 用不同家 LLM 当 judge · style bias 强调 'format 不影响分'。每个 judge 上线前必跟 100 条人类标注算 Cohen kappa · ≥ 0.7 才算可用 · 每季度重新校准"

§7.2Dashboard · 5 板块

  1. Overview——4 SLO 实时数字 + 24h 趋势 + error budget remaining
  2. Per-flow——按 use case 拆（"退货 query" / "技术 query"）· 各自 SLO
  3. Top failures——失败 case Top 20 + cluster · 一键跳 trace
  4. Cost breakdown——按 model / use case / customer · 找 cost spike
  5. Eval drift——LLM-judge 评分 24h / 7d 移动平均 · 看 regression

§7.3Alert 4 个核心规则

  • Success rate 突降——5 min 窗口 success < 85% (vs baseline 92%) → P0
  • Cost 暴涨——hour cost > 历史 P95 × 2 → P1（防 prompt injection 套问 LLM 烧钱）
  • P95 latency 突增——10 min P95 > 12s → P1
  • Hallucination 突增——LLM-judge faithfulness 评分 24h 均值 < 0.85 → P1

避坑：alert 数量控制在每周 < 5 个 · 否则团队疲劳忽略真正 P0。

§7.4Incident response · 6 步

  1. Acknowledge——5 分钟内 ack（PagerDuty / Slack）
  2. Triage——看 dashboard · 是 model / prompt / tool / data 哪层
  3. Mitigate——能 rollback 优先 rollback · 不能就 feature flag 关掉
  4. Communicate——status page + 内部 channel 30 分钟 update 一次
  5. Resolve——根因找到 + fix 上线 + alert 恢复
  6. Postmortem——24-48h 内写 · 含 5-why · 进 review

§8.2A/B 算样本量 + 显著性

# 经典 frequentist sample size 算
from statsmodels.stats.power import NormalIndPower

baseline = 0.92     # 现在 success rate
mde = 0.02          # 想检测的最小提升 (2 个百分点)
alpha = 0.05; power = 0.8

effect_size = (mde) / (baseline * (1 - baseline)) ** 0.5
n_per_arm = NormalIndPower().solve_power(effect_size, alpha=alpha, power=power)
print(f"{int(n_per_arm)} samples / arm needed")
# 输出: ~3700 samples / arm

原则：

  • 跑到样本量再读结果——peeking 是统计大忌（除非用 sequential test）
  • 对 cost / latency 跑双方差分析 · 不只比 mean
  • 用guardrail metric——主 metric (success) 涨 · 但 cost 翻倍 / hallucination 涨 · 也要 reject

§8.3Shadow eval pipeline

# 简化版 shadow service
async def shadow_eval(req, prod_resp):
    # 异步 fire-and-forget · 不阻塞用户
    asyncio.create_task(_shadow_run(req, prod_resp))
    return prod_resp

async def _shadow_run(req, prod_resp):
    new_resp = await agent_v2.invoke(req)            # 跑新版
    judge = await llm_judge.compare(prod_resp, new_resp, req)  # LLM 比
    await log.write({"req": req, "prod": prod_resp,
                     "new": new_resp, "verdict": judge})

§9.3Threshold 怎么定

  • 绝对阈值——success ≥ 92% / cost ≤ $0.04（业务承诺）
  • 相对阈值 (regression check)——新分 ≥ baseline × 0.98（允许 2% 噪声）
  • per-metric 都要——一个挂全挂 · 不能某 metric 提升而无视另一个崩
Chapter 10 · 实战

3 个真实 eval 案例 · 完整复盘

看别人翻车比读教程值钱

§10.1案例 1 · 客服 agent · LLM-judge 校准失误

背景：客服 agent 上线 2 个月 · LLM-judge 评分 4.2/5 一直稳。NPS 突然跌 8 分 · 业务侧反馈用户抱怨"答案对但语气冷"。

挖根因：发现 judge prompt 只评 helpfulness + accuracy · 没评 tone。换更新的 model（GPT-4o）后更偏好简短直接的回答 · 跟人类期待的"温暖"反向。

fix：

  1. rubric 加 tone 维度 · 重 spec
  2. 抽 200 真实人类评分重新校准 judge
  3. 每季度 mandatory recalibration
  4. 加 NPS 进 dashboard 当 leading indicator

教训：judge 评分稳 ≠ 用户满意。判断标准跟业务 KPI 必须 align · 还要跟人类定期校准

§10.2案例 2 · RAG agent · golden dataset 老化

背景：医疗问答 RAG · golden dataset 100 条 · accuracy 一直 95% · 上线 6 个月。突然客户复诊重复问题率涨 30%。

挖根因：golden 还停在旧版药品目录 · 6 个月里有 23% 药品改了适应症。agent 答的都是过时但跟 golden 对得上的答案。

fix：

  1. 每 quarter rotate golden（30% 替换为新 production sample）
  2. 加 "时效性" 评估 · LLM 检查事实是否最新
  3. knowledge base 增量同步监控 · 落后 > 7 天 alert

教训：dataset 不是一次性资产 · 必须有维护节奏

§10.3案例 3 · 销售助手 · A/B test 假赢

背景：销售助手新 prompt A/B 跑 2 周 · A 组 conversion +3.2% · 推全量 3 周后 conversion 反而 -1.8%。

挖根因：A/B 时 A 组分到的是新客户多 (随机 hash 跟客户注册时间相关) · 新客户基数 conversion 本来就高。这是分组偏差不是真效果。

fix：

  1. 分组算法用 deterministic hash on user_id · 不能跟时间相关
  2. 看 A/B 报告必看分组 baseline 是否平衡（pre-test 关键 segment 各组占比）
  3. 用 stratified sampling 强制平衡

教训：A/B 不是分流就完事 · 分组偏差 / Simpson 悖论会骗你
